**토론 일자:** 2026-02-11
**참여자:** 입장A 옹호자, 입장B 옹호자, 중립 심판

## 1. 배경 및 쟁점 요약

PII Scanner는 정규식+체크섬 1차 탐지 후, 의심 텍스트에 한해 NER(Named Entity Recognition) 모델을 적용하는 하이브리드 전략을 채택했다. NER 모델은 KoELECTRA-Small v3 (INT8 양자화)이며, ONNX Runtime을 통해 CPU에서 추론한다. 이 모델을 **언제 메모리에 올리고, 언제 내릴 것인가**가 본 토론의 핵심 쟁점이다.

**핵심 쟁점:**
- **입장 A:** "스캔 세션 단위 로드/언로드". 스케줄 스캔 시작 시 모델을 로드하고, 스캔 완료 후 즉시 언로드하여 메모리를 회수한다. Phase 2 실시간 감지에서는 debounce window(30초) 내 이벤트를 누적한 뒤 배치 로드하고, idle 5분 후 언로드한다.
- **입장 B:** "모델 상시 상주 + 메모리 하드캡". 모델을 한 번 로드하면 메모리에 계속 유지하되, 시스템 메모리 압박(예: 가용 메모리 10% 미만) 시에만 언로드한다. 로드/언로드 반복으로 인한 thrashing 우려를 원천 해소한다.

**합의 실패 이유:** MVP에서 NER이 베타/OFF 상태이므로 논의 우선순위가 낮았고, Phase 2 실시간 NER 도입 시 모델 메모리 관리가 결정적으로 중요해진다. 4GB 저사양 PC와 8GB 일반 PC에서의 최적 전략이 다를 수 있어 단일 결론에 도달하지 못했다.

**잠정 추천:** Claude안(세션 단위)을 MVP 기본으로 하되, Phase 0에서 ONNX Runtime 로드 시간을 실측한 후 최종 결정.

**참고 벤치마크 데이터 (Gemini 제공):**
- 모델 크기: KoELECTRA-Small v3 INT8 양자화, 약 14-18MB (토크나이저 포함)
- ONNX Runtime + 모델 상주 시 RSS: 약 40-60MB
- 로드 지연: SSD 150ms-400ms, HDD(단편화) 1.5s-2.0s
- 토크나이저 초기화 오버헤드: 50ms-100ms 추가

**고려 요소:**
- 4GB PC에서의 메모리 압박과 "Quiet Agent" 철학 (유휴 시 0MB 영향)
- Windows Working Set Trimming 동작과 mmap 기반 lazy loading 가능성
- Phase 2 실시간 NER에서의 로드/언로드 빈도와 thrashing 시나리오
- ONNX Runtime의 메모리 해제 완전성 (FFI 경계에서의 메모리 누수 가능성)

## 2. 입장 A 옹호 (세션 단위 로드/언로드)

### 2.1 핵심 논거

1. **"Quiet Agent" 철학의 완벽한 구현:** PII Scanner의 핵심 차별화는 사용자 업무를 방해하지 않는 것이다. 세션 단위 로드/언로드는 비스캔 시간에 **0MB의 NER 메모리 점유**를 보장한다. 사용자가 웹 브라우징, 메일 확인, 문서 작업을 할 때 PII Scanner가 40-60MB의 메모리를 점유하고 있다면, 작업 관리자에서 이를 발견한 사용자(또는 IT 관리자)는 "가벼운 에이전트"라는 인상을 갖지 못한다.

2. **4GB PC에서의 현실적 메모리 예산:** 4GB(실질 약 3.5GB 가용) 환경에서 Windows OS(~1.5GB) + 오피스/브라우저(~1GB) + AV/EDR(~200MB) + 기타 에이전트를 실행하면, 여유 메모리는 500MB-1GB 수준이다. 여기서 60MB를 항상 점유하면 가용 메모리의 6-12%에 해당하며, 다른 "상시 상주" 보안 에이전트들과의 "RAM 크리프" 현상을 가중시킨다.

3. **로드 비용이 무시할 수 있는 수준:** SSD 환경에서 150-400ms, 토크나이저 포함 200-500ms의 로드 시간은 스케줄 스캔(수십 분~수 시간) 대비 무시할 수 있다. 1시간 스캔에서 500ms 로드는 0.01%의 오버헤드이다. HDD 환경에서도 2초 이내이며, 이는 스캔 시작 시 한 번만 발생한다.

4. **Phase 2 실시간에서도 debounce로 관리 가능:** 실시간 파일 감지에서 매 이벤트마다 NER을 호출하는 것은 어차피 비효율적이다. Debounce window(30초)로 이벤트를 누적하고, 배치로 처리한 뒤, idle 5분 후 언로드하는 전략은 로드 비용을 최소화하면서도 메모리를 회수한다. 활발한 파일 작업 시에는 모델이 메모리에 유지되고, 작업이 멈추면 자동으로 해제된다.

### 2.2 기술적 근거

**Phase 1 (스케줄 스캔) 메모리 프로파일:**

```
[유휴 상태]
PII Scanner RSS: ~8-12MB (서비스 프로세스 + 워커 대기)
NER 모델: 0MB (미로드)

[스캔 시작]
t=0s: 스캔 시작 신호
t=0.2-0.5s: ONNX Session 생성 + 모델 로드 (SSD 기준)
PII Scanner RSS: ~50-70MB (서비스 + 워커 + 모델)

[스캔 진행: 30분-2시간]
모델 상주, 추론 반복

[스캔 완료]
t=end: ort::Session drop → 메모리 해제
PII Scanner RSS: ~8-12MB (유휴 복귀)
```

**Phase 2 (실시간 감지) debounce 전략:**

```
[이벤트 수신]
t=0s: 파일 변경 이벤트 감지
t=0-30s: debounce window — 추가 이벤트 누적

[배치 처리]
t=30s: 큐에 NER 대상 파일이 있으면 모델 로드 (200-500ms)
t=30.5s~: 배치 NER 추론 실행

[idle 타이머]
마지막 NER 요청 이후 5분간 새 요청 없음 → 모델 언로드
```

**ONNX Runtime 메모리 해제 검증 방안:**

세션 단위 전략의 전제 조건은 `ort::Session` drop 시 메모리가 실제로 OS에 반환되는 것이다. 이를 Phase 0에서 다음과 같이 검증한다:
- `ort` crate의 Session 생성/소멸을 100회 반복하며 RSS 추이 모니터링
- Rust의 jemalloc/mimalloc 대신 시스템 할당자를 사용하여 OS 반환을 보장
- FFI 경계(ONNX Runtime C API)에서의 메모리 누수 여부 확인
- Windows Task Manager의 Working Set 및 Private Bytes 비교

**mmap 기반 최적화 가능성:**

ONNX Runtime은 내부적으로 모델 파일을 mmap으로 로드할 수 있다. 이 경우:
- 물리 메모리는 실제 접근한 페이지만 할당 (lazy loading)
- 메모리 압박 시 OS가 자동으로 페이지를 회수 (file-backed이므로 swap 불필요)
- "언로드"가 아니라 OS의 페이지 관리에 위임하는 하이브리드 접근도 가능
- 단, mmap 사용 시에도 ONNX Runtime의 내부 버퍼(추론 작업 공간)는 별도 힙 할당이므로 완전한 "자동 회수"는 아님

### 2.3 예상 반론에 대한 재반박

**반론 1: "HDD 환경에서 2초 로드는 사용자가 체감할 수 있는 지연이다."**

> 재반박: 스케줄 스캔은 사용자가 요청하는 인터랙티브 작업이 아니다. 백그라운드에서 2초 로드가 발생해도 사용자는 이를 인지하지 못한다. Phase 2 실시간에서도 debounce window 30초 이후에 로드가 시작되므로, 사용자가 파일을 저장한 직후가 아니라 30초 뒤에 백그라운드로 처리된다. 사용자 체감과 무관한 지연이다.

**반론 2: "Phase 2에서 파일 이벤트가 빈번하면 로드/언로드가 반복되어 thrashing이 발생한다."**

> 재반박: 이것이 debounce window와 idle timeout을 설계한 이유이다. idle timeout이 5분이므로, 5분 이내에 새 이벤트가 도착하면 모델은 메모리에 유지된다. 실질적으로 thrashing이 발생하려면 "5분 이상 조용했다가 이벤트 발생 → 로드 → 다시 5분 조용 → 언로드 → 또 이벤트"가 반복되어야 한다. 이 패턴에서 로드는 회당 500ms 미만이며, 5분 간격이면 시간당 최대 12회 = 6초의 총 로드 오버헤드이다. 이는 thrashing이라 부르기 어렵다.

> 만약 이 패턴이 비정상적으로 빈번하다면(예: 정확히 5분 주기의 자동 저장), idle timeout을 10분 또는 15분으로 조정하는 것은 설정 변경 한 줄이다. 핵심은 **기본값을 "언로드"로 두고 필요 시 유지 시간을 늘리는 것**이 기본값을 "상주"로 두고 압박 시 내리는 것보다 안전하다는 점이다.

**반론 3: "Windows의 Working Set Trimming이 어차피 유휴 메모리를 회수하므로, 상주해도 실질 영향이 적다."**

> 재반박: Working Set Trimming은 물리 메모리에서 페이지를 제거하지만, 해당 메모리는 pagefile(스왑)로 이동한다. 이는 두 가지 문제를 야기한다. 첫째, pagefile I/O가 디스크 부하를 유발한다(특히 HDD). 둘째, 다음 NER 추론 시 pagefile에서 다시 읽어야 하므로 "콜드 스타트"보다 느릴 수 있다 (pagefile read > SSD file read). 명시적 언로드는 메모리를 완전히 해제하여 pagefile 부하를 원천 제거한다.

> 또한 Working Set Trimming은 OS의 판단에 의존하므로 **예측 불가능하다.** 메모리 압박이 심하지 않으면 Trimming이 발생하지 않고, 60MB가 물리 메모리에 그대로 남는다. "OS가 알아서 해줄 것"이라는 가정에 의존하는 것보다 명시적으로 관리하는 것이 엔터프라이즈 소프트웨어에 적합하다.

## 3. 입장 B 옹호 (모델 상시 상주 + 메모리 하드캡)

### 3.1 핵심 논거

1. **로드 비용은 "평균"이 아니라 "최악"으로 판단해야 한다:** 벤치마크에서 SSD 기준 150-400ms, HDD 기준 1.5-2.0s라고 하지만, 이는 이상적 조건의 수치이다. 실제 기업 PC에서는 AV의 실시간 스캔이 모델 파일 읽기를 가로채어 지연이 발생하고, 디스크 단편화가 심한 HDD에서는 3초 이상 소요될 수 있다. 또한 ONNX Runtime 초기화 시 DLL 로드, 커널 오브젝트 생성 등 OS 레벨 오버헤드가 추가된다. **최악 조건에서 3-5초의 로드 지연이 반복되면 이것이 곧 thrashing이다.**

2. **Phase 2 실시간에서 debounce가 만능이 아니다:** Debounce window 30초, idle timeout 5분이라는 설정은 "일반적인 사용 패턴"에 최적화된 것이다. 그러나 기업 환경에서는 다양한 자동화 도구(문서 관리 시스템, 그룹웨어 자동 동기화, 백업 소프트웨어)가 예측 불가능한 패턴으로 파일 이벤트를 생성한다. 이런 환경에서 debounce 파라미터를 고객사별로 튜닝하는 것은 운영 부담이며, 잘못된 설정은 탐지 지연 또는 thrashing으로 직결된다.

3. **40-60MB는 2026년 기준으로 합리적인 상주 비용이다:** 4GB PC에서 60MB는 약 1.5%에 불과하다. Chrome 탭 하나가 100-300MB를 소비하는 시대에, 보안 에이전트가 60MB를 상주시키는 것은 합리적 트레이드오프이다. 문제는 절대 크기가 아니라 **메모리 압박 상황에서의 대응**이다. 이를 위해 하드캡 + 메모리 감시 메커니즘을 설계한다.

4. **"한 번 로드, 영원히 빠르게"의 단순함:** 세션 단위 전략은 로드/언로드 타이밍, debounce 파라미터, idle timeout, 메모리 누수 검증 등 관리해야 할 변수가 많다. 상시 상주는 "로드한다 → 끝"이라는 단순한 모델이다. 소프트웨어 복잡도를 낮추는 것 자체가 안정성 향상이며, 엣지 케이스에서의 예기치 못한 동작을 줄인다.

### 3.2 기술적 근거

**메모리 하드캡 + 압박 감지 설계:**

```
[초기 로드]
서비스 시작 시 또는 첫 NER 요청 시 모델 로드
→ 이후 프로세스 생존 기간 동안 메모리에 상주

[메모리 압박 감지]
Windows Memory Notification API (CreateMemoryResourceNotification)
→ LowMemoryResourceNotification 수신 시:
  1단계: Working Set Trim 요청 (SetProcessWorkingSetSizeEx)
  2단계: 가용 메모리 10% 미만 지속 시 모델 언로드
  3단계: 가용 메모리 회복 후 다음 NER 요청 시 재로드

[메모리 예산 하드캡]
전체 PII Scanner 프로세스의 RSS 상한: 150MB
- 서비스 + 워커 기본: ~15MB
- NER 모델 + ONNX Runtime: ~60MB
- 텍스트 추출 버퍼: ~50MB
- 여유: ~25MB
```

**Windows Memory Notification 활용:**

```rust
// 의사 코드: 메모리 압박 감지
let low_memory = CreateMemoryResourceNotification(
    LowMemoryResourceNotification
);

// 별도 스레드에서 모니터링
loop {
    WaitForSingleObject(low_memory, INFINITE);
    // 시스템 메모리 부족 상태 진입
    if ner_model.is_loaded() && !ner_model.is_actively_inferring() {
        ner_model.unload();  // 긴급 메모리 회수
        log::warn!("NER 모델 언로드: 시스템 메모리 부족");
    }
}
```

**상주 모델의 Working Set 관리:**

모델이 메모리에 상주하더라도, Windows의 Working Set 관리와 협력하여 실질적 메모리 영향을 최소화할 수 있다:
- 모델 데이터가 mmap file-backed이면, OS가 물리 메모리 압박 시 자동으로 페이지를 회수
- 회수된 페이지는 pagefile이 아니라 **원본 파일에서 다시 읽을 수 있으므로** pagefile 부하 없음
- 다음 추론 시 필요한 페이지만 fault-in되므로, 전체 60MB가 아니라 실제 사용 부분만 물리 메모리 점유
- 이는 "명시적 언로드" 없이도 OS 수준에서 메모리가 관리되는 효과

**로드 지연의 최악 시나리오 분석:**

| 조건 | 예상 로드 시간 | 비고 |
|:---|:---|:---|
| SSD + AV 미간섭 | 200-500ms | 이상적 조건 |
| SSD + AV 실시간 스캔 | 500ms-1.5s | AV가 ONNX 파일을 스캔 |
| HDD + 단편화 | 1.5-2.0s | Gemini 벤치마크 |
| HDD + 단편화 + AV | 2.0-4.0s | 최악 조건 |
| 1코어 VM + 2GB RAM + HDD | 3.0-5.0s | 극한 환경 |

최악 조건에서 5초 로드가 하루에 여러 번 반복되면, 사용자가 체감하지 못하더라도 디스크 I/O 부하가 다른 프로세스에 영향을 줄 수 있다. 상시 상주는 이 부하를 **서비스 시작 시 1회로 한정**한다.

### 3.3 예상 반론에 대한 재반박

**반론 1: "4GB PC에서 60MB 상주는 'Quiet Agent' 철학에 반한다."**

> 재반박: "Quiet Agent"의 본질은 **사용자의 업무를 방해하지 않는 것**이지, **메모리 0바이트를 추구하는 것**이 아니다. 60MB 상주가 사용자 업무에 영향을 주는가? 4GB PC에서도 Windows는 SuperFetch(SysMain)에 수백 MB를 할당하고, 메모리 압박 시 이를 자동 해제한다. 우리의 60MB도 동일한 메커니즘으로 관리될 수 있다.

> 오히려 세션 단위 전략에서 로드 시 발생하는 **순간적 디스크 I/O 스파이크**가 사용자 체감에 더 영향을 줄 수 있다. 특히 HDD 환경에서 모델 로드 중 사용자가 파일을 열려고 하면 디스크 경합이 발생한다. 상시 상주는 이 I/O 스파이크를 서비스 시작 시 1회로 제한한다.

**반론 2: "메모리 압박 시에만 언로드하면, 그 '압박'을 감지하는 시점이 이미 늦을 수 있다."**

> 재반박: Windows의 `CreateMemoryResourceNotification`은 **OS 커널 수준의 조기 경고 시스템**이다. 가용 메모리가 임계치 이하로 떨어지기 시작할 때 알림을 보내므로, OOM 상황에 도달하기 전에 선제적으로 대응할 수 있다. 추가로, 자체적인 메모리 모니터링(예: 30초 주기로 `GlobalMemoryStatusEx` 호출)으로 가용 메모리 비율을 추적하고, 20% 이하로 진입하면 선제적 언로드를 시작하는 2중 안전장치를 구현한다.

> 또한 모델 언로드 자체는 즉각적이다(메모리 해제는 ms 단위). 문제는 "감지 → 판단 → 해제"까지의 지연인데, 이는 1초 이내로 완료된다. 이 1초 동안 시스템이 OOM에 빠질 확률은 극히 낮다.

**반론 3: "상주 모델이 ONNX Runtime 내부 메모리 누수를 은폐할 수 있다."**

> 재반박: 이는 상주/세션 전략과 무관한 별개의 품질 이슈이다. 세션 단위 전략에서도 긴 스캔(2시간) 중 수천 회 추론을 반복하면 동일한 누수가 발생할 수 있다. 메모리 누수는 **추론 횟수에 비례**하지, 모델의 로드 지속 시간에 비례하지 않는다.

> 오히려 세션 단위 전략에서 "로드 → 수천 회 추론 → 언로드"를 반복하면, 누수가 있을 경우 매 세션마다 일부 메모리가 회수되지 않아 **점진적 메모리 증가**가 발생할 수 있다. 상주 모델은 RSS 모니터링으로 누수를 쉽게 탐지할 수 있다(상주 중 RSS가 꾸준히 증가하면 누수).

## 4. 중립 심판 평가

### 4.1 입장 A의 강점과 약점

**강점:**

- **"Quiet Agent" 철학과 가장 정합한다.** 유휴 시 NER 메모리 0MB라는 약속은 마케팅적으로도, 기술적으로도 명쾌하다. 작업 관리자에서 PII Scanner의 메모리 점유가 10MB 미만인 것을 IT 관리자가 확인할 수 있다면, 이는 경쟁 제품 대비 강력한 차별화이다.
- **로드 비용이 실측 데이터 기반으로 충분히 낮다.** Gemini 벤치마크에서 SSD 200-500ms, HDD에서도 2초 이내라는 수치가 제시되었고, 이는 스케줄 스캔 맥락에서 무시 가능한 수준이다.
- **Phase 2 debounce 설계가 구체적이고 합리적이다.** 30초 누적 → 배치 처리 → 5분 idle 언로드라는 3단계 전략은 대부분의 사용 패턴을 커버한다.
- **메모리 해제의 결정론적 보장.** 명시적 언로드는 "OS가 알아서 해주길 바란다"가 아니라 "우리가 직접 해제한다"이므로, 메모리 상태를 예측 가능하게 관리할 수 있다.

**약점:**

- **HDD + AV 환경의 최악 조건을 과소평가할 수 있다.** 벤치마크는 통제된 환경이며, 실제 기업 PC에서는 AV 간섭, 디스크 경합, 시스템 서비스 경합 등 추가 변수가 존재한다. 최악 조건에서 3-5초 로드가 Phase 2 실시간에서 반복되면 사용자 체감에 간접적 영향을 줄 수 있다.
- **debounce 파라미터의 튜닝 부담.** 30초 debounce, 5분 idle timeout이라는 기본값이 모든 고객사 환경에 적합하다는 보장이 없다. 문서 관리 시스템의 자동 동기화, 그룹웨어의 주기적 파일 업데이트 등 환경별로 다른 패턴이 존재하며, 잘못된 파라미터는 과도한 로드/언로드 또는 탐지 지연을 유발한다.
- **ONNX Runtime의 메모리 해제 완전성이 아직 검증되지 않았다.** `ort::Session` drop 시 모든 메모리가 OS에 반환되는지는 Phase 0에서 실측해야 한다. 만약 메모리 반환이 불완전하면 세션 단위 전략의 전제가 무너진다.

### 4.2 입장 B의 강점과 약점

**강점:**

- **설계의 단순성.** "로드한다 → 끝"이라는 모델은 로드/언로드 타이밍, debounce 파라미터, idle timeout 등의 복잡한 상태 관리를 제거한다. 상태 머신이 단순할수록 버그 가능성이 낮아진다.
- **최악 조건의 I/O 스파이크를 서비스 시작 시 1회로 제한.** HDD + AV 환경에서도 모델 로드는 서비스 기동 시 단 한 번만 발생한다. 이후 모든 NER 추론은 메모리에서 즉시 처리된다.
- **Phase 2 실시간 NER에서 thrashing 우려가 원천 제거된다.** 어떤 파일 이벤트 패턴이 오더라도 모델은 이미 메모리에 있으므로, 즉시 추론을 시작할 수 있다.
- **Windows Memory Notification API 활용이 기술적으로 타당하다.** OS 커널 수준의 메모리 압박 감지를 활용하므로, 폴링 기반 모니터링보다 효율적이고 반응이 빠르다.

**약점:**

- **"Quiet Agent" 약속과의 갈등.** 유휴 시에도 40-60MB를 점유하는 것은 "0 영향"이라는 이상과 거리가 있다. 마케팅적으로 "유휴 시 메모리 10MB 미만"이라고 주장할 수 없다. 경쟁사 대비 차별화 메시지가 약해진다.
- **메모리 압박 감지의 신뢰성이 검증되지 않았다.** `CreateMemoryResourceNotification`의 임계치가 환경별로 다를 수 있으며, 특히 4GB + HDD 환경에서 메모리 부족 감지 → 모델 언로드 → OS가 회수된 메모리를 활용하기까지의 전체 지연이 충분히 짧은지 확인이 필요하다.
- **pagefile 의존도가 증가할 수 있다.** 모델이 상주하지만 한동안 추론이 없으면 Windows가 해당 페이지를 Working Set에서 제거하고 pagefile로 보낸다. 이후 NER 요청이 오면 pagefile에서 다시 읽어야 하므로, mmap file-backed가 아닌 경우 "콜드 스타트보다 느린 웜 스타트"가 발생할 수 있다.
- **"RAM 크리프" 현상에 기여한다.** 기업 PC에는 AV, EDR, DLP, 패치 관리, 자산 관리 등 다수의 에이전트가 설치되어 있다. 각각이 "60MB 정도는 합리적"이라고 판단하면, 합산 메모리가 수백 MB에 달한다. PII Scanner가 이 문제에 기여하지 않는 것이 차별화 전략이다.

### 4.3 핵심 트레이드오프 분석

| 평가 항목 | 입장 A (세션 단위) | 입장 B (상시 상주) | 비고 |
|:---|:---|:---|:---|
| **유휴 시 메모리** | 우월 (0MB) | 열등 (40-60MB) | "Quiet Agent" 핵심 지표 |
| **NER 응답 지연** | 열등 (로드 200ms-5s) | 우월 (즉시) | Phase 2 실시간에서 차이 발생 |
| **디스크 I/O 부하** | 열등 (로드마다 I/O) | 우월 (서비스 시작 시 1회) | HDD 환경에서 유의미 |
| **설계 복잡도** | 높음 (상태 머신, debounce, idle) | 낮음 (로드 → 유지) | 버그 가능성에 영향 |
| **파라미터 튜닝** | 필요 (debounce, idle timeout) | 불필요 | 고객사별 커스터마이징 부담 |
| **4GB PC 적합성** | 우월 | 보통 | 메모리 여유에 직접 기여 |
| **8GB+ PC 적합성** | 보통 | 우월 | 60MB가 무의미한 수준 |
| **Phase 1 MVP 적합성** | 우월 | 보통 | 스케줄 스캔에서 상주 불필요 |
| **Phase 2 실시간 적합성** | 보통 | 우월 | 빈번한 이벤트에서 유리 |
| **최악 조건 안정성** | 보통 (로드 실패 가능) | 우월 (이미 로드됨) | 디스크 장애 시 차이 |
| **마케팅 차별화** | 우월 ("0MB 유휴") | 열등 | 경쟁사 대비 어필 |

**결정적 변수 3가지:**

1. **ONNX Runtime 로드 시간 실측치 (Phase 0):** SSD에서 300ms 이내이면 세션 단위의 비용이 무시 가능. HDD에서 3초 이상이면 상주의 이점이 커진다.

2. **대상 고객의 PC 사양 분포:** 4GB PC가 30% 이상이면 세션 단위가 안전한 선택. 8GB 이상이 80%라면 상주도 합리적.

3. **Phase 2 실시간 NER의 이벤트 빈도:** 분당 1-2회 수준이면 세션 단위로 충분. 분당 10회 이상이면 상주가 유리.

### 4.4 조건부 결론

두 입장 모두 기술적으로 유효하며, 최적 선택은 **배포 환경의 하드웨어 분포와 Phase 2 사용 패턴**에 의해 결정된다. 그러나 **MVP 시점에서의 판단 기준**을 제시한다면:

1. **MVP(Phase 1)에서는 입장 A가 명백히 우월하다.** Phase 1은 스케줄 스캔만 지원하며, NER은 베타/OFF 상태이다. 스캔 시작 시 1회 로드, 완료 후 언로드라는 단순한 패턴에서 상시 상주의 이점은 거의 없다. 반면 유휴 시 0MB라는 "Quiet Agent" 약속은 제품의 첫인상을 결정한다.

2. **Phase 2(실시간 NER)에서는 환경에 따라 달라진다.** 4GB PC 비율이 높고 이벤트 빈도가 낮으면 입장 A(debounce + idle timeout), 8GB PC가 대다수이고 이벤트가 빈번하면 입장 B(상주 + 메모리 하드캡)가 적합하다.

3. **두 전략을 상호 배타적으로 볼 필요가 없다.** "세션 단위"를 기본으로 구현하되, 시스템 메모리가 8GB 이상이고 실시간 NER이 활성화된 환경에서는 "상주 모드"를 옵션으로 제공하는 **적응형 전략**이 가능하다.

## 5. 최종 추천안

**추천: 입장 A(세션 단위 로드/언로드)를 MVP 기본으로 채택하되, 적응형 전략으로 확장**

### MVP (Phase 1) 구현

```
[스케줄 스캔 모드]
1. 스캔 시작 신호 수신
2. ONNX Session 생성 + 모델 로드 (200ms-2s)
3. 스캔 진행: Regex/Checksum 1차 → NER 대상에 한해 추론
4. 스캔 완료: Session drop → 메모리 해제
5. 유휴 복귀: NER 메모리 0MB
```

### Phase 2 확장: 적응형 전략

```
[시스템 메모리 기반 자동 전략 선택]

IF 시스템 메모리 < 6GB:
  → 세션 단위 전략 (입장 A)
  → debounce 30초, idle timeout 5분
  → 유휴 시 NER 메모리 0MB 보장

ELSE IF 시스템 메모리 >= 6GB AND 실시간 NER 활성:
  → 상주 전략 (입장 B) 옵션 제공
  → 서비스 시작 시 또는 첫 NER 요청 시 로드
  → Memory Notification API로 압박 감지 시 언로드
  → RSS 하드캡 150MB

ELSE:
  → 세션 단위 전략 (기본값)
```

### 핵심 설계 원칙

1. **기본값은 항상 "세션 단위(언로드)"이다.** 상주 모드는 옵트인(opt-in)으로만 활성화된다. "Quiet Agent"를 기본으로 보장하면서, 고성능이 필요한 환경에서 선택적으로 상주를 허용한다.

2. **Phase 0에서 반드시 실측한다.** 다음 데이터를 Phase 0에서 수집하여 Phase 2 전략을 확정한다:
   - i3/4GB/HDD VM에서 `ort::Session` 생성/소멸 시간
   - Session drop 후 RSS가 완전히 복원되는지 여부 (100회 반복)
   - AV(Windows Defender) 활성 상태에서의 로드 시간 증가폭
   - mmap 모드 사용 시 Working Set 동작 확인

3. **idle timeout은 서버 정책으로 원격 조정 가능하게 설계한다.** 기본값 5분이지만, 고객사 환경에 따라 서버에서 1분~30분 범위로 변경할 수 있도록 한다. 이는 debounce window(기본 30초)도 동일하다.

4. **메모리 예산을 명시적으로 관리한다.** 어떤 전략을 택하든, 전체 PII Scanner 프로세스의 RSS 상한(150MB)을 하드캡으로 설정하고, 이를 초과하면 NER을 비활성화하고 Regex-only 모드로 폴백한다.

5. **ONNX Runtime의 mmap 로딩을 적극 활용한다.** mmap file-backed 로딩이 가능하면, "상주"와 "언로드"의 경계가 흐려진다. OS가 메모리 관리를 담당하되, 우리는 Session 객체를 유지할지 여부만 결정한다. 이는 두 입장의 장점을 결합하는 기술적 중간지대이다.

### Phase 0 검증 후 최종 결정 기준

| 실측 결과 | 최종 결정 |
|:---|:---|
| 로드 시간 500ms 미만 (i3/4GB/SSD) | 세션 단위를 Phase 2 기본으로 확정 |
| 로드 시간 500ms-2s (i3/4GB/HDD) | 세션 단위 기본, HDD 환경에 한해 상주 옵션 제공 |
| 로드 시간 2s 초과 | 상주 전략을 Phase 2 기본으로 격상 (4GB에서도) |
| Session drop 후 메모리 미반환 | 상주 전략으로 전환 (세션 단위의 전제 무너짐) |
| mmap 모드에서 OS 자동 관리 확인 | mmap 상주를 기본으로, 명시적 Session 유지 |

### 이 추천이 양측 입장을 어떻게 수용하는가

- **입장 A에서 수용한 것:** 세션 단위를 기본 전략으로 채택, "Quiet Agent" 철학을 최우선으로 존중, debounce + idle timeout 설계를 Phase 2 세션 전략에 그대로 반영, 유휴 시 0MB 보장을 기본 동작으로 설정.
- **입장 B에서 수용한 것:** 상시 상주를 완전히 배제하지 않고 옵트인 옵션으로 제공, Memory Notification API 기반 압박 감지를 상주 모드에 적용, 최악 조건(HDD + AV)에서의 로드 비용 우려를 실측 데이터로 해소하는 결정 프레임워크 제시, RSS 하드캡(150MB)을 양쪽 전략 모두에 적용.
- **절충 지점:** "시스템 메모리 6GB 기준"으로 전략을 자동 분기하여, 저사양 PC에서는 메모리를 아끼고 고사양 PC에서는 성능을 극대화. Phase 0 실측 데이터에 따라 최종 결정을 유보하여, 벤치마크 수치가 아닌 실제 환경 데이터로 판단.
