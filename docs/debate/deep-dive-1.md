# Deep Dive 1: Aho-Corasick 프리필터의 적용 범위

**토론 일자:** 2026-02-11
**참여자:** 입장A 옹호자, 입장B 옹호자, 중립 심판

---

## 1. 배경 및 쟁점 요약

PII Scanner의 탐지 파이프라인은 3단계 퍼널 구조를 가진다: **Aho-Corasick 키워드 프리필터 -> Regex/Checksum 구조화 탐지 -> NER 비정형 탐지.** 이 중 가장 앞단인 Aho-Corasick 프리필터의 역할 범위에 대해 두 가지 입장이 대립한다.

**핵심 쟁점:** Aho-Corasick 프리필터에서 키워드가 하나도 발견되지 않은 파일에 대해, 후속 단계를 어디까지 스킵할 수 있는가?

- **입장 A:** 키워드("주민", "전화", "카드", 숫자 패턴 등)가 전혀 없는 파일은 Regex 단계조차 건너뛴다. 기업 환경에서 대다수를 차지하는 코드/로그/설정 파일 등 PII가 없는 파일을 대량으로 걸러내어 성능을 극대화한다.
- **입장 B:** 키워드가 없어도 구조화 PII(주민등록번호, 카드번호, 이메일, 전화번호)는 존재할 수 있으므로, Regex/Checksum은 항상 실행한다. 키워드 프리필터는 비용이 큰 NER 호출 여부를 결정하는 게이팅에만 사용한다.

**배경 데이터:** Codex 팀이 수행한 PoC(`poc/unresolved1_keyword_prefilter_poc.py`)에서, 키워드가 전혀 없는 순수 숫자열 텍스트에 체크섬이 유효한 주민등록번호가 포함될 수 있음을 실증했다. 한편, Gemini 팀은 파일 타입별 차등 전략("Context-Aware Funnel")을 제안하여, 비구조화 파일(로그, 소스코드)은 전체 스킵하되 구조화 데이터 파일(CSV, XLSX)은 Regex를 실행하는 계층적 접근을 제시했다.

이 토론은 두 입장을 정면으로 대비하고, 최적의 결론을 도출하는 것을 목표로 한다.

## 2. 입장 A 옹호 (키워드 없는 파일 전체 스킵)

### 2.1 핵심 논거

1. **압도적 성능 이득:** 기업 PC 환경에서 스캔 대상 파일의 70-85%는 소스코드, 로그, 설정 파일, 바이너리 텍스트 등 PII와 무관한 파일이다. 이 파일들을 Aho-Corasick 한 번의 패스로 제거하면, Regex 엔진이 처리해야 할 파일 수를 5분의 1 이하로 줄일 수 있다.

2. **"Quiet Agent" 철학과의 일치:** 프로젝트의 핵심 차별화 요소인 "사용자 업무를 방해하지 않을 것"을 달성하려면, CPU 사용량을 최소화해야 한다. 특히 i3/4GB RAM급 저사양 PC에서 수만 개 파일을 스캔할 때, 불필요한 Regex 실행을 제거하는 것은 체감 성능에 직접적 영향을 미친다.

3. **Aho-Corasick의 이론적 효율성:** Aho-Corasick 알고리즘은 O(n + m + z) 시간복잡도로 동작한다(n: 입력 크기, m: 패턴 총 길이, z: 매칭 수). 수십 개의 키워드를 동시에 검색해도 입력 텍스트를 한 번만 순회하므로, 파일당 추가 비용이 거의 없다. 반면, 파일마다 5-10개의 정규식을 실행하는 것은 각각 별도의 NFA/DFA 탐색이 필요하다.

### 2.2 기술적 근거

**성능 수치 비교 (추정치, 1MB 텍스트 파일 기준):**

| 단계 | 처리 시간 (추정) | 비고 |
|:---|:---|:---|
| Aho-Corasick (키워드 20개) | 0.5-2ms | 단일 패스, SIMD 가속 가능 |
| Regex 풀 스캔 (패턴 8개) | 5-30ms | 패턴별 NFA/DFA 탐색 |
| NER 추론 (ONNX, CPU) | 50-500ms | 토큰화 + 모델 추론 |

50,000대 PC 환경에서 PC당 평균 100,000개 파일을 스캔한다고 가정하면:
- 키워드 미탐지 파일 비율 80%일 때, 전체 스킵 시 Regex 실행 횟수: 20,000회
- 항상 실행 시 Regex 실행 횟수: 100,000회
- **PC당 절감 시간:** (80,000회 x 15ms 평균) = 약 1,200초 = **20분**

풀스캔 소요 시간이 2시간에서 1시간 40분으로 줄어드는 것은, 사용자 체감으로는 "스캔 중 PC가 버벅이는 시간"의 의미 있는 감소다.

**기업 파일 타입 분포 (15년 운영 경험 기반 추정):**

| 파일 유형 | 비율 | PII 포함 가능성 | 키워드 동반 여부 |
|:---|:---|:---|:---|
| 소스코드 (.java, .py, .js 등) | 25-35% | 극히 낮음 | 해당 없음 |
| 로그/설정 (.log, .conf, .xml) | 15-25% | 낮음 | 해당 없음 |
| 문서 (.docx, .hwp, .pdf) | 10-20% | 높음 | 대부분 동반 |
| 스프레드시트 (.xlsx, .csv) | 5-10% | 높음 | **불확실** |
| 기타 (이미지, 바이너리 등) | 15-25% | 낮음 | 해당 없음 |

소스코드 + 로그/설정 + 기타를 합산하면 55-85%에 달하며, 이 파일들은 키워드 프리필터만으로도 안전하게 스킵할 수 있다.

### 2.3 예상 반론에 대한 재반박

**반론 1: "숫자만 나열된 CSV/Excel에서 주민번호를 놓칠 수 있다."**

재반박: 이 시나리오는 실재하지만, 빈도가 매우 낮다. 정상적인 기업 문서에서 주민등록번호가 포함된 스프레드시트는 거의 반드시 "주민등록번호", "생년월일", "성명" 등의 헤더를 갖고 있다. 헤더 없이 순수 숫자만 나열된 채 주민번호가 포함되는 경우는 다음과 같은 극단적 상황에 한정된다:
- 시스템 간 데이터 덤프 중간 파일
- 암호화/마스킹 전 임시 추출 파일
- 레거시 시스템의 고정폭 텍스트 파일

이런 파일의 비율은 Phase 0에서 실측해야 하지만, 전체 파일의 0.1% 미만일 것으로 추정된다. 0.1% 미만의 미탐 위험을 위해 나머지 99.9%에 불필요한 Regex를 실행하는 것은 비용 대비 효과가 낮다.

**반론 2: "Regex 비용은 어차피 낮으므로 항상 실행해도 무방하다."**

재반박: 개별 파일 단위에서는 Regex 비용이 낮지만(5-30ms), 50,000대 PC x 100,000파일 규모에서는 집합적 영향이 크다. 특히 "Quiet Agent" 철학에서 중요한 것은 **피크 CPU 사용률**이다. 사용자가 문서 작업 중일 때 백그라운드 스캔이 CPU를 점유하는 정도가 Regex 실행 여부에 따라 달라진다. Regex를 스킵하면 CPU 버스트 구간이 줄어들어, 사용자가 스캔 프로세스의 존재를 인지하지 못할 확률이 높아진다.

## 3. 입장 B 옹호 (Regex/Checksum은 항상 실행)

### 3.1 핵심 논거

1. **미탐은 오탐보다 위험하다:** 개인정보 탐지 솔루션에서 미탐(false negative)은 규제 위반으로 이어질 수 있다. 개인정보보호법 위반 시 과징금은 매출의 3% 이하이며, 2023년 기준 최대 과징금 사례는 수백억 원에 달한다. 성능 최적화를 위해 미탐 위험을 감수하는 것은 제품의 존재 이유를 훼손한다.

2. **PoC가 실증한 구조적 결함:** Codex 팀의 PoC에서 명확히 증명되었듯이, 키워드가 전혀 없는 텍스트에 체크섬이 유효한 주민등록번호가 포함될 수 있다. 이것은 이론적 가능성이 아니라 코드로 검증된 사실이다. `"1043321819601  1043321819601  1043321819601"` 같은 입력에서 입장 A는 탐지 건수 0건, 입장 B는 3건을 보고한다.

3. **Regex/Checksum의 실제 비용은 충분히 낮다:** Rust의 `regex` 크레이트는 SIMD 가속 프리필터를 내장하고 있어, 단순 패턴 매칭의 경우 수백 MB/s의 처리량을 보인다. 주민등록번호 패턴(`\d{6}[- ]?\d{7}`)처럼 단순한 숫자 패턴은 특히 빠르다. 파일당 추가 비용은 Aho-Corasick 대비 3-10배 수준이지만, 절대값으로는 여전히 수 밀리초 이내다.

### 3.2 기술적 근거

**PoC 결과 상세 분석:**

PoC(`poc/unresolved1_keyword_prefilter_poc.py`)의 핵심 결과를 재정리한다:

```
[example 1] keyword_hit=False
text: '1043321819601  1043321819601  1043321819601'
  입장 A 결과: [] (미탐 3건)
  입장 B 결과: [RRN(checksum_ok=True) x 3]

[example 2] keyword_hit=False
text: '01012345678 1043321819601 user@example.com'
  입장 A 결과: [] (미탐 3건 - RRN, EMAIL, PHONE)
  입장 B 결과: [RRN, EMAIL, PHONE]

[example 3] keyword_hit=True
text: '주민등록번호: 104332-1819601'
  입장 A 결과: [RRN(checksum_ok=True)]
  입장 B 결과: [RRN(checksum_ok=True)]
```

example 1과 2는 키워드 없이 구조화 PII가 존재하는 현실적 시나리오다. 입장 A는 이 파일들을 완전히 놓친다.

**실제 기업 환경에서의 키워드 없는 구조화 PII 시나리오:**

| 시나리오 | 파일 형태 | 키워드 유무 | PII 유형 |
|:---|:---|:---|:---|
| ERP 데이터 덤프 | CSV, 고정폭 텍스트 | 없음 (컬럼 코드만) | 주민번호, 전화번호 |
| 엑셀 원본 데이터 시트 | XLSX (data-only 탭) | 없음 (헤더가 별도 시트) | 카드번호, 계좌번호 |
| 레거시 시스템 인터페이스 파일 | TXT, DAT | 없음 (위치 기반 파싱) | 주민번호, 사업자번호 |
| 데이터베이스 백업 텍스트 | SQL 덤프, TSV | 컬럼명은 영문 약어 | 모든 구조화 PII |
| 개발자 테스트 데이터 | JSON, CSV | 없음 (필드명이 "col1" 등) | 주민번호, 전화번호 |

Gemini 팀의 분석에서도 인용된 바와 같이, **기업 문서 코퍼스 분석에서 구조화 PII 인스턴스의 약 5-10%는 주변에 식별 가능한 헤더나 키워드 없이 "raw" 테이블에 존재한다.** 이 수치는 무시할 수 있는 수준이 아니다.

**Regex 비용의 실측적 추정:**

Gemini 팀의 제안에서 언급된 기준을 적용하면:
- 엑셀 파일 10,000개에 대한 "Regex-only" 오버헤드가 **50ms 미만/파일**이면 입장 B가 확정된다.
- Rust의 `regex` 크레이트 벤치마크 기준, 1MB 텍스트에 대해 단순 숫자 패턴 8개를 실행하는 데 **3-15ms** 소요된다.
- 따라서 50ms 기준을 충분히 만족한다.

**스트리밍 + 조기 종료 최적화:**

Regex/Checksum을 항상 실행하더라도 비용을 최소화할 수 있다:
- **스트리밍 스캔:** 파일 전체를 메모리에 로드하지 않고 청크 단위로 스캔. 메모리 사용량을 제한하면서도 O(n) 탐색 유지.
- **조기 종료(Early Termination):** 정책에서 정의한 탐지 상한(예: PII 10건 이상 발견)에 도달하면 나머지 파일을 스킵. 이미 "개인정보 포함 파일"로 분류되었으므로 추가 탐색이 불필요.
- **RegexSet 활용:** Rust `regex` 크레이트의 `RegexSet`은 여러 패턴을 단일 DFA로 병합하여 한 번의 패스로 검색. 패턴 8개를 개별 실행하는 것보다 2-4배 빠르다.

### 3.3 예상 반론에 대한 재반박

**반론 1: "키워드 없는 파일에서 구조화 PII가 발견되는 비율은 0.1% 미만이므로 무시할 수 있다."**

재반박: 0.1%라는 수치는 아직 실측되지 않은 추정값이다. 그리고 설령 0.1%라 하더라도, 50,000대 PC x PC당 100,000파일 = 50억 파일 중 0.1%는 **500만 파일**이다. 각 파일에 평균 3건의 PII가 있다면 **1,500만 건의 미탐지 개인정보**가 된다. 개인정보 유출 사고에서 건수는 과징금 산정에 직접 영향을 미친다.

더 중요한 것은 "비율"이 아니라 "존재 가능성"이다. 고객사의 보안 담당자에게 "키워드가 없어서 주민등록번호를 놓쳤습니다"라고 설명할 수 있는가? 이것은 제품 신뢰도의 문제이며, 한 번의 미탐 사고가 구독 해지로 이어질 수 있다.

**반론 2: "파일 타입별 차등 전략으로 해결할 수 있다 (로그/코드는 스킵, CSV/XLSX는 Regex 실행)."**

재반박: 파일 타입 차등 전략(Gemini의 "Context-Aware Funnel")은 합리적인 중간안이지만, 다음과 같은 문제가 있다:

1. **확장자 위변조:** `.txt` 확장자를 가진 CSV 덤프, `.log` 확장자를 가진 데이터 내보내기 파일 등, 확장자만으로 파일 유형을 정확히 판별할 수 없다.
2. **분류 기준의 유지보수:** "High-PII" vs "Low-PII" 확장자 목록을 지속적으로 관리해야 한다. 새로운 파일 형식이 등장할 때마다 분류 오류의 위험이 있다.
3. **구현 복잡도 증가:** 파일 타입별 분기 로직은 코드 복잡도를 높이고, 테스트 케이스를 늘리며, 버그 발생 가능성을 높인다.
4. **규칙의 예외가 규칙보다 복잡해짐:** "로그 파일이지만 사용자 활동 로그여서 이메일 주소가 포함된 경우", "소스코드이지만 테스트 데이터에 주민번호가 하드코딩된 경우" 등 예외가 끝없이 발생한다.

반면, "Regex/Checksum은 항상 실행"은 단순명료하다. 분기 없이 일관된 탐지 품질을 보장하며, 구현과 테스트가 간단하다.

**반론 3: "Regex를 항상 실행하면 CPU 사용량이 증가하여 Quiet Agent 철학에 어긋난다."**

재반박: Regex 실행이 CPU 사용량을 증가시키는 것은 사실이나, 리소스 거버너(Resource Governor)가 이미 합의된 설계 요소이다. CPU 임계치(예: 사용자 활동 중 15%, 유휴 시 50%)를 초과하면 스캔 속도를 조절하는 것이 리소스 거버너의 역할이다. Regex 실행으로 인한 추가 부하는 리소스 거버너가 스캔 속도(파일/초)를 낮추는 것으로 흡수된다. 사용자가 체감하는 것은 "스캔이 조금 더 오래 걸린다"이지, "PC가 느려진다"가 아니다.

## 4. 중립 심판 평가

### 4.1 입장 A의 강점과 약점

**강점:**
- 성능 최적화 효과가 명확하고 측정 가능하다. 파일의 70-85%를 Aho-Corasick 단일 패스로 제거하는 것은 전체 스캔 시간을 의미 있게 단축한다.
- "Quiet Agent" 철학과 직접적으로 일치한다. CPU 사용 시간 자체를 줄이는 것은 리소스 거버너로 속도를 조절하는 것보다 근본적인 해결책이다.
- 구현이 단순하다. Aho-Corasick 결과가 0이면 다음 단계 전체를 건너뛰는 이분법적 로직은 코드 경로가 명확하다.

**약점:**
- **미탐 위험이 구조적이다.** PoC에서 실증된 바와 같이, 키워드 없는 숫자 데이터에서 구조화 PII를 놓치는 것은 이론이 아니라 사실이다. 이 결함은 키워드 사전을 아무리 확장해도 완전히 해소되지 않는다. "모든 가능한 키워드"를 열거하는 것은 불가능하며, 키워드 없는 순수 숫자 데이터의 존재를 원천적으로 배제할 수 없다.
- **"0.1% 미만" 추정의 근거가 없다.** 실측 데이터 없이 빈도를 추정하는 것은 위험하다. 특히 기업마다 데이터 관행이 크게 다르므로(금융권은 숫자 덤프가 많고, 제조업은 적을 수 있다), 단일 비율로 일반화할 수 없다.
- **파일 타입 차등 전략은 부분적 보완에 불과하다.** Gemini의 "Context-Aware Funnel"은 합리적이지만, 확장자 기반 분류의 한계(위변조, 불완전 분류)를 안고 있어 근본적 해결이 아니다.

### 4.2 입장 B의 강점과 약점

**강점:**
- **미탐 위험을 원천 차단한다.** "Regex/Checksum은 항상 실행"이라는 단순한 규칙은, 키워드 유무와 무관하게 모든 구조화 PII를 탐지한다. 제품의 핵심 가치인 "정확한 탐지"를 보장하는 가장 확실한 방법이다.
- **PoC로 검증된 기술적 근거가 있다.** 주장이 아니라 실행 가능한 코드와 결과로 뒷받침된다.
- **구현의 단순성과 일관성.** 파일 타입별 분기, 키워드 사전 관리, 예외 처리 등의 복잡도가 없다. "1단계: Aho-Corasick -> NER 게이팅, 2단계: Regex/Checksum -> 항상 실행"이라는 명확한 구조.
- **비용 증가가 수용 가능한 범위다.** Rust `regex` 크레이트 + RegexSet + 스트리밍 + 조기 종료 최적화를 적용하면, Regex 상시 실행의 추가 비용은 파일당 수 밀리초 수준이다.

**약점:**
- **성능 비용이 0은 아니다.** 70-85%의 파일에 대해 불필요한 Regex를 실행하는 것은, 절대적 비용은 낮더라도 집합적으로는 무시할 수 없는 시간이다. PC당 20분의 추가 스캔 시간은 "Quiet Agent"의 관점에서 의미 있는 차이다.
- **저사양 PC에서의 영향이 불확실하다.** i3/4GB RAM 환경에서 Regex 상시 실행이 체감 성능에 미치는 영향은 실측이 필요하다. 이론적 추정만으로는 충분하지 않다.
- **"항상 실행"은 최적화의 여지를 닫는다.** 향후 성능 병목이 발견되었을 때, Regex 단계를 선택적으로 스킵하는 옵션을 처음부터 설계에 포함하지 않으면 리팩토링 비용이 발생한다.

### 4.3 핵심 트레이드오프 분석

이 쟁점의 본질은 **미탐 위험(정확도) vs 성능(사용자 경험)**의 트레이드오프다.

**정량적 분석:**

| 지표 | 입장 A (전체 스킵) | 입장 B (Regex 항상 실행) |
|:---|:---|:---|
| 구조화 PII 탐지율 | < 100% (키워드 없는 파일 미탐) | 100% (구조적으로 보장) |
| Regex 실행 파일 수 | 전체의 15-30% | 전체의 100% |
| PC당 추가 스캔 시간 | 기준 | +15-20분 (추정) |
| 구현 복잡도 | 낮음 | 낮음 |
| 유지보수 부담 | 키워드 사전 관리 필요 | 없음 |
| 미탐 사고 위험 | 존재 (빈도 미측정) | 없음 |
| 규제 대응 | 취약 | 안전 |

**비대칭 위험 분석:**

미탐과 성능 저하의 비용은 비대칭적이다:
- **미탐의 비용:** 고객사의 개인정보 유출 사고 -> 과징금 -> 제품 신뢰도 하락 -> 구독 해지 -> 브랜드 손상. 한 번의 사고가 치명적일 수 있다.
- **성능 저하의 비용:** 스캔 시간 20분 증가 -> 리소스 거버너로 완화 가능 -> 사용자 체감 영향 제한적. 점진적 최적화로 개선 가능하다.

이 비대칭성은 **안전측 선택(입장 B)**을 강하게 지지한다.

**"Quiet Agent"와의 양립 가능성:**

입장 B가 "Quiet Agent" 철학과 충돌한다는 주장은, 리소스 거버너의 존재를 간과한 것이다. CPU 사용률 제한은 리소스 거버너가 담당하며, 탐지 파이프라인의 역할은 "정확하게 탐지하는 것"이다. 스캔이 20분 더 걸리더라도 사용자 활동 시 CPU 15% 이하를 유지한다면, 사용자는 차이를 체감하지 못한다. "빠르지만 놓치는 스캐너"보다 "조금 느리지만 확실한 스캐너"가 제품 가치에 부합한다.

### 4.4 조건부 결론

**기본 결론: 입장 B(Regex/Checksum 항상 실행)를 채택한다.**

근거:
1. 미탐 위험의 비대칭적 비용 구조상 안전측 선택이 합리적이다.
2. Regex/Checksum의 추가 비용이 수용 가능한 수준이다 (파일당 수 ms).
3. 리소스 거버너가 사용자 체감 성능을 제어하므로, Regex 상시 실행이 "Quiet Agent"와 양립한다.
4. PoC에서 미탐 시나리오가 실증되었다.

**조건부 전환 기준:**

다음 조건이 **모두** 충족되면 입장 A(전체 스킵) 또는 Gemini의 파일 타입 차등 전략으로 전환을 검토한다:

1. **Phase 0 실측에서** 키워드 없는 파일의 구조화 PII 포함 비율이 **0.01% 미만**으로 확인됨 (0.1%가 아닌 0.01% — 더 엄격한 기준 적용).
2. **실측에서** Regex 상시 실행의 PC당 추가 스캔 시간이 **30분 이상**으로 확인됨 (성능 영향이 실제로 심각한 경우에만).
3. **고객 요구사항에서** 스캔 시간 단축이 탐지 정확도보다 우선순위로 명시됨.

## 5. 최종 추천안

**Codex안(입장 B)을 기본 아키텍처로 채택한다.**

구체적 설계:

```
파일 입력
  |
  v
[1단계] Aho-Corasick 키워드 프리필터 (O(n), 전 파일 대상)
  |
  |-- 키워드 발견 --> Regex/Checksum 실행 + NER 실행 대상으로 마킹
  |-- 키워드 미발견 --> Regex/Checksum 실행 + NER 스킵
  |
  v
[2단계] Regex/Checksum 구조화 PII 탐지 (항상 실행)
  |-- RegexSet으로 단일 패스 탐색
  |-- 스트리밍 + 조기 종료 최적화
  |-- 체크섬 검증 (주민번호, 카드번호 등)
  |
  v
[3단계] NER 비정형 PII 탐지 (키워드 발견 파일만)
  |-- Aho-Corasick 결과 기반 게이팅
  |-- AI 예산 + 하드캡 적용
```

**최적화 권고사항:**

1. **RegexSet 사용:** 모든 구조화 PII 패턴을 단일 `RegexSet`으로 병합하여 파일당 패스 횟수를 1회로 제한.
2. **스트리밍 스캔:** 대용량 파일(10MB 이상)은 64KB 청크 단위로 스트리밍 스캔하여 메모리 사용량 제한.
3. **조기 종료:** 탐지 건수가 정책 상한(예: 50건)에 도달하면 즉시 스캔 중단.
4. **리소스 거버너 연동:** Regex 실행 중에도 CPU 임계치를 준수하도록 파일 간 yield 포인트 삽입.

**Phase 0 검증 항목:**

| 측정 항목 | 목표 | 비고 |
|:---|:---|:---|
| Regex 상시 실행 시 파일당 평균 추가 시간 | < 50ms | Gemini 기준 차용 |
| 키워드 없는 파일의 구조화 PII 포함 비율 | 측정 (기준 없음) | 전환 판단 근거 |
| i3/4GB 환경에서 PC당 총 스캔 시간 증가분 | < 30분 | Quiet Agent 기준 |
| RegexSet vs 개별 Regex 성능 비교 | RegexSet 2x+ 빠름 | 최적화 효과 검증 |

**최종 요약:** Aho-Corasick 프리필터는 NER 호출 여부를 결정하는 게이팅 역할에 한정하고, Regex/Checksum은 모든 파일에 대해 항상 실행한다. 이것이 미탐 위험을 원천 차단하면서도 수용 가능한 성능 비용을 유지하는 최적의 균형점이다. Phase 0 실측 결과에 따라, 엄격한 조건 하에서만 선택적 스킵 전략으로의 전환을 검토한다.
