## 논점 1: NER 모델 선택

### Gemini 제안 요약
- **기본 모델:** KoELECTRA-Small-v3 (INT8 양자화, ~14MB, ~20ms/sentence)
- **고성능 옵션:** KLUE-RoBERTa-Base (GPU 환경용)
- GPU/CPU 이중 전략 제안. GLiNER(Korean)도 후보로 검토했으나 학습 데이터 부족으로 보류.
- F1 기준 KoELECTRA-Small이 크기 대비 최적이라는 결론.

### Codex 제안 요약
- **모델:** KoELECTRA-Base를 Fine-tuning하여 사용.
- 단, MVP에서는 NER을 **"beta" 기능**으로만 제공. 기본은 Regex+Checksum.
- NER 호출에 하드캡 설정 (분당 최대 추론 횟수 제한).
- Base 모델을 선택한 이유: Small의 정확도가 실제 기업 문서에서 충분하지 않을 수 있다는 우려.

### Claude 의견

**입장: Gemini에 부분 동의, Codex의 MVP 전략에 강하게 동의**

1. **모델 크기:** KoELECTRA-Small v3 INT8이 엔드포인트 배포의 현실적 선택이다. Base 모델은 ~110MB(FP32)이며, INT8로도 ~30MB 수준이다. 50,000대 PC에 배포할 때 모델 크기 차이는 업데이트 트래픽과 메모리 사용량에 직접 영향을 준다. Gemini의 선택이 운영 관점에서 합리적이다.

2. **정확도 현실:** 그러나 내 자체 리뷰에서 지적했듯이, KoELECTRA-Small의 공개 벤치마크 F1(~87%)은 **학술 데이터셋 기준**이다. 실제 기업 문서(엑셀 혼합 데이터, 비정형 주소, 약어 포함 이름)에서는 **F1 80-85% 수준으로 하락**할 가능성이 높다. Codex가 Base를 선택한 이유가 여기에 있으며, 이 우려는 타당하다.

3. **결론:** MVP 단계에서는 NER을 beta로 제한하고(Codex 동의), 모델은 Small로 시작하되(Gemini 동의), **파일럿 100-500대에서 Small vs Base의 실제 F1을 A/B 테스트**한 뒤 GA 모델을 결정하는 것이 올바른 순서다. 벤치마크 수치만으로 모델을 확정하는 것은 위험하다.

4. **추가 제안:** Fasoo의 AI-R Privacy가 이미 F1 93.1%를 달성했다고 공개한 상황에서, KoELECTRA 단독으로는 경쟁력이 부족하다. 자체 학습 데이터(합성 PII + 기업 문서 구조 반영)의 품질이 모델 선택보다 더 결정적인 변수가 될 것이다.

## 논점 2: 탐지 파이프라인 구조

### Gemini 제안 요약
- 4단계 퍼널: Metadata Filter → Fast Scanner(Regex/Aho-Corasick) → Validator(Checksum) → NER.
- 각 단계에서 점진적으로 후보를 줄이는 구조. NER에 도달하는 데이터를 최소화.
- Regex는 느슨하게(High Recall), Checksum으로 구조화 PII 확정, NER은 비정형만 처리.

### Codex 제안 요약
- Gemini와 유사한 다단계 하이브리드이나, **형식화 수준이 훨씬 높음.**
- Rust trait 기반 추상화: `TextExtractor`, `CandidateDetector`, `Verifier`, `Fusion`.
- YAML DSL로 탐지 규칙 정의 (정규식 패턴, 검증 함수, 신뢰도 가중치 등 선언적 관리).
- NER 게이팅: Regex가 발견한 후보 중 신뢰도 중간 영역만 NER에 전달. 하드캡 존재.
- 6단계 정밀도 제어 시스템 (Extract → Detect → Verify → Contextualize → Fuse → Policy).

### Claude 의견

**입장: 세 팀 모두 핵심 구조(저비용→고비용 필터링)에 합의. Codex의 형식화를 높이 평가하되, 과잉 설계 우려.**

1. **합의점:** Regex/Checksum 선행 → NER 후행이라는 파이프라인 구조는 세 팀 모두 동일하다. 이는 엔드포인트 리소스 제약을 고려하면 사실상 유일하게 합리적인 접근이며, 이 방향에 이견이 없다.

2. **Codex의 YAML DSL:** 탐지 규칙을 코드가 아닌 선언적 파일로 관리하는 것은 운영 시 규칙 업데이트를 바이너리 재배포 없이 할 수 있다는 강점이 있다. 그러나 YAML DSL은 자체적으로 파서, 검증기, 문서화가 필요한 **미니 언어**가 된다. MVP에서 이 수준의 추상화는 과잉이다. 초기에는 Rust 코드로 직접 규칙을 정의하고, 규칙이 20개 이상 누적되면 DSL 도입을 검토하는 것이 현실적이다.

3. **Codex의 Trait 추상화:** `TextExtractor`, `CandidateDetector` 등의 trait 분리는 아키텍처적으로 건전하다. 다만 Codex가 제안한 ~20개 crate 분리는 빌드 시간과 의존성 관리 부담을 과소평가한 것이다. 내 제안(10-12개 crate)이 MVP에는 더 적합하다.

4. **NER 격리:** 내 설계에서 제안한 **NER을 별도 자식 프로세스로 격리**하는 방식이 안정성 면에서 우월하다. ONNX Runtime이 크래시하더라도 메인 서비스에 영향을 주지 않는다. Gemini/Codex 모두 이 점을 명시적으로 다루지 않았다.

5. **"실시간" 정의:** Codex가 "실시간"을 SLO로 정의한 것(큐잉 30초, 탐지 5분)은 매우 실용적이다. "실시간=즉시"라는 마케팅 용어를 엔지니어링 관점에서 명확히 한 것에 동의한다. Gemini는 이 정의가 모호하다.

## 논점 3: 기술 스택 (Rust vs 대안)

### Gemini 제안 요약
- **클라이언트:** Rust (확정). `notify`, `windows-rs`, `ort` 크레이트 등 구체적 의존성 조사 완료.
- **서버:** Go 또는 Rust. Go를 API 개발 속도 이유로 선호.
- Rust의 Windows 서비스 개발, Minifilter 통신, USN Journal 파싱 등 기술 조사를 별도 문서로 수행.

### Codex 제안 요약
- **클라이언트:** Rust 기본이나, **팀 역량에 따라 C#/.NET 하이브리드도 현실적 대안**으로 제시.
- 핵심 엔진(Regex, NER 호출)만 Rust로, UI/서비스 래퍼는 C#으로 하는 혼합 구조 가능.
- **서버:** 팀 역량에 따라 결정. 구체적 스택 미확정.
- 가장 상세한 Cargo workspace 구조 제안 (~20 crates).

### Claude 의견

**입장: Rust 클라이언트에 강하게 동의. C#/.NET 하이브리드는 반대. 서버는 Go.**

1. **Rust 클라이언트:** 50,000대 PC에 배포되는 상주 프로세스에서 메모리 안전성과 성능 예측 가능성은 협상 불가능한 요구사항이다. C/C++은 메모리 버그 위험이 너무 높고, C#은 GC 정지(Stop-the-World)가 실시간 모니터링과 충돌한다. Rust가 사실상 유일한 합리적 선택이며, 세 팀 모두 이에 동의한다.

2. **C#/.NET 하이브리드 반대:** Codex의 "팀 역량 부족 시 C# 래퍼" 제안은 현실적인 우려에서 나왔지만, 기술적으로 **최악의 선택지**다.
   - Rust↔C# FFI 경계에서 메모리 관리, 에러 전파, 비동기 호출이 복잡해진다.
   - 두 언어의 빌드/테스트/배포 파이프라인을 모두 유지해야 한다.
   - Windows 서비스를 C#으로 만들면 .NET Runtime 의존성이 추가되어 설치 크기가 증가한다.
   - **대안:** Rust 숙련도가 부족하면, 차라리 전체를 C#으로 하거나(성능 타협), Rust 전문가를 충원하는 것이 낫다. 중간은 양쪽의 단점을 모두 가져온다.

3. **서버 Go:** API 서버, gRPC, PostgreSQL 연동 등 서버 업무는 Go의 강점 영역이다. Rust 서버는 개발 속도가 느리고, 이 프로젝트에서 서버 성능이 병목이 될 가능성은 낮다. Gemini의 Go 선택에 동의하며, 내 설계에서도 Go + Fiber 조합을 제안했다.

4. **Cargo workspace 규모:** Codex의 ~20 crate는 과도하다. 내 제안(10-12 crate)도 충분히 모듈화되어 있으며, crate 경계가 너무 세분화되면 순환 의존성과 빌드 시간 문제가 발생한다. 초기에는 더 적은 crate로 시작하고, 실제 필요에 따라 분리하는 것이 Rust 생태계의 관행이다.

## 논점 4: MVP 범위

### Gemini 제안 요약
- MVP 범위를 명시적으로 정의하지 않음. 전체 기능을 설계 수준에서 다룸.
- Review에서 "프로토타이핑 즉시 시작"을 권고했으나 구체적 MVP 스코프 미제시.
- "Conditional Go" 판정 — i3/4GB에서 NER 추론 100ms 초과 시 전략 재검토.

### Codex 제안 요약
- **가장 보수적이고 구체적인 MVP 정의.**
- Phase 0 (2-4주): PoC — 단일 PC, Regex만, CLI 출력.
- Phase 1 MVP (6-10주): 100-500대, Regex+Checksum, 기본 서버 대시보드, MSI 설치.
- Phase 2 Beta (8-12주): NER beta 추가, 실시간 모니터링, 라이선스.
- NER과 실시간 탐지 모두 MVP에서 제외. MVP = 구조화 PII + 증분 스캔 + 기본 리포팅.

### Claude 의견

**입장: Codex의 단계별 접근에 강하게 동의. 단, Phase 0을 더 공격적으로 활용해야 한다.**

1. **MVP에서 NER 제외:** 내 자체 리뷰에서도 동일한 결론을 내렸다. NER은 학습 데이터 구축, 모델 Fine-tuning, 양자화 검증, 실제 문서 F1 측정까지 **독립적인 R&D 트랙**이다. MVP의 핵심 가치는 "안정적으로 50,000대에 배포되어 구조화 PII를 정확히 찾는 에이전트"이며, NER이 없어도 이 가치는 성립한다.

2. **MVP에서 실시간 탐지 제외:** 내 리뷰에서 가장 과감한 제안이었고, 여전히 유효하다. `ReadDirectoryChangesW`는 Gemini의 리뷰가 지적했듯 이벤트 유실 위험이 있고, USN Journal 파싱은 NTFS 버전별 복잡도가 높다. MVP는 **스케줄 기반 증분 스캔**(마지막 스캔 이후 변경된 파일만)으로 충분하다. 실시간은 Phase 2에서 추가하는 것이 안전하다.

3. **Phase 0 강화:** Codex의 Phase 0(2-4주, CLI PoC)에서 다음을 반드시 검증해야 한다:
   - HWP 파일 파싱 성공률 (hwp-rs 크레이트의 성숙도가 매우 낮음)
   - XLSX/DOCX 내 한국어 텍스트 추출 정확도
   - KoELECTRA-Small INT8 추론 시간 (i3/4GB 실측)
   - 이 세 가지 중 하나라도 실패하면 전체 기술 방향이 바뀔 수 있으므로, Phase 0에서 **Kill/Pivot 기준을 명확히 설정**해야 한다.

4. **대상 규모:** Codex의 100-500대 파일럿이 적절하다. 50,000대 전체 배포를 MVP 목표로 잡으면 인프라(서버 스케일링, 인증서 관리, 업데이트 배포)에 공수가 분산된다.

## 논점 5: 보안/라이선스 설계

### Gemini 제안 요약
- mTLS로 모든 에이전트 인증. 조직별 또는 머신별 인증서.
- JWT 기반 라이선스 (Ed25519 서명). 오프라인 검증 가능.
- 바이너리 보호: `obfstr` 크레이트, 심볼 스트리핑, 안티디버깅.
- 자체 Review에서 "50,000개 인증서 관리는 운영 지옥"이라고 자체 비판. Join Token + JWT 하이브리드 제안.

### Codex 제안 요약
- **가장 상세한 PKI 설계.** Bootstrap Token → CSR → 인증서 발급 전체 라이프사이클.
- 라이선스: Signed Lease 모델. 주기적 체크인 + 하드 데드라인. 오프라인 유예 기간.
- 데이터 토큰화: Device-scoped HMAC vs Tenant-scoped HMAC 분리.
- 미래 대비: OPRF/VOPRF 언급.
- PIPA(개인정보보호법) 컴플라이언스 상세 매핑.

### Claude 의견

**입장: Codex의 설계가 가장 포괄적이나, MVP에는 과잉. Gemini Review의 "간소화" 권고에 동의.**

1. **mTLS 범위 축소:** Gemini의 자체 리뷰와 내 설계 모두 동일한 결론이다 — 50,000대 엔드포인트에 개별 클라이언트 인증서를 발급·갱신·폐기하는 것은 전담 PKI 인프라가 필요한 작업이다. MVP에서는:
   - **설치 시:** 1회용 Join Token으로 등록.
   - **세션:** API Token (JWT) + Refresh Token 방식.
   - **mTLS:** 서버 간 통신에만 적용.
   - Phase 2 이후 클라이언트 인증서를 선택적으로 도입.

2. **라이선스:** Codex의 Signed Lease 모델은 건전하지만, MVP에서는 **단순 JWT 라이선스 + 온라인 활성화**로 충분하다. 오프라인 유예, 하드 데드라인, Lease 갱신 로직은 엣지 케이스가 많아 초기 구현 비용이 높다.

3. **내 고유 제안 — Fuzzy HW Fingerprinting:** 라이선스 복제 방지를 위해 내 설계에서 제안한 가중치 기반 하드웨어 핑거프린팅(CPU 25%, 마더보드 25%, 디스크 20%, MAC 15%, SID 10%, BIOS 5%)은 Gemini/Codex 모두 다루지 않은 영역이다. 단순 Machine ID 대신 부분 하드웨어 변경(RAM 추가, NIC 교체)에도 라이선스가 유지되는 유연성이 있다. 이 방식은 기존 15년 운영 경험에서 가장 빈번한 고객 불만("PC 부품 바꿨더니 라이선스 풀렸다")에 대한 실질적 해결책이다.

4. **바이너리 보호:** Gemini가 제안한 `obfstr`, 안티디버깅 등은 상용 솔루션에서 기본이다. 다만 이것은 **"속도 저하"** 수준의 보호이며, 결정적 방어는 아니다. 라이선스 검증 로직의 핵심은 서버측 체크인에 있어야 하며, 클라이언트측 검증은 보조 수단으로만 취급해야 한다.

## 논점 6: 구현 실현 가능성 (팀 규모/일정)

### Gemini 제안 요약
- 구체적 팀 규모/일정 미제시. 설계 문서에 집중.
- Review에서 "i3/4GB에서 NER 100ms 초과 시 전략 재검토" — 기술 실현 가능성에 집중.
- 운영 현실(인증서 관리, 업데이트 배포)에 대한 우려는 있으나 공수 추정 없음.

### Codex 제안 요약
- **가장 상세한 실현 가능성 분석.**
- 컴포넌트별 난이도 평가 (텍스트 추출: 높음, HWP 파싱: 매우 높음, Minifilter: 매우 높음).
- 4단계 페이즈 (Phase 0~3), 각 단계별 주 단위 추정.
- Rust 숙련도 부족 시 C#/.NET 대안까지 고려.
- MVP 대상: 100-500대.
- 총 예상 기간: Phase 0+1만 8-14주 (PoC~MVP).

### Claude 의견

**입장: Codex의 분석이 가장 현실적. 추가로 핵심 리스크 3가지를 강조한다.**

1. **총 공수 추정:** 내 리뷰에서 **23-35 인월(person-months)**로 추정했다. Rust 클라이언트(10-15), 서버/대시보드(8-12), NER R&D(5-8) 기준이다. 4-5명 팀 기준 6-9개월이며, 이는 Codex의 Phase 0+1+2 합산(16-26주 ≈ 4-6.5개월)보다 보수적이다. Codex는 **서버/대시보드 개발 공수를 과소평가**한 것으로 보인다.

2. **핵심 리스크 #1 — HWP 파싱:** 한국 기업 환경에서 HWP는 필수 지원 포맷이다. 그러나 `hwp-rs` 크레이트는 미성숙하며, HWP 5.x 포맷의 OLE compound document 구조는 복잡하다. **Phase 0에서 HWP 파싱 실패 시, C 라이브러리(libhwp) FFI 또는 Python subprocess 폴백을 준비해야 한다.** 이것만으로 2-4주가 추가될 수 있다.

3. **핵심 리스크 #2 — NER 학습 데이터:** 공개 한국어 NER 데이터셋(AI Hub modu corpus)은 뉴스/위키 도메인이다. 기업 문서(급여명세서, 계약서, 엑셀 고객 리스트)의 PII 패턴과 차이가 크다. **합성 데이터 생성 + 도메인 적응에 최소 4-6주가 필요**하며, 이는 Gemini/Codex 모두 과소평가한 부분이다.

4. **핵심 리스크 #3 — Windows 서비스 안정성:** 50,000대 PC의 하드웨어/OS 다양성(Windows 10 1809~22H2, Windows 11, 32bit 레거시)에서 Rust 바이너리의 호환성을 보장하려면 광범위한 QA가 필요하다. Gemini의 리뷰가 지적한 "듀얼코어 노트북에서 5% CPU도 체감된다"는 경고는 과장이 아니다. **CI에 다양한 Windows 버전 매트릭스 테스트가 필수**이며, 이 인프라 구축만으로도 1-2주가 소요된다.

## 논점 7: 경쟁 차별화 전략

### Gemini 제안 요약
- **"Zero-Interruption Performance":** Rust 기반 경량 에이전트로 사용자 업무 무중단.
- **"Context-Aware Precision":** AI(NER)로 오탐률 대폭 감소.
- 경쟁사 분석: Somansa(Privacy-i), Comtrue Technology, Fasoo, Jiran Security.
- Rust + AI 조합을 기술적 해자(moat)로 포지셔닝.

### Codex 제안 요약
- 차별화 전략을 명시적으로 다루지 않음. 기술 실현 가능성에 집중.
- 암묵적 차별화: 가장 상세한 엔지니어링 설계 자체가 실행력의 증거.
- MVP를 극도로 좁혀서 "실제 동작하는 제품"을 빠르게 만드는 것이 경쟁 전략.

### Claude 의견

**입장: Gemini의 차별화 방향에 부분 동의하나, AI를 핵심 USP로 내세우는 것은 위험. 대안 제시.**

1. **Fasoo의 선점:** 내 리뷰에서 가장 강하게 경고한 부분이다. Fasoo의 AI-R Privacy는 이미 **F1 93.1%를 공개**했고, 대기업 레퍼런스를 확보 중이다. "AI로 오탐 줄인다"는 메시지는 **후발주자가 선점자와 동일한 전장에서 싸우는 것**이며, 특히 우리의 NER이 아직 PoC도 없는 상태에서 이를 핵심 USP로 마케팅하면 신뢰를 잃을 수 있다.

2. **현실적 차별화 축:**
   - **경량 에이전트:** Gemini의 "Zero-Interruption"에 동의한다. 기존 솔루션(Privacy-i 등)은 C++/C# 기반으로 풀스캔 시 CPU 50%+ 점유가 보고된다. Rust 기반 에이전트가 5% 이하를 유지한다면 **체감 가능한 차이**다. 이것은 15년 운영 경험에서 가장 빈번한 고객 불만("스캔 돌리면 PC가 느려진다")에 대한 직접 해결이며, PoC 없이도 주장 가능한 구조적 강점이다.
   - **Privacy by Design:** 서버에 원본 PII를 전송하지 않는 아키텍처(메타데이터 + 마스킹만 전송)는 PIPA/GDPR 컴플라이언스에서 강력한 차별점이다. 기존 솔루션 대부분은 중앙 서버에 탐지된 PII 원문을 저장하는데, 이 자체가 개인정보 유출 리스크다.
   - **운영 효율성:** 50,000대 운영 경험 기반의 자동 업데이트, 장애 자동 복구(Panic Mode), 정책 원격 배포 등은 기술 스타트업이 쉽게 따라올 수 없는 영역이다.

3. **AI 포지셔닝 수정:**
   - MVP: "정규식+검증 알고리즘으로 구조화 PII 탐지. 오탐률 < 5%."
   - Phase 2: "AI 기반 비정형 PII 탐지 (이름, 주소) — Beta."
   - Phase 3: "AI 컨텍스트 분석으로 오탐률 < 3% — GA."
   - AI는 로드맵의 **성장 스토리**로 활용하되, Day 1 USP로는 사용하지 않는 것이 안전하다.

## 논점 8 (추가): 텍스트 추출 전략의 과소평가

### 세 팀의 공통 맹점

Gemini, Codex, Claude 모두 탐지 엔진과 NER 모델에 집중했으나, **파일에서 텍스트를 추출하는 단계**를 상대적으로 가볍게 다뤘다.

### Claude 의견

**입장: 텍스트 추출이 전체 파이프라인의 가장 큰 기술 리스크 중 하나다.**

1. **HWP:** 이미 논점 6에서 언급. `hwp-rs`는 미성숙하고, HWP 5.x의 바이너리 포맷은 리버스 엔지니어링에 가깝다.

2. **XLSX/DOCX:** Rust 생태계의 `calamine`(XLSX 읽기)과 `docx-rs`는 기본 텍스트 추출에는 적합하지만, **셀 내 줄바꿈, 병합 셀, 하이퍼링크 텍스트, 주석(comment)** 등에서 PII가 숨어 있는 경우를 처리하려면 추가 작업이 필요하다.

3. **PDF:** PDF 텍스트 추출은 악명 높은 난제다. `pdf-extract` 크레이트의 한국어 지원 수준이 불확실하며, 스캔 PDF는 OCR이 필요하다(Phase 2로 미뤄야 함).

4. **ZIP/암호화:** 압축 파일 내부 스캔, 암호화된 문서 처리 정책 등도 MVP에서 범위를 명확히 해야 한다.

5. **제안:** Phase 0에서 지원 대상 포맷별 텍스트 추출 성공률을 **정량 측정**하고, 성공률 90% 미만 포맷은 MVP 지원에서 제외하는 기준을 세워야 한다. "지원한다"와 "정확히 추출한다"는 완전히 다른 문제다.

## 논점 9 (추가): DRM 연동 전략

### 세 팀의 차이

- **Gemini:** DRM을 언급했으나 구체적 설계 없음.
- **Codex:** DRM을 MVP 이후로 미룸. 실현 가능성 리뷰에서 난이도 "매우 높음"으로 평가.
- **Claude:** DRM 벤더 어댑터 아키텍처를 설계 (Fasoo, SoftCamp, MarkAny 등 벤더별 SDK 연동 trait 정의).

### Claude 의견

**입장: DRM 연동은 한국 시장에서 필수이나, MVP에서는 "DRM 파일 감지만" 하고 내용 스캔은 Phase 2 이후.**

1. **한국 시장 현실:** 대기업/공공기관의 80%+ 이상이 문서 DRM(Fasoo, SoftCamp 등)을 사용한다. DRM이 걸린 파일을 스캔할 수 없으면 "개인정보가 없다"고 보고하게 되어, 솔루션의 신뢰도가 치명적으로 손상된다.

2. **기술 난이도:** 각 DRM 벤더의 SDK는 NDA 하에 제공되며, API가 불안정하고 문서화가 부족한 경우가 많다. 벤더당 연동에 4-8주가 소요될 수 있다.

3. **MVP 전략:**
   - DRM 파일 여부만 감지 (파일 헤더/매직넘버로 판별).
   - 스캔 결과에 "DRM 보호됨 — 내용 스캔 불가" 표시.
   - Phase 2에서 주요 벤더(Fasoo, SoftCamp) SDK 연동 시작.
   - 이렇게 하면 MVP에서도 "DRM 파일이 몇 개 있는지"는 보고할 수 있어, 관리자에게 가시성을 제공한다.

1. **즉시 실행:** Phase 0 PoC (2-4주) — HWP 파싱, 텍스트 추출, NER 추론 속도 실측. Kill/Pivot 기준 사전 정의.

4. **기술 검증 우선순위:** (1) HWP/PDF 텍스트 추출 → (2) NER F1 실측 → (3) Windows 서비스 안정성.
