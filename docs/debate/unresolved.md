## 미합의 1: Aho-Corasick 프리필터의 적용 범위

- **대립하는 입장들:**
  - **입장 A (Gemini → Claude 수용):** 키워드("주민", "전화", "카드", 숫자 패턴 등)가 전혀 없는 파일은 **Regex 단계조차 건너뛴다.** 코드/로그 파일 등 PII가 없는 파일을 대량으로 걸러내어 성능 최적화.
  - **입장 B (Codex):** 키워드 없으면 전체 스킵은 **구조화 PII(주민번호, 카드번호, 이메일, 전화번호)를 놓친다.** 키워드는 NER 게이팅에만 사용하고, Regex/Checksum은 항상 실행해야 한다. 예: 엑셀에 숫자만 나열된 경우 한국어 키워드가 없지만 주민번호가 있을 수 있음.
- **합의 실패 이유:** Gemini의 제안은 성능 최적화에 효과적이나, Codex의 반론(키워드 없는 숫자 데이터에서 구조화 PII 누락)이 기술적으로 타당하다. Claude가 Gemini 제안을 수용했으나 Codex 반론을 검토하지 못함.
- **추가 정보가 있으면 결론 가능한가:** Phase 0에서 실제 기업 문서 코퍼스를 대상으로 "키워드 없는 파일에 구조화 PII가 포함된 비율"을 측정하면 결론 가능. 0.1% 미만이면 Gemini안(전체 스킵) 채택, 그 이상이면 Codex안(NER 게이팅만) 채택.
- **사용자 결정이 필요한가:** 아니오. Phase 0 데이터로 결정 가능.
- **추천:** **Codex안을 기본으로 채택** (안전측). Regex/Checksum은 비용이 낮으므로 항상 실행하고, Aho-Corasick은 NER 호출 여부 결정에만 사용. 성능 문제가 실측으로 확인되면 Gemini안으로 전환.

## 미합의 2: NER 모델 로드/언로드 전략

- **대립하는 입장들:**
  - **입장 A (Claude):** "스캔 세션 단위 로드/언로드". 스케줄 스캔 시작 시 로드, 완료 후 언로드. Phase 2 실시간에서는 debounce window(30초) 내 이벤트 누적 후 배치 로드, idle 5분 후 언로드. 4GB PC에서 비스캔 시간 메모리 회수.
  - **입장 B (Codex):** thrashing 위험 경고만 하고 대안은 명시하지 않음. "AI 예산 + 하드캡"으로 호출 횟수를 제한하는 것은 합의했으나, 모델 자체의 메모리 상주 정책은 미확정.
  - **입장 C (Gemini):** 이 주제를 깊이 다루지 않음.
- **합의 실패 이유:** MVP에서 NER이 베타/OFF이므로 당장의 우선순위가 낮아 논의가 깊어지지 않음. Phase 2에서 실시간 NER을 도입할 때 구체화 필요.
- **추가 정보가 있으면 결론 가능한가:** Phase 0에서 i3/4GB 환경의 ONNX Runtime 로드/언로드 시간을 실측(예상: 500ms-2초). 로드 비용이 2초 이상이면 상주 + 메모리 제한이 유리. 500ms 미만이면 세션 단위 로드/언로드가 합리적.
- **사용자 결정이 필요한가:** 아니오. 실측 데이터로 결정 가능.
- **추천:** Claude안(세션 단위)을 MVP 기본으로 하되, Phase 2에서 ONNX Runtime 로드 시간 실측 후 최종 결정.

## 미합의 3: 하드웨어 핑거프린팅 구현 방식

- **대립하는 입장들:**
  - **입장 A (Claude):** 퍼지(fuzzy) 가중치 핑거프린팅 유지. 단, 용도를 "라이선스 바인딩"에서 **"이상 탐지 신호"**로 격하. 유사도 임계치 미달 시 즉시 차단이 아니라 서버에 "재검증 필요" 플래그. WMI 실패 항목은 가중치 0. 콘솔 재바인딩 UX 필수.
  - **입장 B (Codex):** 퍼지 핑거프린팅보다 **TPM Non-exportable key(또는 mTLS 개인키)**가 1차 바인딩으로 더 신뢰성 있음. TPM 불가 환경에서만 "퍼지 fingerprint + 콘솔 재바인딩 UX"를 폴백으로 제공.
  - **입장 C (Gemini):** 이 주제를 깊이 다루지 않음.
- **합의 실패 이유:** TPM 가용성에 대한 현실 데이터가 없음. 한국 기업 PC 환경에서 TPM 2.0 활성화 비율이 높으면 Codex안이 우월하고, 낮으면 Claude안이 더 현실적.
- **추가 정보가 있으면 결론 가능한가:** 파일럿 고객(100-500대)의 TPM 2.0 활성화 비율을 조사하면 결론 가능.
- **사용자 결정이 필요한가:** **예 — 부분적.** 기존 15년 운영 환경의 TPM 활성화 비율을 사용자가 알고 있을 가능성이 높음. 이 데이터를 기반으로 결정해야 한다.
- **추천:** **Codex안(TPM 우선 + 퍼지 폴백)을 기본 구조**로 하되, 구현은 TPM 가용성 데이터 확인 후 Phase 1에서. MVP에서는 Device ID(machine GUID) + 서버측 집행으로 단순 시작. 퍼지 핑거프린팅은 Phase 2.

## 미합의 4: 모델 파일 보호 수준 (암호화 vs 서명)

- **대립하는 입장들:**
  - **입장 A (Gemini):** 모델 파일 디스크 **암호화 필수**(at-rest encryption). 메모리 로드 시에만 복호화. IP 보호 목적.
  - **입장 B (Codex):** 암호화보다 **서명 검증(무결성)이 우선**. 암호화 at-rest는 IP 보호 효과가 제한적(메모리 덤프 가능). MVP 우선순위: 서명 검증 > 배포/업데이트 체인 > 옵션으로 at-rest 보호.
  - **입장 C (Claude):** 부분 수용. AES-256-GCM 암호화 + DPAPI 키 보호를 하되, 복호화 실패 시 NER 비활성화(Regex-only) 폴백. mlock/zeroize/honey-pot은 MVP 제외.
- **합의 실패 이유:** 세 팀 모두 "서명 검증은 필수"에 동의하나, "at-rest 암호화가 MVP에 필요한가"에서 갈림. Gemini=필수, Codex=옵션, Claude=하되 폴백 필수.
- **추가 정보가 있으면 결론 가능한가:** 경쟁사(Fasoo AI-R Privacy 등)의 모델 보호 수준을 조사하면 "시장 기대치"를 파악할 수 있음.
- **사용자 결정이 필요한가:** **예.** 모델 IP 보호의 비즈니스 중요도는 기술적으로 결정할 수 없음. 자체 학습 모델의 유출이 경쟁 위험으로 판단되면 암호화 필수, 범용 공개 모델 기반이면 서명만으로 충분.
- **추천:** MVP에서는 **서명 검증 필수 + at-rest 암호화는 옵션(Codex안)**. Phase 2에서 자체 fine-tuned 모델을 배포할 때 암호화를 기본으로 격상. 복호화 실패 시 Regex-only 폴백은 어느 경우든 필수.

## 미합의 5: C#/.NET 하이브리드의 완전 배제 여부

- **대립하는 입장들:**
  - **입장 A (Claude + Gemini):** C#/.NET 하이브리드는 **완전 배제.** FFI 복잡도, 이중 빌드 파이프라인, .NET Runtime 의존성 등 득보다 실이 크다. Rust 숙련도가 부족하면 전체 C# 또는 Rust 전문가 충원.
  - **입장 B (Codex):** FFI 하이브리드는 비추천에 동의. 그러나 "C# 서비스 + Rust 워커" 조합은 **프로세스 경계(IPC, Named Pipe)**로만 연결하면 현실적 예외안으로 유효. 팀이 .NET에 강하고 Rust에 약할 때 MVP 속도를 위한 선택지.
- **합의 실패 이유:** 기술적으로는 "IPC로만 연결하면 가능"이지만, 실제로 이 경로를 택하면 두 언어 생태계를 유지해야 하는 운영 부담이 남는다. 팀 역량에 대한 정보 없이는 판단 불가.
- **추가 정보가 있으면 결론 가능한가:** 팀의 Rust 숙련도를 확인하면 결론 가능.
- **사용자 결정이 필요한가:** **예.** 팀 구성과 기술 역량은 사용자만 판단 가능.
- **추천:** **기본안 = 전체 Rust(Claude/Gemini안).** Rust 숙련도가 심각하게 부족하고 .NET 전문가가 다수인 극단적 경우에만 Codex의 IPC 예외안 검토. 이 경우에도 MVP 이후 Rust 통합을 로드맵에 포함.

## 미합의 6: 정확한 공수/일정 추정

- **대립하는 입장들:**
  - **입장 A (Claude):** 전체 제품 = **30-45 인월**(제품화 버퍼 +30% 포함). 4-5명 기준 6-9개월(MVP까지). Codex는 서버/대시보드 공수를 과소평가.
  - **입장 B (Codex):** Phase 0(2-4주) + Phase 1 MVP(6-10주) = **8-14주**로 파일럿 가능. 단, "상용급"이 아닌 "파일럿 운영 가능" 수준. 서버는 모놀리스 + 최소 콘솔로 공수 축소.
  - **입장 C (Gemini):** 구체적 공수 미제시. Phase 0(2주) 검증 후 조정 제안.
- **합의 실패 이유:** 추정 스코프가 다르다. Claude는 "MVP + 서버/대시보드 + 제품화"를 포함한 전체 추정이고, Codex는 "최소 파일럿"만의 추정. 같은 대상을 다른 범위로 추정하고 있어 비교 자체가 부정확.
- **추가 정보가 있으면 결론 가능한가:** Phase 0 완료 후 실측 데이터(추출기 개발 시간, 서버 API 수, 콘솔 화면 수)가 나오면 정밀 추정 가능.
- **사용자 결정이 필요한가:** **예.** "MVP"의 정의(최소 파일럿 vs 유료 판매 가능 제품)를 사용자가 확정해야 공수 추정이 수렴.
- **추천:** **Codex의 Phase 정의 + Claude의 보수적 버퍼**를 결합.
  - Phase 0: 2-4주 (검증)
  - Phase 1 MVP(파일럿): 8-12주 (Codex 추정 상한)
  - Phase 1 → 유료 판매 가능 수준: +4-8주 (서버 안정화, 콘솔 완성, 보안 감사)
  - **총 MVP(판매 가능): 14-24주 (4-5명 기준)**

## 미합의 7: 규칙 정의 형식의 도입 시점

- **대립하는 입장들:**
  - **입장 A (Claude):** MVP에서는 **Rust 코드로 직접 규칙 정의.** 규칙이 20개 이상 누적되면 DSL/스키마 도입 검토. YAML DSL은 파서/검증기/문서화가 필요한 미니 언어가 되므로 MVP에서 과잉.
  - **입장 B (Codex):** MVP에서도 **JSON/YAML 스키마(열거형만, 표현식 금지)**로 규칙 데이터를 관리. 규칙이 코드에 하드코딩되면 바이너리 재배포 없이 규칙 업데이트 불가.
  - **입장 C (Gemini):** 이 주제를 깊이 다루지 않음.
- **합의 실패 이유:** "서명된 룰 번들 업데이트"에는 전원 동의하나, 그 번들의 형식(Rust 코드 vs JSON/YAML)에서 갈림. Codex의 "열거형만 허용" 제약이 Claude의 "미니 언어 우려"를 상당 부분 해소하지만, 완전한 합의에는 이르지 못함.
- **추가 정보가 있으면 결론 가능한가:** Phase 0에서 초기 규칙 수(한국 PII 유형)를 확정하면 판단 가능. 10개 미만이면 코드 직접 정의가 합리적, 15개 이상이면 스키마 도입이 유리.
- **사용자 결정이 필요한가:** 아니오. 기술적으로 결정 가능.
- **추천:** **Codex안(열거형 전용 JSON 스키마)을 MVP부터 채택.** 이유: 한국 PII 유형(주민/사업자/카드/전화/이메일/계좌/여권/운전면허/건강보험 등)이 이미 10개를 넘으며, 고객사별 커스텀 규칙(사내 ID 포맷 등)이 추가될 것이 확실. 다만 표현식/조건문은 금지하고, 규칙 추가/수정은 서명된 번들로만 배포.

## 미합의 8: MVP에서의 실시간 감지 세부 범위

- **대립하는 입장들:**
  - **입장 A (Claude R1 → R2 수정):** Round 1에서 "실시간 완전 제외" → Round 2에서 Gemini안 수용하여 **3개 폴더(바탕화면/문서/다운로드) 한정 ReadDirectoryChangesW + 폭주 시 eventual mode 전환.** 나머지는 USN 증분만.
  - **입장 B (Codex):** 핵심 폴더 워처는 동의하나, 워처의 역할을 **"즉시 스캔"이 아니라 "큐잉까지만"**으로 제한. 실제 스캔 타이밍은 리소스 거버너가 CPU/사용자 활동 기반으로 결정.
  - **입장 C (Gemini):** 핵심 경로(바탕화면, 문서)만 실시간 감시. 나머지는 스케줄 증분. *(Round 3 추가)*: Gemini는 **Codex의 "큐잉 + 리소스 거버너" 방식을 전적으로 지지**한다. 사용자가 파일을 저장하는 순간(Write)은 가장 방해받고 싶지 않은 순간이며, "Quiet Agent" 차별화를 위해 이벤트를 큐에 넣고 시스템이 Idle/저부하 상태일 때 처리해야 한다는 입장.
- **합의 실패 이유:** "핵심 폴더 워처"에는 합의했으나, 워처 이벤트 수신 후 **즉시 스캔인지, 큐잉 후 리소스 여유 시 스캔인지**가 미확정. *(Round 3 업데이트: Gemini가 Codex안을 명시적으로 지지하여 사실상 Codex+Gemini vs Claude 구도가 됨)*
- **추가 정보가 있으면 결론 가능한가:** Phase 0에서 "핵심 3폴더의 평균 이벤트 빈도(파일 수/분)"를 측정하면 판단 가능. 빈도가 높으면(분당 10+) 큐잉 필수, 낮으면(분당 1-2) 즉시 스캔도 가능.
- **사용자 결정이 필요한가:** 아니오. Phase 0 데이터로 결정 가능.
- **추천:** **Codex안(큐잉 + 리소스 거버너)을 채택.** 즉시 스캔은 사용자가 문서 저장 중 CPU 스파이크를 유발할 수 있으며, 이는 "Quiet Agent" 차별화와 충돌. 이벤트 수신 → 큐 적재 → idle/저부하 시 스캔이 UX 관점에서 안전.
