# Deep Dive 3: 하드웨어 핑거프린팅 구현 방식

**토론 일자:** 2026-02-11
**참여자:** 입장A 옹호자, 입장B 옹호자, 중립 심판

---

## 1. 배경 및 쟁점 요약

PII Scanner는 구독형 라이선스 모델로 기업에 판매되며, 수만 대 규모의 PC에 클라이언트를 배포한다. 라이선스를 특정 PC에 바인딩하기 위해 하드웨어 핑거프린팅이 필요하다. 쟁점은 이 바인딩의 **1차 메커니즘**을 무엇으로 삼을 것인가이다.

**핵심 쟁점:**
- **입장 A:** 퍼지(fuzzy) 가중치 핑거프린팅을 1차 메커니즘으로 사용하되, 용도를 "라이선스 바인딩"이 아닌 "이상 탐지 신호"로 격하하여 운용한다. 유사도 임계치 미달 시 즉시 차단하지 않고 서버에 "재검증 필요" 플래그를 전송한다.
- **입장 B:** TPM 2.0의 Non-exportable Key(또는 mTLS 개인키)를 1차 바인딩으로 사용하고, TPM 불가 환경에서만 퍼지 핑거프린트를 폴백으로 제공한다.

**합의 실패 이유:** 한국 기업 PC 환경에서 TPM 2.0 활성화 비율에 대한 현실 데이터가 부족하다. TPM 가용률이 높으면 입장 B가 우월하고, 낮으면 입장 A가 더 현실적이다.

**고려 요소:**
- 한국 기업 Windows 11 전환율 및 TPM 2.0 활성화 현황
- WMI 기반 하드웨어 정보 수집의 신뢰성
- 하드웨어 변경 시나리오 (RAM 증설, 디스크 교체 등)
- VM/VDI 환경 지원
- 각 접근법의 보안 강도
- 구현 복잡도 및 운영 오버헤드
- 하드웨어 변경 시 사용자 마찰

---

## 2. 입장 A 옹호 (퍼지 가중치 핑거프린팅)

### 2.1 핵심 논거

1. **현실적 호환성:** 한국 기업 환경에서 TPM 2.0 가용률은 아직 보편적이지 않다. 2025년 10월 Windows 10 지원 종료 이후에도 상당수 기업이 Windows 10을 유지하고 있으며, 특히 중소기업은 전환에 수년이 소요될 수 있다. 글로벌 데이터 기준으로도 테스트된 워크스테이션 중 약 74.8%만이 TPM 2.0을 지원한다. 이는 최소 25%의 엔드포인트에서 폴백이 필요하다는 의미이며, 폴백이 곧 주요 경로가 되는 상황이다.

2. **단일 메커니즘의 단순성:** 모든 환경에서 동일한 메커니즘이 작동하므로, 코드 경로가 하나로 통일된다. TPM 경로와 퍼지 경로 두 가지를 모두 구현·테스트·유지보수할 필요가 없다.

3. **용도 격하를 통한 실용성:** 핑거프린트를 "차단 기준"이 아닌 "이상 탐지 신호"로 사용함으로써, 하드웨어 변경으로 인한 오차단 문제를 원천 해소한다. 서버 측에서 "재검증 필요" 플래그를 통해 관리자가 판단할 수 있다.

4. **사용자 마찰 최소화:** RAM 증설, SSD 교체 같은 일상적 하드웨어 변경에서 즉시 차단이 발생하지 않아 사용자 업무 중단이 없다.

### 2.2 기술적 근거

**퍼지 가중치 핑거프린팅 설계:**

| 항목 | 가중치 | 수집 방법 | 안정성 |
|------|--------|-----------|--------|
| CPU ID | 30% | WMI 또는 CPUID 명령어 | 높음 (교체 거의 없음) |
| 메인보드 시리얼 | 25% | WMI `Win32_BaseBoard` | 중간 (일부 보드 미보고) |
| BIOS UUID | 20% | WMI `Win32_ComputerSystemProduct` | 높음 |
| MAC 주소 (1차 NIC) | 15% | WMI `Win32_NetworkAdapter` | 중간 (NIC 교체 가능) |
| 디스크 시리얼 | 10% | WMI `Win32_DiskDrive` | 낮음 (교체 빈번) |

- **유사도 계산:** 코사인 유사도 또는 가중 해밍 거리를 사용한다.
- **임계치 전략:** 70% 이상이면 정상 통과, 50~70%이면 "재검증 필요" 플래그, 50% 미만이면 관리 콘솔에서 재바인딩 요구.
- **WMI 실패 처리:** 특정 항목의 WMI 쿼리가 실패하면 해당 항목의 가중치를 0으로 설정하고 나머지 항목의 가중치를 비례 재분배한다. "Not Applicable" 또는 "Unknown" 반환값도 동일하게 처리한다.

**WMI 실패율 대응 설계:**
WMI는 저장소 손상, 권한 문제, 하드웨어별 한계 등으로 실패할 수 있다. 이에 대한 방어적 설계:
- WMI 대체 경로: `SetupAPI`, 레지스트리 직접 쿼리, `CPUID` 어셈블리 명령어 등 WMI 외 수집 경로를 병행한다.
- 수집 결과 캐싱: 마지막 성공한 핑거프린트를 암호화하여 로컬에 캐싱하고, WMI 전체 실패 시 캐시와 비교한다.

### 2.3 예상 반론에 대한 재반박

**반론 1: "퍼지 핑거프린팅은 보안이 약하다. 하드웨어 값을 스푸핑할 수 있다."**

> 재반박: 맞다. 하지만 우리 제품은 DRM이 아니다. PII Scanner의 라이선스 바인딩 목적은 "우발적 라이선스 남용 방지"이지 "적극적 크래킹 방어"가 아니다. 기업 내부 배포 환경에서 직원이 WMI 값을 스푸핑해가며 라이선스를 우회할 동기와 기술 수준을 갖출 가능성은 극히 낮다. 보안 강도를 과도하게 높이면 오히려 정상 사용자의 마찰만 증가한다.

**반론 2: "TPM이 가용한 환경에서도 퍼지만 쓰면 보안 수준이 낮아진다."**

> 재반박: 퍼지 핑거프린팅을 "이상 탐지 신호"로 격하했으므로, 실질적 인증은 서버 발급 토큰(JWT 등)이 담당한다. 핑거프린트는 보조 신호일 뿐이므로, TPM 수준의 보안이 이 계층에 반드시 필요한 것은 아니다.

**반론 3: "결국 TPM 가용률이 올라가면 퍼지만으로는 부족해진다."**

> 재반박: 그 시점에 TPM 경로를 추가하는 것은 "퍼지 핑거프린트 인프라 위에 TPM을 선택적 강화 계층으로 얹는 것"이므로 입장 A와 호환된다. 지금 당장 TPM을 1차로 요구하는 것보다 점진적 도입이 리스크가 낮다.

## 3. 입장 B 옹호 (TPM 우선 + 퍼지 폴백)

### 3.1 핵심 논거

1. **암호학적 보안:** TPM 2.0의 Non-exportable Key는 하드웨어 칩 내부에서 생성되어 외부로 추출할 수 없다. 이는 소프트웨어 기반 핑거프린팅과 근본적으로 다른 보안 수준을 제공한다. 라이선스 키가 복제 불가능하므로, 라이선스 남용을 원천 차단한다.

2. **결정론적 바인딩:** 퍼지 핑거프린팅은 확률적이다(유사도 70% 이상이면 동일 PC로 판정). 이는 오탐과 미탐의 경계가 모호하다. TPM 기반 바인딩은 "키가 있거나 없거나"의 이진 판정이므로 애매한 상황이 발생하지 않는다.

3. **미래 지향성:** Windows 11이 TPM 2.0을 필수 요구하며, Microsoft는 이를 "협상 불가"라고 명시했다. 시간이 갈수록 TPM 가용률은 상승할 수밖에 없다. 2025년 10월 Windows 10 지원 종료 이후 기업의 Windows 11 전환이 가속화되고 있으며, 2026년 현재 전환 모멘텀이 증가하고 있다.

4. **mTLS 시너지:** TPM에 mTLS 클라이언트 인증서의 개인키를 저장하면, 라이선스 바인딩과 통신 채널 인증을 단일 메커니즘으로 통합할 수 있다. 별도의 핑거프린팅 레이어가 불필요해진다.

### 3.2 기술적 근거

**TPM 기반 라이선스 바인딩 흐름:**

```
[초기 등록]
1. 클라이언트 설치 시 TPM 2.0 존재 여부 확인
2. TPM 가용 시:
   a. TPM 내부에서 RSA-2048 또는 ECC-P256 키페어 생성 (Non-exportable)
   b. 공개키를 서버에 전송, 서버가 클라이언트 인증서 발급 (mTLS용)
   c. 인증서를 TPM-backed 저장소에 보관
3. TPM 불가 시:
   a. 퍼지 핑거프린트 생성 및 서버 등록 (폴백 경로)
   b. 관리 콘솔에 "TPM 미지원 장비" 태그 표시

[주기적 인증]
1. TPM 경로: mTLS 핸드셰이크로 자동 인증 (추가 로직 불필요)
2. 폴백 경로: 핑거프린트 유사도 비교 + 서버 토큰 검증
```

**VM/VDI 환경 지원:**
- VMware vSphere 7 이상: vTPM 지원 (VM 암호화 + Key Provider 필요)
- Citrix: vTPM 지원이 Citrix Hypervisor 8.2부터 추가됨
- Hyper-V: vTPM 기본 지원
- vTPM이 가용한 VDI 환경에서는 물리 TPM과 동일한 경로로 처리 가능

**mTLS 인증서 관리:**
- 서버가 자체 CA 역할을 하여 인증서를 발급하므로, 외부 PKI 의존 없음
- 인증서 수명: 1년, 자동 갱신 (만료 30일 전 클라이언트가 자동 갱신 요청)
- 인증서 폐기: 관리 콘솔에서 특정 PC의 인증서를 즉시 폐기 가능 (CRL 또는 OCSP)

### 3.3 예상 반론에 대한 재반박

**반론 1: "TPM 가용률이 75% 수준이면 25%가 폴백으로 빠진다. 결국 두 경로를 모두 구현해야 한다."**

> 재반박: 75%는 글로벌 평균이며, 한국 대기업은 하드웨어 교체 주기가 3~5년으로 비교적 짧다. 우리의 주요 고객(수만 대 PC를 운영하는 기업)은 대개 일괄 구매로 하드웨어를 도입하므로, 최근 도입분은 거의 100% TPM 2.0을 탑재하고 있다. 또한 Windows 11 전환이 진행 중이므로 TPM 비율은 시간이 갈수록 빠르게 상승한다. 폴백 경로는 구현하되, 이는 "레거시 지원"이지 "주요 경로"가 아니다.

**반론 2: "mTLS 인증서 관리가 복잡하고 운영 오버헤드가 크다."**

> 재반박: 자체 CA로 운영하면 외부 PKI 의존이 없으므로 복잡도가 크게 낮아진다. 인증서 발급·갱신·폐기는 모두 서버 내부 로직으로 자동화된다. 수만 대 규모의 인증서 관리는 확실히 고려사항이지만, 이미 서버-클라이언트 아키텍처에서 세션 관리를 해야 하는 상황이므로 추가 오버헤드는 인증서 저장소와 갱신 스케줄러 정도이다. OCSP 응답 캐싱을 적용하면 실시간 검증 부하도 관리 가능하다.

**반론 3: "VDI 환경에서 vTPM 설정이 추가 작업을 요구한다."**

> 재반박: VDI 환경은 이미 관리자가 인프라를 중앙 관리하는 구조이므로, vTPM 활성화는 템플릿 이미지에 한 번 설정하면 된다. VMware, Hyper-V, 최신 Citrix 모두 vTPM을 지원하며, 이는 VDI 관리자에게 익숙한 작업이다. 오히려 퍼지 핑거프린팅은 VDI에서 더 문제가 될 수 있다 — 비영속적(non-persistent) VDI에서는 매 세션마다 하드웨어 프로필이 달라질 수 있어 유사도 판정이 불안정하다.

## 4. 중립 심판 평가

### 4.1 입장 A의 강점과 약점

**강점:**
- **즉시 배포 가능:** TPM 가용 여부를 확인할 필요 없이 모든 환경에서 동일하게 동작한다. 영업 시점에 "이 PC에 TPM이 있습니까?"라고 물을 필요가 없다.
- **구현 복잡도 최소:** 코드 경로가 하나이므로 테스트 커버리지, 디버깅, 유지보수 부담이 적다.
- **하드웨어 변경 내성:** 가중치 기반 유사도 판정으로 부분적 하드웨어 변경에 자연스럽게 대응한다. RAM 증설이나 NIC 교체 같은 일상적 변경이 차단으로 이어지지 않는다.
- **용도 격하가 합리적:** "이상 탐지 신호"로의 격하는 핑거프린팅의 본질적 한계(스푸핑 가능성, 확률적 판정)를 인정하고, 서버 토큰에 실질적 인증을 위임하는 현실적 설계이다.

**약점:**
- **보안 강도 한계:** 결정적 증명(proof)이 아닌 통계적 추정이므로, 의도적 우회에 취약하다. 경쟁 제품 대비 "보안 어필"이 약해질 수 있다.
- **비영속 VDI 문제:** Non-persistent VDI에서 세션마다 할당되는 VM이 다를 수 있어, 핑거프린트 변동이 심할 수 있다. 이 환경에서는 별도 예외 처리가 필요하다.
- **WMI 의존성:** WMI 저장소 손상, 권한 문제, 일부 하드웨어의 시리얼 미보고 등 WMI 자체의 신뢰성 문제가 핑거프린트 품질에 직접 영향을 미친다.
- **미래 확장 비용:** 나중에 TPM 경로를 추가할 때, 기존 핑거프린트 기반 바인딩과의 마이그레이션 전략이 필요하다.

### 4.2 입장 B의 강점과 약점

**강점:**
- **암호학적 견고성:** TPM Non-exportable Key는 소프트웨어로 복제할 수 없다. 라이선스 바인딩의 가장 강력한 형태이다.
- **결정론적 판정:** "키가 있다/없다"의 이진 판정으로 애매한 임계치 조정이 불필요하다.
- **mTLS 통합:** 라이선스 바인딩과 통신 인증을 단일 메커니즘으로 해결할 수 있어 아키텍처가 깔끔해진다.
- **미래 지향적:** Windows 11 필수 요건으로 TPM 보급률은 계속 상승할 것이므로, 시간이 갈수록 유리해진다.
- **VDI 친화적:** vTPM이 지원되는 VDI에서는 비영속 환경에서도 안정적으로 동작한다.

**약점:**
- **이중 코드 경로:** TPM 경로와 퍼지 폴백 경로를 모두 구현·테스트·유지보수해야 한다. 엣지 케이스(TPM 감지 실패, TPM 펌웨어 이슈 등)도 처리해야 한다.
- **초기 배포 마찰:** TPM이 비활성화된 PC에서는 BIOS 설정 변경이 필요할 수 있다. IT 관리자가 수천 대의 BIOS를 일괄 변경하는 것은 상당한 작업이다.
- **인증서 관리 오버헤드:** 수만 대 규모의 클라이언트 인증서 발급·갱신·폐기를 관리하는 인프라가 필요하다. 자체 CA라 하더라도 인증서 저장소, 갱신 스케줄러, CRL/OCSP 인프라는 구현해야 한다.
- **TPM 접근 경합:** 기업 환경에서 BitLocker, Windows Hello 등 다른 시스템도 TPM을 사용한다. TPM 리소스 경합이나 잠금(lockout) 문제가 발생할 수 있다.
- **구형 VDI 미지원:** Citrix Hypervisor 8.2 이전, 구형 VMware 환경 등 vTPM을 지원하지 않는 VDI 플랫폼이 여전히 존재한다.

### 4.3 핵심 트레이드오프 분석

| 평가 항목 | 입장 A (퍼지 핑거프린팅) | 입장 B (TPM 우선 + 폴백) |
|-----------|--------------------------|--------------------------|
| **보안 강도** | 중 (통계적 추정, 스푸핑 가능) | 상 (암호학적 증명, 복제 불가) |
| **호환성** | 상 (모든 Windows PC) | 중 (TPM 가용 시 최적, 불가 시 폴백) |
| **구현 복잡도** | 하 (단일 경로) | 중~상 (이중 경로 + PKI) |
| **운영 오버헤드** | 하 (임계치 튜닝 정도) | 중 (인증서 라이프사이클 관리) |
| **하드웨어 변경 대응** | 상 (가중치 흡수) | 중 (TPM은 교체 시 재등록 필요) |
| **VDI 지원** | 중 (비영속 VDI 문제) | 중~상 (vTPM 지원 시 우수, 미지원 시 폴백) |
| **미래 확장성** | 중 (TPM 추가 시 마이그레이션 필요) | 상 (이미 TPM 중심 설계) |
| **사용자 마찰** | 하 (즉시 차단 없음) | 하~중 (TPM 가용 시 무마찰, 불가 시 폴백 전환) |
| **고객 영업 어필** | 중 (기술적 단순함) | 상 ("하드웨어 기반 보안" 마케팅) |

**결정적 변수:** 우리 고객사의 TPM 2.0 실질 가용률.

- **TPM 가용률 80% 이상:** 입장 B가 명백히 우월. 대다수가 최적 경로를 사용하고, 폴백은 소수 레거시만 담당.
- **TPM 가용률 50~80%:** 입장 B가 우세하나, 폴백 경로의 품질이 중요해짐. 사실상 두 경로 모두 "주요 경로"가 됨.
- **TPM 가용률 50% 미만:** 입장 A가 더 현실적. 폴백이 과반수의 주 경로가 되면 "TPM 우선"이라는 설계 전제가 무너짐.

### 4.4 조건부 결론

한국 기업 PC 환경의 특수성을 고려한 분석:

1. **대기업 (PC 5,000대 이상):** 일괄 하드웨어 구매 및 주기적 교체를 하므로 최근 도입분의 TPM 2.0 탑재율이 높다. Windows 11 전환도 IT 부서 주도로 진행되고 있다. 이 세그먼트에서는 TPM 가용률 80% 이상을 기대할 수 있으며, **입장 B가 적합**하다.

2. **중견기업 (PC 500~5,000대):** 하드웨어 교체 주기가 불규칙하고, Windows 11 전환이 부분적으로 진행된 상태이다. TPM 가용률 50~80%로 추정되며, **입장 B를 적용하되 폴백 경로의 완성도가 중요**하다.

3. **중소기업 (PC 500대 미만):** 구형 하드웨어가 혼재하고, IT 인력이 제한적이다. TPM 가용률이 50% 미만일 가능성이 있으며, 이 세그먼트만 놓고 보면 **입장 A가 더 현실적**이다. 단, 우리 솔루션의 1차 타겟이 수만 대 규모의 기업이라면 이 세그먼트의 비중은 상대적으로 낮다.

**핵심 인사이트:** 우리 제품의 주요 고객(수만 대 PC 운영 기업)은 대기업~중견기업이다. 이 세그먼트에서 TPM 가용률은 50% 이상으로 추정되며, Windows 11 전환 가속에 따라 빠르게 상승 중이다. 따라서 TPM 우선 + 퍼지 폴백 전략이 "현재도 동작하면서 미래에 더 강해지는" 방향이다.

## 5. 최종 추천안

**추천: 입장 B (TPM 우선 + 퍼지 폴백) 채택, 단 아래 조건 부여**

### 구현 전략

```
[Phase 1: v1.0 출시]
├── TPM 2.0 감지 → 가용 시 TPM 경로 (1차)
│   ├── TPM 내부에서 Non-exportable 키 생성
│   ├── 서버에 공개키 등록
│   └── 이후 인증은 서명 기반 챌린지-응답
├── TPM 불가 시 → 퍼지 핑거프린트 경로 (폴백)
│   ├── 입장 A의 가중치 핑거프린팅 설계 그대로 적용
│   ├── 용도: "이상 탐지 신호" (즉시 차단 없음)
│   └── 콘솔 재바인딩 UX 필수 포함
└── 관리 콘솔에 TPM/폴백 비율 대시보드 제공

[Phase 2: v1.x 업데이트]
├── 고객사별 TPM 가용률 데이터 수집 (자체 텔레메트리)
├── mTLS 통합 (TPM 경로에 한해 점진 적용)
└── 폴백 경로의 임계치 튜닝 (실제 운영 데이터 기반)
```

### 핵심 설계 원칙

1. **TPM 경로는 "강한 바인딩"으로, 폴백 경로는 "이상 탐지 신호"로 운용한다.** 두 경로의 보안 수준이 다르므로 동일한 정책을 적용하지 않는다.

2. **폴백 경로에서는 즉시 차단하지 않는다.** 입장 A의 "재검증 필요" 플래그 설계를 그대로 수용한다. 이는 WMI 불안정성과 하드웨어 변경에 대한 현실적 대응이다.

3. **관리 콘솔 재바인딩 UX는 양쪽 경로 모두 필수이다.** TPM 경로도 메인보드 교체 시 재바인딩이 필요하다.

4. **mTLS는 v1.0에서 선택적으로 적용한다.** TPM + mTLS 통합은 매력적이나, 인증서 관리 인프라의 복잡도를 감안하여 초기에는 TPM 챌린지-응답 방식으로 시작하고, 이후 mTLS로 전환한다.

5. **텔레메트리 수집을 통해 데이터 기반 결정을 내린다.** v1.0 배포 후 실제 고객사의 TPM 가용률, 폴백 비율, 하드웨어 변경 빈도 데이터를 수집하여 Phase 2 전략을 조정한다.

### 리스크 완화 방안

| 리스크 | 완화 방안 |
|--------|-----------|
| TPM 가용률이 예상보다 낮음 | 폴백 경로가 입장 A와 동일 품질이므로 서비스 영향 없음 |
| mTLS 인증서 관리 부담 | Phase 1에서 mTLS를 배제하고 챌린지-응답으로 대체 |
| TPM 접근 경합 (BitLocker 등) | TPM 키 생성은 설치 시 1회만 수행, 이후는 서명 연산만 사용 |
| VDI vTPM 미지원 | VDI 환경은 관리 콘솔에서 별도 정책으로 관리 (폴백 자동 적용) |
| 이중 코드 경로 테스트 부담 | TPM 경로와 폴백 경로의 인터페이스를 통일하여 상위 레이어는 단일 코드 경로 유지 |

### 이 추천이 양측 입장을 어떻게 수용하는가

- **입장 A에서 수용한 것:** 퍼지 핑거프린팅의 전체 설계(가중치 체계, WMI 실패 처리, 임계치 전략), "이상 탐지 신호"로의 용도 격하, 즉시 차단 금지 정책, 콘솔 재바인딩 UX. 이들은 폴백 경로에 그대로 반영된다.
- **입장 B에서 수용한 것:** TPM을 1차 메커니즘으로 채택, 미래 지향적 설계 방향, mTLS 통합 로드맵.
- **절충 지점:** mTLS를 v1.0 필수가 아닌 Phase 2 선택 사항으로 격하하여 초기 구현 복잡도를 낮춤. 폴백 경로의 정책을 입장 A 수준으로 유지하여 사용자 마찰을 최소화.
