# Deep Dive 7: 규칙 정의 형식의 도입 시점

**토론 일자:** 2026-02-11
**참여자:** 입장A 옹호자, 입장B 옹호자, 중립 심판

---

## 1. 배경 및 쟁점 요약

PII Scanner의 탐지 규칙(정규식 패턴, 체크섬 알고리즘, 키워드 리스트 등)을 어떤 형식으로 정의하고 관리할 것인지가 쟁점이다. 이미 "서명된 룰 번들을 서버에서 클라이언트로 푸시하여 업데이트한다"는 원칙에는 전원 동의했으나, **번들의 내부 형식**에서 의견이 갈린다.

**입장 A (Rust 코드 직접 정의, 스키마는 나중에):** MVP에서는 탐지 규칙을 Rust 코드의 `const`/`static` 또는 열거형으로 직접 정의한다. 규칙이 20개 이상 누적되어 관리 복잡도가 체감되는 시점에 DSL이나 외부 스키마 도입을 검토한다. YAML/JSON DSL은 파서, 검증기, 문서화가 필요한 미니 언어가 되므로 MVP에서는 과잉 엔지니어링이다.

**입장 B (MVP부터 JSON 스키마):** MVP에서도 규칙 데이터를 JSON 스키마(열거형만 허용, 표현식/로직 금지)로 분리한다. 규칙이 코드에 하드코딩되면 바이너리 재배포 없이 규칙 업데이트가 불가능하며, 50,000대 PC에 10MB 바이너리를 재배포하는 것보다 1KB JSON 번들을 푸시하는 것이 운영적으로 압도적 우위이다.

**합의된 사항:** 서명된 룰 번들 업데이트 메커니즘 자체에는 전원 동의.

**잠정 추천:** Codex안(열거형 전용 JSON 스키마)을 MVP부터 채택.

**쟁점의 본질:** YAGNI(You Aren't Gonna Need It) 원칙 vs 이미 알려진 미래 요구사항에 대한 선제 대응. 컴파일 타임 타입 안전성 vs 런타임 유연성의 트레이드오프.

## 2. 입장 A 옹호 (Rust 코드 직접 정의, 스키마는 나중에)

### 2.1 핵심 논거

MVP의 목표는 **탐지 정확도의 검증**이지, 규칙 배포 파이프라인의 완성이 아니다. 규칙을 외부 JSON으로 분리하면 "스키마 설계 -> 파서 구현 -> 검증기 구현 -> 에러 처리 -> 서명/검증 -> 번들 빌드 도구"라는 부수적 인프라 작업이 발생하며, 이는 MVP의 핵심 가치(탐지 품질)와 무관한 작업에 시간을 소모하는 것이다. Rust 코드로 규칙을 정의하면 컴파일러가 타입 안전성을 보장하므로, 런타임 검증 로직이 불필요하다.

### 2.2 기술적 근거

**1) 컴파일 타임 안전성의 가치**

Rust 코드로 규칙을 정의하면 다음이 컴파일 시점에 보장된다:

```rust
enum ChecksumAlgorithm {
    LuhnMod11,
    Mod10,
    WeightedSum,
    None,
}

struct PiiRule {
    id: &'static str,
    name: &'static str,
    pattern: &'static str,      // 정규식 문자열
    checksum: ChecksumAlgorithm,
    keywords: &'static [&'static str],
    min_confidence: f32,
}

static RULES: &[PiiRule] = &[
    PiiRule {
        id: "kr_rrn",
        name: "주민등록번호",
        pattern: r"\d{6}-[1-4]\d{6}",
        checksum: ChecksumAlgorithm::LuhnMod11,
        keywords: &["주민", "번호", "생년월일"],
        min_confidence: 0.9,
    },
    // ...
];
```

- 존재하지 않는 체크섬 알고리즘 참조 -> 컴파일 에러
- 필수 필드 누락 -> 컴파일 에러
- 타입 불일치(예: `min_confidence`에 문자열 할당) -> 컴파일 에러

JSON 스키마로 전환하면 이 모든 검증을 **런타임 역직렬화 시점**에 수행해야 하며, serde의 역직렬화 에러 메시지가 아무리 훌륭해도 컴파일 에러보다 늦게 발견된다.

**2) Inner Platform Effect 회피**

외부 스키마를 도입하면 불가피하게 다음이 발생한다:

- JSON 스키마 자체의 문서화 필요 (어떤 필드가 필수인지, 유효한 값은 무엇인지)
- 스키마 버전 관리 필요 (v1 번들과 v2 번들의 호환성)
- 번들 빌드 도구 필요 (JSON 작성 -> 검증 -> 서명 -> 패키징)
- 스키마 변경 시 하위 호환성 정책 필요

이것은 "프로그래밍 언어 위에 또 다른 프로그래밍 언어를 만드는" Inner Platform Effect의 초기 증상이다. 열거형만 허용한다고 해도, 고객 요구에 따라 "조건부 적용", "규칙 간 우선순위", "규칙 그룹핑" 같은 기능이 추가되면 결국 미니 DSL로 진화한다. Rust 코드라면 이런 로직을 자연스럽게 표현할 수 있지만, JSON 스키마에서는 표현력의 한계에 부딪혀 스키마를 확장해야 한다.

**3) MVP 단계에서 규칙 변경 빈도는 낮다**

MVP는 내부 파일럿(수백~수천 대) 수준이다. 이 단계에서 규칙 변경이 발생하면 바이너리를 재배포하는 것이 현실적으로 문제되지 않는다. 50,000대 규모의 프로덕션 배포에서 규칙 핫 업데이트가 필요해지는 시점은 Phase 2 이후이며, 그때 JSON 스키마를 도입해도 충분하다.

**4) YAGNI 원칙의 정당성**

"규칙이 코드에 있으면 바이너리 재배포가 필요하다"는 사실이지만, MVP에서 이것이 실제로 문제가 되는지는 검증되지 않았다. 문제가 될 것이라는 **가정** 하에 인프라를 선제 구축하는 것은 YAGNI 위반이다. 실제 운영에서 "규칙만 바꾸고 싶은데 바이너리를 재배포해야 한다"는 고통이 체감되는 시점에 도입하면, 그때의 설계가 실제 요구사항에 더 잘 맞는다.

### 2.3 예상 반론에 대한 재반박

**반론: "한국 PII 유형이 이미 10개 이상이므로 규칙이 20개 이하로 유지될 수 없다."**

> 규칙의 "수"와 "관리 복잡도"는 비례하지 않는다. 10-15개의 정적 규칙은 하나의 Rust 파일에 500줄 이내로 깔끔하게 정의할 수 있으며, IDE의 검색/치환으로 충분히 관리 가능하다. 문제는 규칙의 수가 아니라 **규칙의 변경 빈도와 변경 주체**이다. MVP에서 규칙을 변경하는 주체는 개발팀 자신이므로, 코드 수정 -> 빌드 -> 배포가 자연스러운 워크플로우이다.

**반론: "serde_json이 Rust에서 trivial하므로 JSON 도입 비용이 거의 없다."**

> `serde_json`으로 JSON을 파싱하는 것 자체는 trivial하다. 그러나 JSON 스키마 도입의 실제 비용은 파싱이 아니라 **주변 인프라**에 있다: 번들 빌드 도구, 서명/검증 파이프라인, 스키마 버전 관리, 번들 로딩 실패 시 폴백 로직, 규칙 편집 UI 또는 CLI 도구. 이 모든 것이 "serde_json이 쉽다"는 말로 가려지는 숨겨진 비용이다.

**반론: "번들 형식을 나중에 도입하면 기존 코드 규칙에서 외부 스키마로의 마이그레이션 비용이 발생한다."**

> Rust 코드의 `PiiRule` 구조체가 이미 JSON 스키마와 1:1 대응하는 구조이므로, `#[derive(Serialize, Deserialize)]`를 추가하고 로딩 소스를 코드 상수에서 파일로 전환하는 것이 마이그레이션의 전부이다. 이 작업은 반나절이면 충분하며, "나중에 마이그레이션이 어려울 것"이라는 우려는 과장이다.

## 3. 입장 B 옹호 (MVP부터 JSON 스키마)

### 3.1 핵심 논거

규칙을 코드에 하드코딩하는 것은 "알려진 미래 요구사항을 의도적으로 무시하는 것"이다. 50,000대 PC에 바이너리를 재배포하지 않고 규칙을 업데이트해야 한다는 것은 가정이 아니라 **확정된 요구사항**이다(기존 15년 운영 경험에서 검증됨). 확정된 요구사항을 YAGNI로 미루는 것은 YAGNI의 오용이다. 더구나 "열거형 전용, 표현식 금지" 수준의 최소 JSON 스키마는 구현 비용이 극히 낮아, 과잉 엔지니어링이라는 비판이 성립하지 않는다.

### 3.2 기술적 근거

**1) 한국 PII 유형의 규모가 이미 외부 관리를 정당화한다**

한국에서 법적/실무적으로 관리해야 하는 PII 유형을 열거하면:

| # | PII 유형 | 비고 |
|:---|:---|:---|
| 1 | 주민등록번호 | 체크섬 검증 필수 |
| 2 | 외국인등록번호 | 주민번호와 유사하나 별도 패턴 |
| 3 | 사업자등록번호 | 10자리, 가중합 체크섬 |
| 4 | 법인등록번호 | 13자리, 체크섬 |
| 5 | 신용카드번호 | Luhn 체크섬, 카드사별 BIN 범위 |
| 6 | 계좌번호 | 은행별 자릿수 상이 |
| 7 | 전화번호 | 유선/무선/VoIP 구분 |
| 8 | 이메일 주소 | RFC 5322 호환 |
| 9 | 여권번호 | 구/신 여권 형식 차이 |
| 10 | 운전면허번호 | 지역코드 + 일련번호 |
| 11 | 건강보험번호 | 형식 다양 |
| 12 | 자동차등록번호 | - |
| 13 | IP 주소 | IPv4/IPv6 |
| 14 | 생년월일 | 다양한 포맷 |

여기에 **고객 맞춤 규칙**이 추가된다. 기존 15년 운영 경험에서 고객사마다 "사내 사번", "고객 ID 형식", "내부 프로젝트 코드" 등의 맞춤 패턴을 요구한 사례가 다수 있었을 것이다. 이런 고객별 규칙을 Rust 코드에 하드코딩하면, 고객 A용 바이너리, 고객 B용 바이너리를 별도 관리해야 하는 악몽이 시작된다.

JSON 번들이면 "기본 규칙 + 고객별 추가 규칙"을 서버에서 조합하여 해당 고객의 에이전트에만 배포할 수 있다.

**2) serde_json + 열거형 기반 스키마의 구현 비용은 극히 낮다**

Codex의 PoC가 이미 증명했듯이, 최소 JSON 스키마의 핵심 구현은 다음과 같다:

```rust
use serde::{Deserialize, Serialize};

#[derive(Deserialize, Serialize)]
#[serde(rename_all = "snake_case")]
enum ValidatorKind {
    Regex,
    RegexWithChecksum,
    KeywordProximity,
}

#[derive(Deserialize, Serialize)]
#[serde(rename_all = "snake_case")]
enum ChecksumType {
    LuhnMod11,
    Mod10,
    WeightedSum,
}

#[derive(Deserialize, Serialize)]
struct RuleDefinition {
    rule_id: String,
    name: String,
    kind: ValidatorKind,
    pattern: Option<String>,
    checksum: Option<ChecksumType>,
    keywords: Vec<String>,
    confidence_threshold: f32,
}

#[derive(Deserialize, Serialize)]
struct RuleBundle {
    version: u32,
    rules: Vec<RuleDefinition>,
}
```

`serde`의 `#[derive(Deserialize)]`가 스키마 검증의 대부분을 처리한다. 알 수 없는 `ValidatorKind` 값이 들어오면 역직렬화가 실패하고, 필수 필드가 누락되면 에러가 반환된다. "별도의 검증기를 만들어야 한다"는 입장 A의 우려는 `serde`의 능력을 과소평가한 것이다.

추가로 필요한 것은 정규식 컴파일 검증뿐이다:

```rust
fn validate_bundle(bundle: &RuleBundle) -> Result<(), BundleError> {
    for rule in &bundle.rules {
        if let Some(ref pattern) = rule.pattern {
            regex::Regex::new(pattern)
                .map_err(|e| BundleError::InvalidRegex {
                    rule_id: rule.rule_id.clone(),
                    error: e.to_string(),
                })?;
        }
    }
    Ok(())
}
```

이 정도 코드량은 "과잉 엔지니어링"이라 부르기 어렵다.

**3) 바이너리 재배포 비용은 MVP에서도 무시할 수 없다**

입장 A는 "MVP는 수백~수천 대 규모이므로 바이너리 재배포가 문제되지 않는다"고 주장하지만, 이는 운영 현실을 간과한 것이다:

- 기업 환경에서 바이너리 업데이트는 보안팀 승인, 변경 관리 프로세스(CAB), 단계적 롤아웃을 거친다
- 바이너리 교체는 에이전트 프로세스 재시작을 요구하며, 이는 진행 중인 스캔을 중단시킨다
- 규칙 하나 수정 때문에 전체 바이너리(10-20MB + ONNX 모델)를 재배포하는 것은 대역폭 낭비이다
- 파일럿 운영 중 "전화번호 패턴에서 070 VoIP를 빠뜨렸다"는 피드백이 오면, 규칙 번들(~1KB)만 푸시하는 것과 바이너리(~10MB)를 재배포하는 것의 차이는 크다

Codex의 PoC가 정확히 이 시나리오를 검증했다: v1 번들에서 v2 번들로의 전환으로 070 VoIP 번호 탐지를 코드 변경 없이 추가.

**4) 서명된 번들과 자연스럽게 결합된다**

이미 합의된 "서명된 룰 번들 업데이트"를 구현하려면, 번들이 **코드 외부에 독립적인 아티팩트로 존재**해야 한다. 규칙이 Rust 코드에 하드코딩되어 있으면 "서명된 번들"이라는 개념 자체가 성립하지 않는다. 결국 입장 A도 프로덕션에서는 외부 스키마로 전환해야 하며, MVP에서 코드 하드코딩으로 시작하는 것은 "같은 작업을 두 번 하는 것"이다.

**5) 업계 표준이 선언적 규칙 형식이다**

모든 주요 보안 제품이 선언적 규칙 형식을 사용한다:

- **YARA**: 악성코드 탐지 규칙을 `.yar` 파일로 정의
- **Snort/Suricata**: IDS/IPS 규칙을 텍스트 형식으로 정의
- **CrowdStrike Falcon**: IoC(Indicator of Compromise)를 JSON 형식으로 관리
- **Microsoft Defender**: 탐지 시그니처를 바이너리 업데이트와 분리하여 배포

규칙을 코드에 하드코딩하는 상용 보안 제품은 사실상 존재하지 않는다. 이는 이유가 있다: 위협 환경(또는 PII 규제 환경)은 코드 릴리스 주기보다 빠르게 변화하며, 규칙 업데이트의 민첩성이 제품 경쟁력을 좌우하기 때문이다.

### 3.3 예상 반론에 대한 재반박

**반론: "Inner Platform Effect가 우려된다. 스키마가 결국 미니 DSL로 진화할 것이다."**

> 이것이 바로 "열거형 전용, 표현식 금지"라는 가드레일의 존재 이유이다. 스키마에 허용되는 것은 오직 serde로 역직렬화 가능한 열거형 값(ValidatorKind, ChecksumType 등)과 문자열 데이터(정규식 패턴, 키워드)뿐이다. 조건문, 반복문, 변수 참조, 규칙 간 의존성 등은 스키마 수준에서 원천적으로 금지한다. 새로운 ValidatorKind나 ChecksumType이 필요하면 Rust 코드에 열거형 variant를 추가하고 바이너리를 업데이트해야 한다. 즉, **로직의 확장은 코드에서, 파라미터의 변경은 JSON에서** 이루어지는 명확한 경계가 있다. 이 경계를 유지하는 한 Inner Platform Effect는 발생하지 않는다.

**반론: "컴파일 타임 안전성을 포기하는 것이다."**

> 포기하는 것이 아니라 **안전성의 시점을 이동**하는 것이다. 컴파일 타임에서 "번들 로딩 시점"으로 옮기되, fail-closed 전략(번들 검증 실패 시 이전 번들 유지 또는 내장 기본 규칙으로 폴백)으로 런타임 안전성을 보장한다. 또한 번들 빌드 파이프라인에서 CI 단계의 스키마 검증을 수행하면, 잘못된 번들이 프로덕션에 도달하기 전에 차단된다. 이는 Docker 이미지의 빌드 타임 검증과 동일한 패턴이다.

**반론: "번들 빌드 도구, 서명 파이프라인 등 숨겨진 비용이 크다."**

> 서명/검증은 이미 합의된 요구사항이므로 어차피 구현해야 한다. 번들 빌드 도구는 MVP에서는 간단한 Python 스크립트(Codex PoC 수준)로 충분하다. 정교한 번들 관리 UI는 Phase 2의 관리 콘솔에서 구현하면 된다. MVP에서의 추가 비용은 `serde` 역직렬화 코드 ~50줄 + 번들 로딩 로직 ~30줄 + 정규식 검증 ~20줄 = **약 100줄**이다. 이것을 "과잉 엔지니어링"이라 부르는 것은 비용을 과대평가한 것이다.

## 4. 중립 심판 평가

### 4.1 입장 A의 강점과 약점

**강점:**

- **컴파일 타임 안전성 논거가 원칙적으로 옳다.** 런타임 에러보다 컴파일 에러가 항상 낫다는 것은 Rust 생태계의 핵심 철학이며, 이 원칙을 지키려는 의지는 존중할 만하다.
- **Inner Platform Effect에 대한 경고가 유효하다.** 스키마가 "열거형 전용"으로 시작해도 고객 요구에 따라 복잡해지는 경향은 실제로 빈번하다. "규칙 A는 주민번호 탐지 후에만 적용", "이 규칙 그룹은 금융 업종에만 활성화" 같은 요구가 오면 스키마 확장 압력이 생긴다.
- **YAGNI 원칙의 적용이 일반론으로는 건전하다.** 불확실한 미래를 위해 현재 복잡도를 추가하는 것을 경계하는 자세는 좋은 엔지니어링 습관이다.

**약점:**

- **YAGNI를 "이미 알려진 요구사항"에 적용하는 것은 오류이다.** 50,000대 PC에 바이너리 재배포 없이 규칙을 업데이트해야 한다는 것은 가정이 아니라 15년 운영 경험에서 도출된 확정 요구사항이다. YAGNI는 "필요할지 모르는 것을 미리 만들지 말라"는 원칙이지, "필요한 것을 미리 만들지 말라"는 원칙이 아니다.
- **마이그레이션 비용을 과소평가한다.** "`#[derive(Deserialize)]` 추가 + 로딩 소스 전환이 반나절"이라고 하지만, 실제로는 번들 로딩 실패 시 폴백 로직, 기존 코드 규칙과 외부 번들의 공존 기간 관리, 테스트 코드 전환 등이 수반되며, 이는 반나절로 끝나지 않는다.
- **"규칙 20개 이상에서 검토"라는 기준이 자의적이다.** 한국 PII 유형만 14개 이상, 고객별 맞춤 규칙을 합산하면 MVP 단계에서도 20개를 초과할 가능성이 높다. 임계치가 이미 도달해 있거나 매우 근접해 있다.

### 4.2 입장 B의 강점과 약점

**강점:**

- **운영 현실에 기반한 논거가 강력하다.** 15년간 50,000대 PC를 운영한 경험에서 "규칙 업데이트를 위한 바이너리 재배포"가 얼마나 고통스러운지는 이미 검증되었다. 이 문제를 MVP에서부터 해결하겠다는 접근은 실용적이다.
- **구현 비용이 실제로 낮다는 것을 PoC로 입증했다.** Codex의 PoC(Ed25519 서명 + JSON 번들 + 규칙 핫 업데이트)가 작동하는 것을 확인했으며, Rust에서 serde 기반 구현은 이보다 더 간결하다.
- **고객 맞춤 규칙에 대한 논거가 설득력 있다.** 고객별 바이너리를 별도 관리하는 것은 운영 악몽이며, JSON 번들이면 서버에서 고객별 규칙 세트를 조합하여 배포할 수 있다.
- **업계 표준과의 정합성이 높다.** 모든 주요 보안 제품이 규칙을 선언적 형식으로 분리 관리한다는 것은 강력한 선례이다.

**약점:**

- **Inner Platform Effect 방지를 위한 가드레일의 장기적 지속 가능성이 불확실하다.** "열거형 전용, 표현식 금지"는 현재 팀이 합의한 규칙이지, 기술적으로 강제되는 것은 아니다. 고객 압박 하에서 "이 한 가지만 예외로" 스키마를 확장하다 보면 가드레일이 무너질 수 있다.
- **번들 로딩 실패 시 폴백 전략에 대한 구체적 설계가 부족하다.** 서명 검증 실패, 정규식 컴파일 실패, 스키마 버전 불일치 등의 시나리오에서 에이전트가 어떻게 동작해야 하는지(이전 번들 유지? 내장 기본 규칙으로 폴백? 탐지 중지?) 명확히 정의되어야 한다. 이 폴백 로직 자체가 상당한 설계 복잡도를 갖는다.
- **"100줄이면 된다"는 주장이 핵심 로직만 카운트한 것이다.** 에러 처리, 로깅, 폴백 로직, 번들 캐싱, 번들 버전 비교, 테스트 코드까지 포함하면 실제 코드량은 300-500줄 수준이 된다. 여전히 관리 가능한 수준이지만, "100줄"이라는 수치는 오해의 소지가 있다.

### 4.3 핵심 트레이드오프 분석

| 평가 기준 | Rust 코드 직접 정의 (입장 A) | JSON 스키마 (입장 B) | 비고 |
|:---|:---|:---|:---|
| **컴파일 타임 안전성** | 우월 | 열등 (런타임 검증) | serde가 상당 부분 커버하나 완전하지 않음 |
| **규칙 업데이트 민첩성** | 열등 (바이너리 재배포) | 우월 (번들 푸시) | 50K PC 환경에서 결정적 차이 |
| **고객 맞춤 규칙 지원** | 열등 (고객별 바이너리) | 우월 (고객별 번들) | 상용 제품의 핵심 요구사항 |
| **MVP 구현 비용** | 약간 우월 | 약간 열등 (300-500줄 추가) | 차이가 크지 않음 |
| **Inner Platform Effect 위험** | 없음 | 존재 (가드레일로 통제) | 장기적 규율 필요 |
| **업계 표준 정합성** | 열등 | 우월 | 모든 주요 보안 제품이 선언적 형식 사용 |
| **프로덕션 전환 비용** | 열등 (마이그레이션 필요) | 우월 (이미 준비됨) | A는 어차피 나중에 전환해야 함 |
| **서명된 번들과의 통합** | 불가 (코드 = 번들 아님) | 자연스러움 | 이미 합의된 요구사항과 직결 |
| **디버깅 용이성** | 우월 (컴파일러 지원) | 보통 (런타임 에러 메시지) | - |

### 4.4 조건부 결론

이 논쟁에서 결정적인 요인은 다음 세 가지이다:

**첫째, "서명된 룰 번들 업데이트"가 이미 합의되었다.** 이 합의를 실현하려면 규칙이 코드 외부에 독립적으로 존재해야 한다. 입장 A는 이 합의와 논리적으로 모순된다. MVP에서 코드 하드코딩으로 시작하면, 합의된 번들 업데이트 기능을 MVP에서는 구현하지 않는다는 뜻이 되며, 이는 서명 검증 로직의 MVP 내 테스트도 불가능하다는 의미이다.

**둘째, 한국 PII 유형이 이미 10개를 초과한다.** 입장 A의 "20개 이상에서 검토" 기준은 MVP 규칙 수의 현실적 추정(기본 14개 + 고객 맞춤 규칙)과 비교하면 이미 도달해 있거나 초과할 가능성이 높다.

**셋째, 구현 비용 차이가 미미하다.** 입장 B의 추가 비용(300-500줄)은 MVP 전체 코드량 대비 무시할 수 있는 수준이며, serde 기반 구현은 Rust 생태계에서 가장 성숙하고 검증된 패턴이다.

따라서 **입장 B(JSON 스키마)가 우세하되**, 다음 가드레일을 필수 전제로 한다:

1. 스키마는 열거형 전용으로 제한하고, 조건문/표현식을 금지하는 원칙을 설계 문서에 명시
2. 번들 검증 실패 시 fail-closed(이전 유효 번들 유지) 전략 적용
3. 번들 서명 검증 필수, 미서명 번들 로딩 금지
4. 새로운 ValidatorKind/ChecksumType 추가는 반드시 Rust 코드 변경을 수반 (스키마 확장은 코드 릴리스 주기를 따름)

## 5. 최종 추천안

### 추천: MVP부터 열거형 전용 JSON 스키마 도입 (입장 B 채택)

**이유:**

1. **합의된 요구사항과의 일관성.** "서명된 룰 번들 업데이트"는 이미 전원 합의된 사항이다. 이 기능의 기반이 되는 외부 규칙 정의를 MVP에서 구현하지 않으면, 번들 업데이트 파이프라인 전체를 Phase 2로 미루는 결과가 된다. MVP에서 번들 서명/검증까지 검증해야 프로덕션 전환이 매끄럽다.

2. **구현 비용이 낮다.** Rust의 serde 생태계가 JSON 스키마 도입의 대부분의 작업을 처리한다. `#[derive(Deserialize)]` + 열거형 정의 + 정규식 검증 + 번들 로딩 로직으로 300-500줄이면 핵심 구현이 완료된다. 이 비용은 MVP 일정에 유의미한 영향을 주지 않는다.

3. **운영적 이점이 명확하다.** 50,000대 PC에 1KB 규칙 번들을 푸시하는 것과 10MB 바이너리를 재배포하는 것의 차이는 파일럿 단계에서도 체감할 수 있으며, 프로덕션에서는 결정적이다. 규칙 오류 발견 시 수 분 내 수정 배포가 가능해진다.

4. **고객 맞춤 규칙의 기반이 된다.** 상용 제품으로서 고객별 맞춤 규칙은 필수 기능이다. JSON 번들 구조가 있으면 "기본 규칙 세트 + 고객 추가 규칙"을 서버에서 조합하여 배포하는 구조를 자연스럽게 확장할 수 있다.

### 필수 가드레일

Inner Platform Effect를 방지하기 위해 다음 원칙을 설계 문서에 명시하고 코드 리뷰에서 강제한다:

| 원칙 | 설명 |
|:---|:---|
| **열거형 전용** | JSON 스키마에 허용되는 타입은 serde로 역직렬화 가능한 Rust 열거형에 1:1 대응하는 값만 |
| **표현식 금지** | 조건문, 반복문, 변수 참조, 규칙 간 의존성, 스크립트 임베딩 일체 금지 |
| **로직은 코드에** | 새로운 검증 로직(새 체크섬 알고리즘, 새 탐지 방식)은 Rust 코드에 구현하고, JSON에는 파라미터만 노출 |
| **fail-closed** | 번들 로딩/검증 실패 시 에이전트는 이전 유효 번들(또는 내장 기본 규칙)으로 폴백. 빈 규칙 세트로 동작하지 않음 |
| **서명 필수** | 미서명 번들, 서명 검증 실패 번들은 로딩 거부 |
| **스키마 버전** | 번들에 스키마 버전 필드를 포함하여, 에이전트가 지원하지 않는 스키마 버전의 번들을 거부할 수 있도록 함 |

### 구현 로드맵

| 단계 | 작업 | 예상 공수 |
|:---|:---|:---|
| MVP Phase 1 | serde 기반 RuleBundle 구조체 정의, JSON 로딩/검증 | 1-2일 |
| MVP Phase 1 | Ed25519 서명 검증 (번들 로딩 시) | 1일 |
| MVP Phase 1 | 기본 한국 PII 규칙 번들 작성 (14개 유형) | 1일 |
| MVP Phase 1 | 번들 빌드 스크립트 (Python/Rust CLI) | 0.5일 |
| Phase 2 | 관리 콘솔에서 규칙 편집/번들 생성 UI | 별도 산정 |
| Phase 2 | 고객별 규칙 세트 관리 기능 | 별도 산정 |

총 MVP 추가 공수: **약 3.5-4.5일**. 이 비용으로 바이너리 재배포 없는 규칙 업데이트, 서명 검증, 고객 맞춤 규칙의 기반을 확보할 수 있다.
