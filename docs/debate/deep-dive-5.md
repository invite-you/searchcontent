# Deep Dive 5: C#/.NET 하이브리드의 완전 배제 여부

**토론 일자:** 2026-02-11
**참여자:** 입장A 옹호자, 입장B 옹호자, 중립 심판

## 1. 배경 및 쟁점 요약

PII Scanner 클라이언트의 기본 기술 스택은 Rust 단일 언어로 합의되었다(합의 수준 3/3). 그러나 Codex는 "팀이 .NET에 강하고 Rust에 약한 경우"라는 조건부 예외안을 유보했다. 이 예외안의 핵심은 FFI(Foreign Function Interface)가 아닌 **프로세스 경계(Named Pipe IPC)**로 C# 서비스와 Rust 워커를 연결하는 구조이다.

**쟁점의 본질:** 기술적 순수성(단일 언어) vs 팀 현실(MVP 속도)의 트레이드오프.

**고려해야 할 핵심 제약 조건:**
- 50,000대 이상 PC에 배포해야 하므로 배포 단순성이 중요
- GPU 없이 CPU만으로 동작해야 함
- 4GB 저사양 PC도 지원해야 함
- "Quiet Agent"가 제품의 핵심 차별화 축
- 상용 제품 수준의 보안과 안정성 필수

## 2. 입장 A 옹호 (전체 Rust, 하이브리드 완전 배제)

### 2.1 핵심 논거

C#/.NET 하이브리드는 어떤 형태로든 배제해야 한다. IPC 분리가 FFI보다 깔끔한 것은 사실이나, **두 언어 생태계를 장기적으로 유지하는 비용은 기술적 문제가 아니라 조직적 문제**이며, 이는 시간이 지날수록 악화된다. Rust 숙련도가 부족하면 "Rust 전문가 충원" 또는 "전체 C#"이 더 정직한 선택이다.

### 2.2 기술적 근거

**1) 배포 복잡도 증가 (50K PC 환경에서 치명적)**

Rust 단일 바이너리는 하나의 `.exe` + ONNX 모델 파일로 배포가 완결된다. 반면 C# 서비스 + Rust 워커 구조는 다음을 요구한다:

- .NET Runtime 배포 (~80MB) 또는 Self-contained 패키징 (~65MB 추가)
- C# 서비스 바이너리 + Rust 워커 바이너리 = 2개의 독립 실행 파일
- 두 프로세스 간 Named Pipe 통신 프로토콜 버전 관리
- 두 프로세스의 독립적 업데이트/롤백 시나리오 관리
- MSI 설치 패키지 내 두 컴포넌트의 의존성 체인 관리

50,000대 PC에 배포할 때 **배포 실패의 표면적이 2배로 증가**한다. .NET Runtime 설치 실패, 버전 불일치, 두 프로세스 중 하나만 업데이트되는 반쪽 업데이트 등 운영 사고의 가짓수가 기하급수적으로 늘어난다.

**2) 메모리 풋프린트 증가 (4GB PC에서 치명적)**

.NET Runtime의 기본 메모리 오버헤드는 약 30-50MB이다. Rust 단일 프로세스가 10-20MB로 동작할 수 있는 것과 비교하면, .NET Runtime만으로도 메모리 예산의 상당 부분을 소모한다. 4GB PC에서 Windows OS + 사용자 업무 애플리케이션 + AV/EDR + PII Scanner를 동시에 구동해야 하는 환경에서 30-50MB는 무시할 수 없는 비용이다.

이는 "Quiet Agent" 차별화 전략에 정면으로 반한다. 사용자가 작업 관리자에서 .NET Runtime 프로세스가 메모리를 점유하고 있는 것을 보는 순간, "가볍다"는 인상은 사라진다.

**3) 이중 빌드/테스트 파이프라인의 유지 비용**

- CI/CD에서 `dotnet build` + `cargo build` 두 파이프라인을 유지
- 통합 테스트 시 두 프로세스를 모두 기동해야 함
- Named Pipe 프로토콜 변경 시 양쪽 코드를 동기화해야 함
- Windows 서비스 설치/제거 로직이 C# 측과 Rust 측에 분산
- 코드 리뷰어가 두 언어를 모두 이해해야 의미 있는 리뷰 가능

**4) Windows 서비스 개발은 Rust에서도 충분히 성숙**

`windows-service` crate는 Windows Service Control Manager와의 통합을 제공하며, 프로덕션 환경에서 사용되고 있다. SCM 등록, 상태 보고, 서비스 복구 설정 등 C#의 `ServiceBase`가 제공하는 핵심 기능을 모두 지원한다. "Windows 서비스는 C#이 편하다"는 2020년대 초반의 인식이며, 2026년 현재 Rust의 Windows 생태계는 상당히 성숙했다.

**5) AI 지원 개발이 Rust 학습 곡선을 실질적으로 완화**

2026년 현재 AI 코딩 어시스턴트(Claude, Copilot 등)는 Rust 코드 생성에 매우 능숙하다. 소유권/라이프타임 오류 해결, unsafe 코드 검토, Win32 API FFI 래핑 등 Rust의 전통적 난관을 AI가 상당 부분 해소한다. .NET 개발자가 Rust를 학습할 때 체감하는 난이도는 3-4년 전과 비교해 현저히 낮아졌다.

### 2.3 예상 반론에 대한 재반박

**반론: "IPC 분리이므로 FFI와 다르다. 결합도가 낮다."**

> 프로세스 경계 분리가 코드 결합도를 낮추는 것은 맞다. 그러나 **운영 결합도**는 여전히 높다. 두 프로세스가 동시에 정상 동작해야 시스템이 작동하므로, 장애 시나리오는 단일 프로세스 대비 4배로 늘어난다(A만 장애, B만 장애, 통신 장애, 양쪽 버전 불일치). Named Pipe 연결이 끊어졌을 때의 복구 로직, 한쪽 프로세스가 비정상 종료했을 때의 재시작 순서 등 "IPC이니까 간단하다"는 말로는 커버되지 않는 운영 복잡도가 존재한다.

**반론: "MVP 속도가 더 중요하다. Rust 학습에 3개월을 쓸 수 없다."**

> MVP 속도를 위해 C# 서비스를 도입하면, Phase 2에서 Rust 통합을 위한 리팩토링 비용이 발생한다. 결국 Rust를 배워야 하는 시점이 "지금"에서 "나중"으로 미뤄질 뿐이며, "나중"에는 이미 C# 코드에 비즈니스 로직이 축적되어 전환 비용이 더 커진다. 기술 부채의 전형적 패턴이다. 차라리 Phase 0(2-4주) 기간에 Rust 집중 학습 + 프로토타입을 병행하는 것이 장기적으로 빠르다.

**반론: "한국 시장에서 Rust 개발자를 구하기 어렵다."**

> 사실이다. 그러나 C# + Rust 하이브리드는 이 문제를 해결하지 않는다. 오히려 "C#도 잘하고 Rust도 이해하는 개발자" 또는 "C# 전문가 + Rust 전문가 혼성 팀"이 필요해져서 채용 난이도가 더 올라간다. 단일 언어 선택이 채용 전략을 명확하게 만든다.

## 3. 입장 B 옹호 (C# 서비스 + Rust 워커 IPC 예외안)

### 3.1 핵심 논거

이론적으로 전체 Rust가 최적이라는 점에 이견은 없다. 그러나 **"지금 이 팀이 6-10주 안에 파일럿 운영 가능한 MVP를 만들 수 있는가"**라는 현실적 질문에 답해야 한다. 팀 전원이 .NET 10년 경력이고 Rust 경험이 0인 경우, 전체 Rust는 MVP 일정을 3-6개월 지연시킬 수 있다. IPC 기반 하이브리드는 FFI의 문제를 피하면서도 **각 팀원이 자신의 강점 언어로 기여할 수 있는 실용적 경로**를 제공한다.

### 3.2 기술적 근거

**1) Named Pipe IPC는 검증된 패턴이다**

Windows에서 Named Pipe는 커널이 관리하는 IPC 메커니즘으로, 오버헤드가 매우 낮다(로컬 머신 내 통신 시 지연 < 0.1ms). 프로토콜 버퍼(protobuf)나 MessagePack 등으로 직렬화하면 타입 안전한 프로세스 간 통신이 가능하다. AV/EDR 제품들도 서비스 프로세스 + 스캔 워커 프로세스를 분리하는 패턴을 널리 사용하고 있으며, 이는 산업 표준에 가깝다.

실제로 이미 합의된 아키텍처에서도 텍스트 추출과 NER은 **별도 워커 프로세스로 격리**하기로 했다. 즉 다중 프로세스 구조는 이미 전제이며, 그 프로세스 중 하나가 C#으로 작성된다고 해서 아키텍처가 근본적으로 달라지는 것은 아니다.

**2) Windows 서비스 영역에서 C#의 생산성 우위는 여전히 실질적**

Windows 서비스 개발에서 C#이 제공하는 이점은 `windows-service` crate로 완전히 대체되지 않는다:

- **WMI 쿼리**: C#에서는 `ManagementObjectSearcher` 한 줄로 가능한 작업이 Rust에서는 COM 인터페이스 직접 호출 필요
- **Windows 이벤트 로그**: C#의 `EventLog` 클래스 vs Rust의 raw Win32 API 래핑
- **Windows 작업 스케줄러 통합**: C#의 `TaskScheduler` COM interop vs Rust에서의 unsafe COM 호출
- **서비스 복구 설정**: SC Manager의 고급 설정(지연 재시작, 복구 액션 등)이 C#에서 더 직관적
- **레지스트리 조작**: C#의 `Microsoft.Win32.Registry` vs Rust의 `winreg` crate

이런 "Windows 시스템 통합" 작업은 MVP에서 상당한 비중을 차지하며, .NET 개발자가 이미 숙달한 영역이다.

**3) 역할 분리가 명확하면 이중 언어의 비용을 통제할 수 있다**

| 컴포넌트 | 언어 | 역할 |
|:---|:---|:---|
| 서비스 프로세스 | C# | SCM 등록, 스케줄링, 정책 수신, 서버 통신, 업데이트 관리 |
| 스캔 워커 | Rust | 파일 열기, 텍스트 추출, 정규식/체크섬 검증, NER 추론 |

서비스 프로세스는 "관리 계층"이고 스캔 워커는 "연산 계층"이다. 이 경계가 명확하면 프로토콜 변경 빈도가 낮아지고, 각 컴포넌트가 독립적으로 진화할 수 있다. 스캔 워커는 stdin/stdout이나 Named Pipe로 작업을 받아 결과를 반환하는 순수한 CLI 도구에 가까워지므로, 나중에 서비스 프로세스도 Rust로 전환할 때 스캔 워커는 그대로 유지된다.

**4) MVP 속도의 비즈니스 가치는 과소평가되고 있다**

Phase 0(2-4주) + Phase 1 MVP(6-10주) = 최대 14주라는 일정은 **팀이 이미 선택한 언어에 숙련되어 있다는 전제**하에 성립한다. Rust 경험 0인 팀이 전체 Rust를 선택하면:

- Rust 기본 학습: 2-4주 (소유권, 라이프타임, 에러 처리, async)
- Windows 특화 학습: 1-2주 (Win32 API, COM, 서비스 개발)
- 생산성이 C#의 50%에 도달하기까지: 추가 4-8주

이를 합산하면 MVP 일정이 최소 2-3개월 지연된다. 스타트업이나 신규 제품에서 3개월 지연은 시장 기회의 상실을 의미할 수 있다. "기술 부채를 나중에 갚겠다"는 계획이 항상 나쁜 것은 아니다. **의도적 기술 부채(deliberate technical debt)**는 비즈니스 가치를 먼저 검증한 뒤 리팩토링하는 합리적 전략이다.

**5) .NET 8+ Self-contained 배포는 런타임 의존성 문제를 완화**

.NET 8 이후 Self-contained 단일 파일 배포(`PublishSingleFile` + `PublishTrimmed`)가 가능하다. Trimmed self-contained 빌드의 크기는 약 15-30MB 수준으로 줄일 수 있으며, 별도 .NET Runtime 설치가 불필요하다. ~80MB라는 수치는 Full Runtime 기준이며, 실제 배포 시나리오에서는 상당히 압축된다.

### 3.3 예상 반론에 대한 재반박

**반론: "배포 복잡도가 2배로 늘어난다."**

> MSI 하나에 C# 서비스 + Rust 워커를 패키징하면 사용자 관점에서는 "설치 파일 하나"로 동일하다. 내부적으로 두 바이너리가 있지만, 이는 이미 합의된 "서비스 + 워커 프로세스 격리" 아키텍처에서 발생하는 구조이다. 전체 Rust라도 서비스 프로세스와 워커 프로세스는 분리되므로, "바이너리 2개"는 어차피 예상되는 구조이다.

**반론: ".NET Runtime의 메모리 오버헤드가 4GB PC에서 문제다."**

> .NET 8의 GC는 `Server GC -> Workstation GC` 전환, `GCHeapHardLimit` 설정 등으로 메모리 상한을 제어할 수 있다. 서비스 프로세스는 스캔을 수행하지 않으므로 메모리 사용량이 20-30MB 수준에서 안정적이다. 실제 메모리 집약적 작업(텍스트 추출, NER 추론)은 Rust 워커에서 수행되므로, .NET의 GC 부담은 서비스 관리 계층에 한정된다.

**반론: "나중에 Rust로 통합해야 할 때 전환 비용이 크다."**

> 서비스 프로세스의 역할이 "스케줄링 + 정책 수신 + 서버 통신"에 한정되어 있으면, 이 부분의 코드량은 전체의 20-30% 수준이다. 핵심 비즈니스 로직(스캔 엔진)은 이미 Rust로 작성되어 있으므로, 전환 비용은 관리 계층에 한정된다. 이는 전체 리라이트가 아니라 **부분 교체**이다.

## 4. 중립 심판 평가

### 4.1 입장 A의 강점과 약점

**강점:**

- **배포 단순성 논거가 가장 강력하다.** 50,000대 PC에 대한 대규모 배포에서 단일 바이너리의 가치는 아무리 강조해도 지나치지 않다. 기존 15년 운영 경험에서 "배포 실패로 인한 운영 사고"가 얼마나 빈번한지를 고려하면, 배포 복잡도를 줄이는 것은 기능 추가보다 중요하다.
- **장기 유지보수 비용 논거가 설득력 있다.** 두 언어 생태계의 유지는 시간이 지날수록 비용이 증가하는 구조이며, 이는 경험적으로도 충분히 검증된 패턴이다.
- **AI 지원 개발이 학습 곡선을 완화한다는 논거가 시의적절하다.** 2026년 현재 AI 코딩 도구의 Rust 지원 수준은 실질적으로 개발 생산성을 높이고 있다.

**약점:**

- **"Rust 전문가 충원"이라는 대안이 한국 시장 현실과 괴리된다.** 한국에서 Rust 전문가 채용은 매우 어려우며, 특히 Windows 시스템 프로그래밍 경험이 있는 Rust 개발자는 극소수이다.
- **팀 역량 전환에 필요한 시간을 과소평가할 수 있다.** AI 도구가 학습을 돕는 것은 사실이나, 소유권 시스템의 심층 이해, 비동기 런타임 선택, unsafe 코드 관리 등은 시간이 필요하다.
- **"전체 C#" 대안을 언급하면서도 구체적으로 분석하지 않는다.** 전체 C#이 전체 Rust보다 나은 선택일 수 있는 시나리오를 진지하게 검토하지 않았다.

### 4.2 입장 B의 강점과 약점

**강점:**

- **현실적 팀 상황을 직시한다.** 기술적 이상과 팀의 현재 역량 사이의 간극을 인정하고, 실행 가능한 경로를 제시한다.
- **역할 분리의 논리가 탄탄하다.** 관리 계층과 연산 계층의 분리는 아키텍처적으로 합리적이며, 이미 합의된 워커 프로세스 격리 원칙과도 정합한다.
- **MVP 속도의 비즈니스 가치를 정당하게 평가한다.** 제품이 시장에 도달하지 못하면 기술적 순수성은 의미가 없다.

**약점:**

- **"나중에 Rust로 전환" 계획의 실현 가능성이 불투명하다.** 역사적으로 "의도적 기술 부채의 나중 상환"은 대부분 실패한다. C# 서비스에 비즈니스 로직이 점진적으로 누적되면 전환 동기가 사라진다.
- **.NET Runtime의 메모리 오버헤드를 과소평가한다.** Workstation GC + GCHeapHardLimit으로 제어 가능하다고 하지만, GC 자체의 간헐적 지연(GC pause)은 "Quiet Agent" 목표와 충돌할 수 있다. 서비스 프로세스가 GC pause 동안 Named Pipe 응답을 지연시키는 시나리오가 가능하다.
- **프로토콜 버전 관리 문제를 경시한다.** C# 서비스와 Rust 워커 사이의 직렬화 프로토콜이 변경될 때, 양쪽을 동시에 업데이트하지 못하면 (한쪽만 업데이트된 상태) 통신 실패가 발생한다. 50,000대 환경에서 "양쪽 동시 업데이트"를 보장하는 것은 단일 바이너리 교체보다 훨씬 어렵다.

### 4.3 핵심 트레이드오프 분석

| 평가 기준 | 전체 Rust (입장 A) | C# + Rust IPC (입장 B) | 비고 |
|:---|:---|:---|:---|
| **배포 단순성** | 우월 | 열등 | 50K PC에서 이 차이는 운영 사고로 직결 |
| **메모리 풋프린트** | 우월 (10-20MB) | 열등 (40-70MB) | 4GB PC에서 유의미한 차이 |
| **MVP 개발 속도** | 팀 역량 의존 | 팀 역량 의존 | .NET팀이면 B 유리, Rust팀이면 A 유리 |
| **장기 유지보수** | 우월 | 열등 | 단일 언어 < 이중 언어 |
| **채용 용이성** | 열등 (한국 Rust 시장) | 보통 (C# 풍부, Rust 부족) | 하이브리드도 Rust 인력은 필요 |
| **Windows 시스템 통합** | 보통 (개선 중) | 우월 (성숙한 생태계) | WMI, 이벤트로그 등 |
| **업데이트/롤백 안전성** | 우월 | 열등 | 단일 바이너리 교체 vs 다중 컴포넌트 동기화 |
| **"Quiet Agent" 달성** | 우월 | 보통 | GC pause, 메모리 오버헤드 |

### 4.4 조건부 결론

이 논쟁은 **팀 역량이라는 단일 변수에 의해 결론이 갈린다.** 기술적 분석만으로는 결정할 수 없으며, 다음 조건에 따라 답이 달라진다.

**조건 1: 팀에 Rust 경험자가 1명 이상 있거나, 2-4주 내 Rust 전문가를 충원할 수 있는 경우**
- 결론: **전체 Rust (입장 A)**
- 이유: 50K PC 배포 환경에서 단일 바이너리의 운영적 이점이 MVP 속도 지연(2-4주)보다 장기적으로 가치가 크다.

**조건 2: 팀 전원이 .NET 전문가이고, Rust 경험이 전무하며, 충원도 불가능한 경우**
- 결론: 두 가지 하위 선택지가 있다.
  - **(2a) 전체 C#:** 스캔 엔진까지 C#으로 작성. 메모리/성능 제약을 AOT 컴파일(NativeAOT)이나 `Span<T>` 최적화로 완화. Phase 2에서 성능 병목이 확인되면 해당 모듈만 Rust로 재작성.
  - **(2b) C# 서비스 + Rust 워커 IPC (입장 B):** 단, 다음 전제 조건을 충족해야 한다:
    - Rust 워커는 CLI 도구 수준의 단순한 인터페이스로 유지
    - 프로토콜은 protobuf로 스키마 관리, 하위 호환성 규칙 적용
    - MSI 패키지에서 두 컴포넌트를 원자적으로 업데이트/롤백하는 메커니즘 구현
    - Phase 2까지 서비스 프로세스의 Rust 전환 로드맵을 문서화하고 실행

**조건 3: 팀 규모가 1-2명인 극소규모이고, 빠른 시장 검증이 최우선인 경우**
- 결론: **전체 C# (NativeAOT)**을 고려할 수 있다. 하이브리드의 운영 복잡도를 감당할 인력이 없다.

## 5. 최종 추천안

### 기본 추천: 전체 Rust (입장 A 채택)

**이유:**

1. **제품의 핵심 제약이 기술 선택을 지배한다.** 50,000대 PC 배포, 4GB 저사양 지원, "Quiet Agent" 차별화라는 세 가지 핵심 제약은 모두 Rust 단일 바이너리에 유리하다. 하이브리드 구조는 이 세 제약 모두에서 열등하다.

2. **IPC 예외안은 문제를 해결하지 않고 미룬다.** Rust 학습은 어차피 필요하다(스캔 엔진은 Rust). 하이브리드로 시작하면 "C# 서비스의 Rust 전환"이라는 추가 작업이 로드맵에 남는다. 처음부터 전체 Rust로 가는 것이 총 비용이 낮다.

3. **2026년의 Rust 학습 환경은 과거와 다르다.** AI 코딩 도구, 풍부한 학습 자료, 성숙한 crate 생태계(특히 `windows` crate)가 학습 곡선을 실질적으로 낮추고 있다. Phase 0(2-4주) 기간을 Rust 집중 학습 + 프로토타입 병행으로 활용하면, Phase 1 진입 시점에서 생산적인 Rust 개발이 가능하다.

### 예외 경로: 조건부로만 허용

팀의 Rust 숙련도가 **심각하게** 부족하고(전원 경험 0), Rust 전문가 충원이 **불가능**하며, MVP 일정이 **절대적으로** 고정된 경우에 한해 다음 순서로 대안을 검토한다:

1. **전체 C# + NativeAOT** (하이브리드보다 이쪽을 먼저 검토)
   - 단일 언어의 운영적 이점을 유지
   - NativeAOT로 GC/Runtime 오버헤드 최소화
   - 성능 병목이 확인되는 모듈만 Phase 2에서 Rust 재작성

2. **C# 서비스 + Rust 워커 IPC** (최후의 선택지)
   - 반드시 protobuf 스키마 + 원자적 업데이트 + Rust 전환 로드맵을 전제
   - Phase 2까지 서비스 프로세스의 Rust 전환을 실행하지 않으면 영구적 기술 부채로 고착

### 사용자 결정 요청 사항

최종 결정을 위해 다음 정보가 필요하다:

1. **현재 팀의 기술 스택 분포**: .NET / Rust / Go / 기타 경험자 비율
2. **Rust 전문가 충원 가능성**: 향후 1-2개월 내 채용 또는 외부 자문 가능 여부
3. **MVP 일정의 유연성**: Phase 0에서 Rust 학습 기간(2-4주)을 허용할 수 있는지
4. **기존 15년 운영 솔루션의 기술 스택**: 현행 시스템이 C#/.NET 기반인지 여부 (기존 코드 자산 활용 가능성)
