# 라운드 1 토론: Gemini 의견서

> 작성일: 2026-02-09
> 역할: Gemini (Main Architect)
> 대상 문서: Claude 및 Codex의 분석 리포트

---

## 개요

Claude와 Codex의 분석을 검토한 결과, **"아키텍처의 방향성(Rust 에이전트 + 하이브리드 탐지)"에는 전원 동의**하나, **"MVP의 현실적 범위"와 "운영 복잡도(mTLS, 실시간성)"에 대해서는 심각한 우려**가 제기되었습니다.

특히 `hwp-rs` 크레이트의 미성숙도(v0.2.0)와 ONNX Runtime의 CPU 환경 성능 제약은 초기 스펙을 재조정해야 할 강력한 근거입니다. 이를 바탕으로 7가지 논점에 대한 최종 의견을 정리합니다.

---

## 논점 1: NER 모델 선택

### 타 AI 의견 요약
- **Claude:** KoELECTRA-Small v3 (INT8 양자화) 추천. 성능/크기/속도 면에서 유일한 현실적 대안으로 평가.
- **Codex:** KoELECTRA-Small v3 (INT8) 동의. 하드웨어 제약(i3/4GB) 고려 시 대형 모델 불가 판정.

### Gemini 의견: **동의 (강력 권장)**
- **근거:**
    1. **크기:** INT8 양자화 시 약 **14MB**로, 배포 및 메모리 상주 부담이 미미합니다.
    2. **성능:** 한국어 PII(주소, 이름) 탐지에서 RoBERTa-Large와 유의미한 차이가 없으면서 속도는 5배 이상 빠릅니다.
    3. **현실성:** CPU Only 환경에서 Transformer 기반 모델을 돌리려면 이 선택지 외에는 대안이 없습니다.
- **보완책:** 모델 파일 자체를 암호화하여 로컬 디스크에 저장하고, 메모리 로드 시에만 복호화하는 보호 조치(Claude 제안)를 필수 적용합니다.

## 논점 2: 탐지 파이프라인 구조

### 타 AI 의견 요약
- **Claude:** Regex(1차) → Checksum(검증) → Context/NER(2차). "컨텍스트 룰"을 통한 오탐 억제 강조.
- **Codex:** Metadata → Regex(Aho-Corasick) → Validator → NER. "깔때기(Funnel)" 구조 강조.

### Gemini 의견: **동의 (하이브리드 + 엄격한 트리거)**
- **구체화:**
    1. **Stage 1 (Metadata):** 확장자/크기 필터링.
    2. **Stage 2 (Fast Scan):** Aho-Corasick으로 "키워드(이름, 주소, 전화 등)" 유무만 초고속 판별. 키워드 없으면 즉시 스킵.
    3. **Stage 3 (Regex & Validate):** 정규식 추출 후 Luhn/Checksum 검증. 통과 시 '확정'.
    4. **Stage 4 (NER):** **"검증 불가능한 패턴(이름, 주소)"이면서 "Stage 2 키워드 주변"인 경우에만** 제한적으로 수행.
- **근거:** NER은 정규식 대비 연산 비용이 수백 배 높으므로, 전체 텍스트가 아닌 **"의심 구간(Suspicious Span)"**만 잘라내어 추론해야 합니다.

## 논점 3: 기술 스택 (Rust vs 대안)

### 타 AI 의견 요약
- **Claude:** Client는 Rust (단일 바이너리, 안전성), Server는 Go (생산성).
- **Codex:** Client는 Rust (성능). Server는 팀 역량에 따라 Go/C# 선택.

### Gemini 의견: **Client는 Rust, Server는 Go 확정**
- **Client (Rust):**
    - 50,000대 PC에 배포되는 에이전트는 **메모리 누수나 GC로 인한 끊김이 없어야 합니다.** Rust는 이를 보장하는 최적의 언어입니다.
    - C++ 대비 생산성과 안전성이 월등합니다.
- **Server (Go):**
    - 고루틴(Goroutine)을 이용한 대규모 동시성 처리가 탁월하여, 수만 대 에이전트의 요청을 처리하기에 적합합니다.
    - Rust로 서버를 짜는 것은 생산성 측면에서 과도할 수 있습니다(Codex 의견 수용).

## 논점 4: MVP 범위

### 타 AI 의견 요약
- **Claude:** **범위 축소 강력 제안.** 실시간 감지와 NER을 MVP에서 제외하거나 제한적으로 적용. `txt/docx` 등 핵심 포맷만 지원.
- **Codex:** **"실시간"은 불가능.** "Eventual Consistency(최종 일관성)" 목표로 변경. "Panic Mode" 도입 제안.

### Gemini 의견: **대폭 축소 및 '베타' 전략 수용**
- **수정 제안:**
    1. **NER:** MVP에는 포함하되 **"Labs(베타)" 기능**으로 기본 비활성화(Opt-in) 출시.
    2. **실시간 감지:** **"핵심 경로(바탕화면, 문서)"만 실시간 감시.** 나머지 전체 볼륨은 **"스케줄링 증분 스캔"**으로 전환.
    3. **파일 포맷:** `HWP` 지원은 **"실험적 기능"**으로 격하.
        - **근거:** 조사 결과 `hwp-rs` 크레이트는 아직 **v0.2.0** 수준으로 프로덕션에 쓰기에 미성숙합니다. 자체 파서를 만들거나 안정화될 때까지 기다려야 합니다.
    4. **업데이트:** 에이전트 자체 자동 업데이트 기능 제외. 기존 배포 시스템(PMS/AD) 활용.

## 논점 5: 보안/라이선스 설계

### 타 AI 의견 요약
- **Claude:** **mTLS 필수**, 서명된 라이선스 리스(Lease), 데이터 최소화(Tokenization).
- **Codex:** **mTLS는 운영 지옥(Operational Hell).** 5만 대 인증서 관리는 불가능하므로 JWT/Token 방식 제안.

### Gemini 의견: **mTLS 유지하되 하이브리드 등록 방식 도입**
- **절충안:**
    - 보안성과 "장치 식별(Device Identity)"을 위해 mTLS는 포기할 수 없는 차별점입니다.
    - 단, **최초 등록(Enrollment)**은 **"일회용 토큰(Join Token)"** 방식을 사용하여 복잡도를 낮추고, 등록 성공 시 인증서를 발급받는 구조로 갑니다.
    - 인증서 갱신 실패 시를 대비한 Fallback(Token 인증)을 예비로 둡니다.
- **데이터 최소화:** Claude의 "원문 미전송" 원칙을 철저히 준수합니다. 이는 보안 리스크를 줄일 뿐 아니라, 고객사 도입 설득에 핵심 논리가 됩니다.

## 논점 6: 현실적 구현 가능성 (팀 규모, 기간)

### 타 AI 의견 요약
- **Claude:** 현재 스펙대로면 3~5명 팀으로 **최소 2년** 소요.
- **Codex:** 현재 범위로는 불가능. 핵심 기능만 남겨야 함.

### Gemini 의견: **"외부 프로세스 격리" 구조로 리스크 완화**
- **구현 전략:**
    - 모든 기능을 하나의 Rust 바이너리에 넣지 않습니다.
    - **`Text Extractor`와 `NER Engine`은 별도 워커 프로세스로 분리**합니다.
    - **이유:** 불안정한 `hwp-rs`나 PDF 파서가 크래시를 일으켜도, 메인 에이전트(서비스)는 죽지 않고 살아남아야 합니다. 이것이 3~5명 팀이 "안정적인" 제품을 만드는 유일한 길입니다.

## 논점 7: 경쟁 차별화 전략

### 타 AI 의견 요약
- **Claude:** "Privacy by Design" (최소 수집) 및 "경량 에이전트".
- **Codex:** "False Positive Zero"는 불가능. 사용자가 튜닝 가능한 "투명성" 강조.

### Gemini 의견: **"투명하고 안전한 경량 에이전트"**
- **메시지:**
    - "우리는 당신의 데이터를 훔쳐보지 않습니다(Privacy by Design)."
    - "당신의 PC를 느리게 하지 않습니다(Rust + Resource Governor)."
    - "오탐인지 아닌지 근거를 보여줍니다(Explainable Detection - Rule/Context 명시)."

## 추가 논점 8: 텍스트 추출 전략 (Critical Risk)

### Gemini 의견: **외부 라이브러리/도구 적극 활용**
- Rust 생태계의 문서 파싱 라이브러리는 아직 `Python`이나 `Java` 생태계만큼 성숙하지 않습니다(`hwp-rs` 사례).
- **대안:** MVP에서는 `Apache Tika` 같은 무거운 서버 사이드 추출기를 쓰지 않고, 클라이언트단에서 가능한 **`txt`, `csv`, `ooxml` (zip 기반이라 Rust로 처리 용이)에 집중**합니다. 난이도 높은 포맷(PDF, HWP, Doc)은 오픈소스 바이너리 래퍼를 쓰거나 지원을 연기합니다.

---

### 결론

우리는 **"꿈의 스펙"에서 "현실의 제품"으로** 내려와야 합니다.
MVP의 핵심 가치는 **"죽지 않는 에이전트"**와 **"거짓말하지 않는 탐지(정규식+체크섬)"**입니다. NER과 실시간 감지는 이 기반 위에 올라가는 **"프리미엄 기능"**이어야 합니다.
