**토론 일자:** 2026-02-11
**참여자:** 입장A 옹호자, 입장B 옹호자, 중립 심판

## 1. 배경 및 쟁점 요약

PII Scanner 프로젝트의 공수와 일정에 대해 두 진영이 크게 다른 추정치를 제시하고 있다. 핵심 쟁점은 **"무엇을 만들 것인가"의 정의 차이**에서 비롯된다.

- **입장 A (보수적 추정):** 전체 제품 = 30-45 인월(제품화 버퍼 +30% 포함). 4-5명 팀 기준 6-9개월(MVP까지). 서버, 관리 대시보드, 보안 감사, MSI 패키징, EV 서명 등 "판매 가능한 제품"을 스코프로 잡는다.
- **입장 B (공격적 추정):** Phase 0(2-4주) + Phase 1 MVP(6-10주) = 8-14주로 파일럿 운영 가능. 서버는 모놀리스 + 최소 콘솔로 공수를 축소하며, "상용급"이 아닌 "파일럿 운영 가능" 수준을 목표로 한다.
- **Gemini 분석:** 업계 벤치마크 기반으로 DLP 코어(20-30 인월) + 관리 콘솔(30-40 인월) + QA/제품화(10-15 인월) = 60-85 인월(GA 기준)을 제시. MVP는 GA의 50-60%인 6-7개월로 추산.

**합의 실패의 본질:** 추정 스코프가 다르다. 입장 A는 "MVP + 서버/대시보드 + 제품화"의 전체 범위를, 입장 B는 "100-500대 내부 테스트 PC에서 동작하는 최소 파일럿"만을 대상으로 한다. 즉 **같은 질문에 답하고 있지 않다.**

**잠정 추천:** Phase 0+1 = 14-24주 (판매 가능 수준의 MVP). 이 문서에서 이 추정의 합리성을 검증한다.

**고려해야 할 핵심 변수:**
- 개발자가 해당 도메인에서 15년 경험을 보유한다는 사실
- Rust 학습 곡선이 일정에 미치는 영향
- 한국 기업 조달 사이클의 특수성
- 소프트웨어 추정의 "불확실성 원뿔(Cone of Uncertainty)"

## 2. 입장 A 옹호 (보수적 추정: 30-45 인월)

### 2.1 핵심 논거

"판매 가능한 제품"을 만드는 것이 목표라면, 8-14주 추정은 서버/대시보드 공수를 심각하게 과소평가하고 있다. 기업용 보안 솔루션은 에이전트(클라이언트)만으로 팔리지 않는다. **관리자가 수만 대 PC의 탐지 현황을 보고, 정책을 배포하고, 보고서를 생성할 수 있는 관리 콘솔이 없으면 기업 고객은 구매하지 않는다.** 이 서버/대시보드 개발 공수는 클라이언트 에이전트 자체보다 같거나 더 클 수 있으며, 이를 "모놀리스 + 최소 콘솔"로 축소하는 것은 기업 영업 현실과 맞지 않는다.

### 2.2 기술적 근거

**1) 서버/대시보드 공수는 전체의 40-60%를 차지한다**

Gemini의 업계 벤치마크가 이를 뒷받침한다. DLP 코어(20-30 인월) 대비 관리 콘솔(30-40 인월)이 더 크다. 이는 우연이 아니라 기업용 보안 솔루션의 구조적 특성이다:

- **정책 관리 시스템:** 부서별/그룹별 스캔 정책 배포, 버전 관리, 롤백
- **대시보드:** 실시간 탐지 현황, 부서별 통계, 시계열 추이, 위험도 히트맵
- **보고서 엔진:** 경영진용 요약 보고서, 감사용 상세 보고서, 스케줄 보고서
- **알림 시스템:** 대량 PII 탐지 시 관리자 알림, 에스컬레이션 워크플로우
- **에이전트 관리:** 배포 상태 모니터링, 버전 관리, 원격 설정 변경, 상태 진단
- **사용자/권한 관리:** RBAC, AD/LDAP 연동, 감사 로그

"목록 뷰(List view)" 하나로 이 모든 것을 대체할 수 있다고 보는 것은 기업 시장의 요구사항을 이해하지 못한 추정이다.

**2) 한국 기업 조달 사이클이 일정을 추가로 압박한다**

한국 대기업/공공기관의 보안 솔루션 도입 과정은 다음 단계를 거친다:

- BMT(Benchmark Test): 경쟁 제품과 나란히 테스트 (2-4주)
- 보안성 검토: ISMS-P 관점의 보안 아키텍처 검토 (2-4주)
- PoC 운영: 실제 환경에서 시범 운영 (4-8주)
- 구매 결재: 내부 승인 프로세스 (2-4주)

즉 **제품이 "파일럿 수준"이면 BMT에서 탈락**한다. 경쟁 제품은 이미 10년 이상 성숙한 관리 콘솔을 갖추고 있다. 파일럿으로 시작하되, 첫 BMT까지 관리 콘솔이 일정 수준에 도달해야 한다.

**3) 제품화 버퍼 +30%는 보수적이 아니라 현실적이다**

소프트웨어 추정의 "불확실성 원뿔(Cone of Uncertainty)"에 따르면, 프로젝트 초기의 추정 오차 범위는 0.25x ~ 4x이다. 즉 초기 추정이 10개월이면 실제는 2.5개월 ~ 40개월 사이에 분포한다. +30% 버퍼는 이 불확실성을 감안하면 오히려 낙관적이다.

제품화에 필요한 항목들은 개발자들이 흔히 간과하는 영역이다:

- **MSI 설치/제거 패키지:** 자동 설치, 그룹 정책 배포(GPO), SCCM 배포 지원
- **EV 코드 서명:** 인증서 발급 프로세스만 1-2주, Windows SmartScreen 신뢰 축적
- **AV 화이트리스트:** 주요 AV 제품에 오탐 방지를 위한 사전 등록
- **보안 감사:** 기업 고객의 보안팀이 요구하는 침투 테스트, 코드 감사
- **문서화:** 관리자 가이드, API 문서, 설치 가이드, 장애 대응 매뉴얼
- **국제화:** 최소한 한국어/영어 UI

**4) Rust 학습 곡선은 실제로 일정에 영향을 미친다**

팀이 Rust에 익숙하지 않다는 전제에서, 다음은 피할 수 없는 현실이다:

- 소유권(ownership) 시스템, 라이프타임, 트레이트 등 Rust 고유 개념 학습: 2-4주
- Windows 특화 개발(Win32 API, COM, 서비스, ONNX Runtime 바인딩): 1-2주
- 생산성이 기존 언어의 70%에 도달하기까지: 추가 4-6주
- 합산: **최소 7-12주의 생산성 감소 기간**

이 기간 동안 주당 산출물은 기존 언어 대비 50-70% 수준이므로, 실효 인월은 명목 인월의 60-70%에 불과하다. 8-14주 추정이 이 감소를 반영하고 있는지 의문이다.

**5) 15년 도메인 경험은 공수를 줄이지만 "만큼"은 아니다**

개발자가 15년간 50,000대 규모에서 PII 탐지 솔루션을 운영한 경험은 명백한 자산이다. 요구사항 분석, 아키텍처 설계, 운영 시나리오 파악에서 신규 진입자 대비 수개월을 절약할 수 있다. 그러나 이 경험이 줄여주는 것은 **"무엇을 만들 것인가"를 파악하는 시간**이지, **"어떻게 구현할 것인가"의 코딩 시간**이 아니다. 설계는 빨리 끝나도 구현은 코드를 타이핑하는 물리적 시간이 필요하다.

### 2.3 예상 반론에 대한 재반박

**반론: "파일럿으로 시작해서 점진적으로 제품화하면 된다. 처음부터 전체 범위를 잡을 필요가 없다."**

> 동의한다. 그러나 "파일럿 8-14주"와 "제품 6-9개월" 사이에는 **4-5개월의 추가 개발**이 필요하다는 점을 명시해야 한다. 파일럿 일정만 제시하고 제품화 일정을 언급하지 않으면, 이해관계자(경영진, 투자자)에게 잘못된 기대를 심어주게 된다. "14주면 끝나는 줄 알았는데 왜 9개월이 걸리나?"라는 질문을 받게 된다.

**반론: "서버를 모놀리스 + 최소 콘솔로 시작하면 공수가 크게 줄어든다."**

> Phase 1에서는 줄어든다. 그러나 Phase 2에서 관리 콘솔을 확장할 때 **모놀리스의 리팩토링 비용**이 추가된다. 초기에 최소한의 모듈화 없이 빠르게 만든 모놀리스는, 사용자 수가 늘어나고 기능이 추가될 때 병목이 된다. 이 리팩토링 비용까지 포함하면 총 공수는 처음부터 적절히 설계한 경우와 크게 다르지 않다.

**반론: "Gemini의 60-85 인월은 업계 평균이다. 우리는 더 빠를 수 있다."**

> 더 빠를 수 있다는 점은 인정한다. 도메인 전문성, AI 코딩 도구, 최신 프레임워크 활용 등으로 업계 평균의 50-70% 수준으로 단축할 수 있다. 이를 반영하면 30-60 인월 범위이며, 이것이 바로 입장 A의 추정(30-45 인월)과 일치한다. 60-85 인월에서 효율화를 감안해도 30 인월 이하로 내려가기는 어렵다.

## 3. 입장 B 옹호 (공격적 추정: 8-14주 파일럿)

### 3.1 핵심 논거

"판매 가능한 완성 제품"이 아니라 **"고객 피드백을 받을 수 있는 최소 동작 버전"**을 먼저 만드는 것이 올바른 전략이다. 30-45 인월을 투입하여 9개월 후에 제품을 공개했는데, 시장이 원하지 않는 방향이었다면 모든 투자가 매몰 비용이 된다. **8-14주 파일럿은 "만들 가치가 있는가?"를 검증하는 최소 비용의 실험**이다. 입장 A는 "무엇을 만들 것인가"에만 집중하고, "만들 가치가 있는가?"라는 선행 질문을 건너뛰고 있다.

### 3.2 기술적 근거

**1) "파일럿 운영 가능"의 가치를 과소평가해서는 안 된다**

파일럿은 "미완성 제품"이 아니라 **"가설 검증 도구"**이다. 이 프로젝트의 핵심 가설은 다음과 같다:

- 가설 1: 정규식+체크섬 1차 필터 + KoELECTRA NER 2차 검증이 기존 정규식 전용 방식보다 오탐률을 유의미하게(50% 이상) 낮출 수 있다.
- 가설 2: CPU-only ONNX 추론이 일반 기업 PC(i3/i5급)에서 사용자 업무에 지장 없는 수준으로 동작한다.
- 가설 3: USN Journal 기반 실시간 감지가 기존 주기적 풀스캔 대비 고객 만족도를 높인다.
- 가설 4: HWP/PDF 등 한국 기업 환경의 주요 문서 포맷에서 텍스트 추출이 안정적이다.

이 네 가지 가설 중 하나라도 실패하면 제품 전략을 수정해야 한다. **9개월을 투자한 뒤 실패를 발견하는 것보다, 14주 만에 발견하는 것이 비즈니스적으로 합리적이다.** 린 스타트업 방법론에서 "Build-Measure-Learn" 사이클을 빠르게 돌리는 것이 이에 해당한다.

**2) 서버/대시보드 최소화는 "생략"이 아니라 "단계적 구축"이다**

Phase 1에서 서버를 모놀리스 + 최소 콘솔로 시작하는 것은 서버 개발을 포기하는 것이 아니다:

| 기능 | Phase 1 (8-14주) | Phase 2 (15-24주) |
|:---|:---|:---|
| 에이전트 등록/인증 | 단순 API 키 | mTLS + 인증서 관리 |
| 정책 배포 | JSON 파일 기반 | 부서별/그룹별 정책 엔진 |
| 탐지 결과 수집 | REST API 로그 저장 | 시계열 DB + 집계 파이프라인 |
| 대시보드 | 에이전트 목록 + 탐지 로그 조회 | 차트, 히트맵, 보고서 |
| 알림 | 이메일 알림 | 에스컬레이션 워크플로우 |
| 사용자 관리 | 단일 관리자 | RBAC + AD 연동 |

Phase 1의 서버 공수는 2-3 인월이면 충분하다. Go(또는 Python/FastAPI) + SQLite/PostgreSQL + 간단한 웹 UI(React 또는 서버 사이드 렌더링)로 구현 가능하다. 이것으로 파일럿 100-500대 PC를 관리하는 데 기능적 부족함이 없다.

**3) 15년 도메인 전문성이 공수를 실질적으로 줄인다**

입장 A는 도메인 경험이 "무엇을 만들 것인가"만 줄인다고 주장하지만, 실제로는 다음 영역에서도 공수가 감소한다:

- **아키텍처 결정:** 후보 기술을 비교 평가하는 시간이 대폭 단축. 어떤 접근이 실패할지 이미 알고 있다.
- **엣지 케이스 대응:** 15년간 축적된 "이런 파일에서 이런 문제가 발생한다"는 지식이 디버깅 시간을 절약.
- **운영 설계:** 50,000대 환경의 배포/업데이트/장애 대응 패턴을 이미 알고 있으므로, 시행착오 없이 올바른 설계를 바로 적용.
- **요구사항의 우선순위:** 고객이 실제로 사용하는 기능과 거의 사용하지 않는 기능을 구분할 수 있어, MVP 스코프를 정확히 잡을 수 있다.

이 효과를 보수적으로 잡아도 **전체 공수의 20-30% 절감**이 가능하다. 신규 진입자 대비 가장 큰 차이는 "만들지 않아야 할 것을 안다"는 점이다.

**4) 애자일 점진 전달이 위험을 분산한다**

8-14주 추정은 "14주 후에 모든 것이 완성된다"가 아니라 "14주 후에 검증 가능한 첫 번째 인크리먼트가 나온다"는 의미이다. 이후 2주 단위 스프린트로 기능을 점진적으로 추가한다:

- Sprint 1-7 (14주): 핵심 에이전트 + 최소 서버 = 파일럿 가능
- Sprint 8-12 (10주): 관리 콘솔 확장 + 보안 강화 = PoC 가능
- Sprint 13-17 (10주): 제품화(MSI, 서명, 문서화) = BMT 가능

총 34주(약 8개월)이면 BMT 참여 가능 수준에 도달한다. 이는 입장 A의 6-9개월과 사실상 유사하지만, **14주 시점에 이미 고객 피드백을 받기 시작한다는 점이 결정적으로 다르다.**

**5) 스코프 크리프를 통제하는 열쇠는 "버전 1.0의 정의"이다**

30-45 인월 추정의 위험은 "어차피 오래 걸리니까"라는 인식이 스코프 크리프(scope creep)를 유발한다는 점이다. "9개월 프로젝트"라는 프레임이 설정되면, 6개월 시점에 "이것도 넣자, 저것도 넣자"가 발생하여 12개월, 18개월로 늘어나는 패턴은 소프트웨어 개발에서 매우 흔하다.

반면 "14주 파일럿"이라는 짧은 타임박스는 **스코프를 물리적으로 제한**한다. 14주 안에 들어갈 수 있는 것만 넣어야 하므로, 자연스럽게 "정말 필수적인 것"만 남는다. 이 규율이 전체 프로젝트의 효율을 높인다.

### 3.3 예상 반론에 대한 재반박

**반론: "파일럿 수준으로는 BMT에서 탈락한다. 한국 기업 시장의 현실을 모른다."**

> BMT에 참여하는 것은 Phase 2-3의 목표이지 Phase 1의 목표가 아니다. Phase 1 파일럿의 목적은 **기술 가설 검증**과 **초기 고객(early adopter) 확보**이다. 15년 운영 경험에서 축적된 기존 고객 관계를 활용하면, 정식 BMT 이전에 비공식 PoC로 파일럿을 운영할 수 있는 고객사를 확보할 수 있다. 이 초기 고객의 피드백이 Phase 2의 방향을 결정한다.

**반론: "서버를 모놀리스로 시작하면 나중에 리팩토링 비용이 든다."**

> Phase 1의 서버는 의도적으로 단순하게 만든다. 코드량 자체가 적으므로(2-3 인월 분량) 리팩토링 비용도 적다. 모놀리스가 문제가 되는 것은 수만 줄의 비즈니스 로직이 축적된 후이지, 수천 줄 수준의 Phase 1 서버에서는 아니다. 필요하면 Phase 2에서 처음부터 다시 작성해도 2-3 인월이면 된다.

**반론: "Rust 학습 곡선을 고려하면 8-14주는 비현실적이다."**

> Phase 0(2-4주)가 바로 이 문제를 해결하기 위해 설계되었다. Phase 0의 목적은 세 가지이다:
> 1. HWP/PDF 추출 성공률 검증
> 2. NER 성능 벤치마크
> 3. **Rust 개발 환경 세팅 + 핵심 패턴 학습**
>
> Phase 0에서 Rust로 파일 스캔 프로토타입을 만들면서 학습과 검증을 동시에 수행한다. Phase 0이 끝나는 시점에 팀은 최소한 Rust로 파일 I/O, 정규식 매칭, ONNX 추론 호출을 할 수 있는 수준에 도달해야 한다. 이것이 불가능하다고 판단되면 Phase 0에서 Kill/Pivot 결정을 내린다.

## 4. 중립 심판 평가

### 4.1 입장 A의 강점과 약점

**강점:**

- **서버/대시보드 공수의 현실적 평가가 가장 강력한 논거이다.** 기업용 보안 솔루션에서 관리 콘솔의 비중을 정확히 짚고 있다. Gemini의 업계 벤치마크(관리 콘솔 30-40 인월 vs DLP 코어 20-30 인월)가 이를 뒷받침한다. "목록 뷰 하나"로 기업 고객을 설득할 수 없다는 지적은 한국 시장 현실에 부합한다.
- **한국 기업 조달 사이클(BMT, 보안성 검토, PoC)을 고려한 역산 일정이 설득력 있다.** 제품이 시장에 도달하려면 BMT 참여 가능 수준까지 올려야 하며, 이것이 최종 마일스톤이 되어야 한다.
- **불확실성 원뿔(Cone of Uncertainty) 언급은 추정론의 정석이다.** 초기 추정의 오차가 0.25x~4x라는 사실을 감안하면, +30% 버퍼는 실제로 보수적이라고 보기 어렵다.
- **제품화 항목(MSI, EV 서명, AV 화이트리스트, 보안 감사)의 구체적 열거가 설득력 있다.** 이 항목들은 개발자가 "개발 완료"를 선언한 후에도 수주에서 수개월이 소요되며, 흔히 일정에서 누락된다.

**약점:**

- **30-45 인월 추정 자체의 근거가 다소 모호하다.** "업계 벤치마크의 50-70%"라는 할인율의 정당성이 감(feeling)에 의존하고 있다. 구체적인 태스크 분해(WBS) 없이 톱다운 추정만으로는 신뢰도가 제한적이다.
- **"처음부터 전체를 만들어야 한다"는 뉘앙스가 폭포수(Waterfall) 사고에 가깝다.** 현대 소프트웨어 개발에서 "6-9개월 후에 첫 릴리스"는 위험한 접근이다. 3개월 시점에 핵심 가설이 무효화되면 나머지 투자가 낭비된다.
- **초기 고객 피드백의 가치를 저평가한다.** 15년 경험이 있어도 "새로운 제품"에 대한 시장 반응은 예측할 수 없다. 빠른 피드백 루프의 가치를 공수 추정에 반영하지 않고 있다.
- **Rust 학습 비용을 일정 영향으로만 보고, Phase 0에서의 흡수 가능성을 충분히 검토하지 않는다.**

### 4.2 입장 B의 강점과 약점

**강점:**

- **가설 검증 우선(Build-Measure-Learn) 접근이 제품 개발 방법론으로 올바르다.** 특히 NER 기반 PII 탐지의 실효성은 실제 고객 데이터로 검증해야만 확인 가능한 가설이다. 이것을 9개월 후가 아니라 14주 후에 검증하는 것은 명백한 이점이다.
- **Phase 0의 Kill/Pivot 결정 포인트가 위험 관리에 효과적이다.** 2-4주의 소규모 투자로 기술적 실현 가능성을 검증한 뒤, 본격 투자 여부를 결정하는 구조는 투자 대비 위험을 최소화한다.
- **타임박스가 스코프 크리프를 자연스럽게 제한한다는 지적이 실용적이다.** "9개월 프로젝트"보다 "14주 파일럿 + 10주 확장 + 10주 제품화"가 관리 가능한 단위이다.
- **15년 도메인 전문성의 "무엇을 만들지 말아야 하는지 안다"는 효과를 잘 포착했다.** 이것이 공수 절감의 가장 큰 원천이다.

**약점:**

- **서버/대시보드 공수를 2-3 인월로 잡은 것은 과소평가이다.** "에이전트 목록 + 탐지 로그 조회"만으로도 인증, 세션 관리, CORS, 에러 처리, 페이지네이션, 검색, 필터링 등 웹 애플리케이션의 기본 공수가 발생한다. 최소 콘솔이라도 3-4 인월이 현실적이다.
- **"파일럿에서 제품까지"의 총 일정을 명시적으로 제시하지 않아 기대 관리에 실패할 수 있다.** 8-14주만 강조하면 이해관계자가 "14주면 끝"으로 오해한다.
- **한국 기업 시장에서 "파일럿"만으로 계약을 따낼 수 있는 경로가 불분명하다.** 비공식 PoC로 초기 고객을 확보할 수 있다는 주장은, 기존 고객 관계가 실제로 존재한다는 전제에 의존한다.
- **보안 감사, 인증 관련 공수가 추정에서 완전히 누락되어 있다.** 기업 보안 솔루션은 보안 감사 없이 판매가 불가능하며, 이 과정은 수주에서 수개월이 소요된다.

### 4.3 핵심 트레이드오프 분석

| 평가 기준 | 입장 A (30-45 인월) | 입장 B (8-14주 파일럿) | 비고 |
|:---|:---|:---|:---|
| **추정 스코프** | 판매 가능 제품 전체 | 파일럿 운영 수준 | 같은 대상을 추정하고 있지 않음 |
| **위험 관리** | 낮음 (대규모 선행 투자) | 높음 (조기 검증) | 입장 B가 투자 위험 분산에 우월 |
| **시장 도달 속도** | 느림 (6-9개월 후 첫 릴리스) | 빠름 (14주 후 첫 파일럿) | 입장 B가 피드백 루프에 우월 |
| **서버/대시보드 현실성** | 현실적 (업계 벤치마크 부합) | 과소평가 위험 | 입장 A가 더 정직한 추정 |
| **제품화 공수 반영** | 포함 (+30% 버퍼) | 미포함 | 입장 A가 더 완전한 추정 |
| **스코프 크리프 위험** | 높음 (긴 타임라인) | 낮음 (짧은 타임박스) | 입장 B가 관리에 우월 |
| **기대 관리** | 명확 (총 공수 명시) | 불명확 (파일럿 이후 불투명) | 입장 A가 투명성에 우월 |
| **도메인 전문성 활용** | 부분 반영 | 적극 반영 | 입장 B가 더 현실적 할인율 적용 |
| **추정 신뢰도** | 중간 (톱다운) | 낮음 (바텀업 미비) | 둘 다 WBS 분해가 부족 |

**핵심 발견:** 두 입장은 **같은 질문에 대한 다른 답이 아니라, 다른 질문에 대한 각각의 답**이다. 입장 A는 "판매 가능한 제품을 만드는 데 얼마나 걸리나?"에 답하고, 입장 B는 "가설을 검증하는 데 얼마나 걸리나?"에 답한다. 따라서 두 입장은 **대립이 아니라 보완** 관계이다.

### 4.4 조건부 결론

**두 추정을 하나의 타임라인에 통합해야 한다.** "8-14주 파일럿" OR "6-9개월 제품"이 아니라, "8-14주 파일럿 THEN 추가 4-5개월 제품화 = 총 7-10개월"이 현실적 일정이다.

이를 위해 **단계별 예산과 Go/No-Go 게이트를 명확히 정의**해야 한다:

**Gate 0 (2-4주 시점): 기술 검증 완료**
- 투입: 2-4 인월
- 판단 기준: HWP/PDF 추출 성공률 > 90%, NER 추론 시간 < 200ms/페이지 (i5 기준), USN Journal 안정성 확인
- Kill 조건: 핵심 기술 가설이 실패하면 프로젝트 중단 또는 피벗

**Gate 1 (12-16주 시점): 파일럿 운영 가능**
- 투입: 추가 8-12 인월 (누적 10-16 인월)
- 판단 기준: 100-500대 PC에서 파일럿 운영 가능, 기본 서버/대시보드 동작
- Go/No-Go: 초기 고객 피드백이 긍정적이면 Phase 2 진행, 부정적이면 피벗

**Gate 2 (24-32주 시점): 판매 가능 MVP**
- 투입: 추가 10-15 인월 (누적 20-31 인월)
- 판단 기준: BMT 참여 가능 수준의 관리 콘솔, MSI 패키징, EV 서명, 기본 보안 감사 완료
- 목표: 첫 유료 고객 계약

**Gate 3 (36-44주 시점): 상용 GA**
- 투입: 추가 10-14 인월 (누적 30-45 인월)
- 판단 기준: 수천 대 규모 운영 가능, 완전한 관리 콘솔, 보안 인증

## 5. 최종 추천안

### 통합 일정: 단계별 게이트 방식 (총 30-45 인월, 9-11개월)

두 입장의 장점을 결합한 **게이트 기반 점진적 투자** 모델을 추천한다.

**Phase 0: 기술 검증 (2-4주, 2-4 인월)**
- Rust 프로토타입 + 핵심 기술 가설 검증
- **이 단계의 산출물이 이후 모든 추정의 근거가 된다.** 불확실성 원뿔이 가장 넓은 시점에서 추정을 확정하지 말고, Phase 0 이후 재추정하라.
- Kill/Pivot 결정 포인트

**Phase 1: 파일럿 MVP (10-12주, 8-12 인월)**
- 에이전트: 파일 스캔 + 정규식/체크섬 + NER + USN Journal 실시간 감지
- 서버: 모놀리스 + 기본 API + 최소 관리 콘솔
- 목표: 100-500대 내부/협력 고객 파일럿 운영
- **파일럿 종료 시점에 전체 일정을 재추정한다.** 이때의 추정은 불확실성 원뿔이 상당히 좁아져 있으므로, 30-45 인월이라는 범위를 +-20% 이내로 정밀화할 수 있다.

**Phase 2: 판매 가능 제품 (10-14주, 10-15 인월)**
- 관리 콘솔 확장: 대시보드, 보고서, 정책 관리
- 제품화: MSI 패키징, EV 서명, AV 화이트리스트
- 보안: 침투 테스트, 보안 감사
- 목표: BMT 참여, 첫 유료 고객

**Phase 3: 상용 GA (8-12주, 10-14 인월)**
- 대규모 운영 지원: 수천-수만 대 스케일링
- 완전한 관리 콘솔: AD 연동, RBAC, 감사 로그
- 문서화, 교육 자료, 기술 지원 체계
- 목표: 일반 판매 개시

### 핵심 원칙

1. **Phase 0 이후에 전체 일정을 확정하라.** 지금은 불확실성 원뿔이 너무 넓다. Phase 0의 4주를 투자하면 추정 정확도가 극적으로 개선된다.

2. **각 Gate에서 Go/No-Go를 판단하라.** 맹목적으로 9개월을 달리지 말고, 각 단계의 산출물과 고객 피드백을 기반으로 다음 단계 진행 여부를 결정하라.

3. **파일럿 일정과 제품 일정을 분리하여 커뮤니케이션하라.** 이해관계자에게 "14주면 파일럿, 판매 가능 제품은 7-8개월, GA는 10-11개월"이라고 명확히 전달하라. 기대 관리 실패가 일정 실패보다 더 파괴적이다.

4. **도메인 전문성을 최대한 활용하되, 기술 학습 비용을 과소평가하지 마라.** 15년 경험은 "무엇을 만들 것인가"와 "무엇을 만들지 말 것인가"를 가속하지만, Rust 학습은 별도의 투자가 필요하다. Phase 0에서 이 두 가지를 병행하라.

5. **공수 추정에 WBS(Work Breakdown Structure)를 적용하라.** 톱다운 추정(30-45 인월)과 바텀업 추정(태스크별 시간 합산)을 교차 검증하여 추정의 신뢰도를 높여라. 현재 두 입장 모두 WBS 없이 추정하고 있으며, 이것이 차이의 근본 원인이다.
