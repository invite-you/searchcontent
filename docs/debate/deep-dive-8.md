## 1. 배경 및 쟁점 요약

PII Scanner는 Windows PC에서 파일 내 개인정보를 실시간으로 탐지해야 한다. 핵심 3개 폴더(바탕화면, 문서, 다운로드)에 대해 `ReadDirectoryChangesW`를 사용한 파일 변경 감시(워처)를 구현하기로 합의했으나, **워처가 이벤트를 감지한 후 스캔을 언제 수행하는가**에 대해 의견이 갈린다.

**핵심 쟁점:**
- **입장 A (이벤트 감지 즉시 스캔):** `ReadDirectoryChangesW`로 파일 변경을 감지하면 즉시 스캔을 수행한다. 폭주(burst) 상황에서는 eventual mode로 전환하여 USN 저널 기반 증분 스캔으로 대체한다. 나머지 폴더는 USN 증분만 적용한다.
- **입장 B (큐잉 + 리소스 거버너):** 워처는 이벤트를 큐에 적재하는 역할만 담당한다. 실제 스캔 타이밍은 리소스 거버너가 CPU 사용률, 사용자 활동 상태(idle/active), 큐 백로그 크기 등을 종합 판단하여 결정한다. "Quiet Agent"로서 사용자 업무를 방해하지 않는 것이 최우선이다.

**합의 실패 이유:** 즉시 스캔의 "PII 노출 시간 최소화"라는 보안적 이점과, 큐잉 방식의 "사용자 업무 무방해"라는 UX 이점 사이의 트레이드오프에 대해 가중치 합의가 이루어지지 않았다.

**구도:** Codex와 Gemini가 입장 B를 지지하고, Claude가 입장 A를 주장하는 2:1 구도이다.

**고려 요소:**
- `ReadDirectoryChangesW`의 버퍼 오버플로우 위험(고활동 시)
- Word 자동 저장 등 단일 문서 저장 시 다수 이벤트 발생(~10개 이상의 임시 파일 생성/삭제/이름변경)
- 기존 솔루션 15년 운영에서 "업무 방해"가 핵심 불만이었다는 사실
- PII 노출 창(파일 생성~스캔 완료 사이의 시간) 관련 컴플라이언스 요구사항
- 디바운스(debounce) 전략
- Windows 유휴 감지(`GetLastInputInfo`)
- 우선순위 큐 설계
- 배치 처리 효율성

## 2. 입장 A 옹호 (이벤트 감지 즉시 스캔)

### 2.1 핵심 논거

1. **PII 노출 창 최소화가 보안의 본질이다.** PII Scanner의 존재 이유는 개인정보가 비보호 상태로 존재하는 시간을 최소화하는 것이다. 파일이 생성된 후 5초 안에 스캔하는 것과 5분 후에 스캔하는 것은 보안적으로 질적 차이가 있다. 특히 사용자가 PII가 포함된 파일을 외부로 전송하려는 시나리오에서, 즉시 감지만이 "전송 전 경고"를 가능하게 한다.

2. **"실시간"이라는 제품 가치의 핵심을 지킨다.** 고객이 "실시간 감지"를 구매할 때 기대하는 것은 파일 변경 즉시 감지이지, "시스템이 한가해지면 감지"가 아니다. 마케팅과 기술적 현실 사이의 괴리는 제품 신뢰도를 훼손한다.

3. **폭주 시 안전장치가 이미 설계되어 있다.** 즉시 스캔이 무조건적인 것이 아니라, 이벤트 폭주 시 eventual mode로 전환하여 USN 저널 기반 증분 스캔으로 대체한다. 이는 CPU 폭주를 방지하는 안전밸브 역할을 하면서도, 정상 상황에서는 즉시 스캔의 이점을 누릴 수 있다.

4. **핵심 3폴더의 이벤트율은 관리 가능한 수준이다.** 바탕화면, 문서, 다운로드 폴더는 사용자가 능동적으로 작업하는 영역이므로, 초당 수천 개의 이벤트가 발생하는 빌드 디렉토리나 temp 폴더와는 성격이 다르다. 일반 사무 업무에서 이 3개 폴더의 평균 이벤트율은 초당 1-5개 수준이며, 버스트 시에도 초당 50-100개를 넘기 어렵다.

### 2.2 기술적 근거

**1) 디바운스를 통한 이벤트 정규화**

Word 자동 저장이 10개 이상의 파일 시스템 이벤트를 발생시키는 것은 사실이나, 이는 디바운스로 해결 가능한 문제이다.

```
파일 변경 이벤트 수신
  → 경로별 디바운스 타이머(500ms) 시작
  → 500ms 내 동일 경로 이벤트 → 타이머 리셋
  → 타이머 만료 → 스캔 트리거
```

이 디바운스 전략은 Word의 `~$filename.docx` → `filename.docx` → 임시파일 삭제 시퀀스를 하나의 스캔 이벤트로 합산한다. 실제 스캔 대상은 최종 안정 상태의 파일 하나이므로, CPU 부하는 이벤트 수가 아닌 최종 파일 수에 비례한다.

**2) 폭주 감지 및 모드 전환 메커니즘**

```
[정상 모드: 즉시 스캔]
  이벤트율 < 임계치(예: 50/s) → 디바운스 후 즉시 스캔
  이벤트율 >= 임계치 → eventual mode로 전환

[Eventual Mode: USN 증분]
  ReadDirectoryChangesW 워처 일시 중단
  USN 저널 기반 증분 스캔 스케줄 등록
  이벤트율 안정화(10초간 임계치 미만) → 정상 모드 복귀
```

이 이중 모드 구조는 정상 업무 환경(95% 이상의 시간)에서 즉시 스캔의 보안 이점을 제공하면서, 예외적 폭주 상황에서만 USN 폴백으로 전환하여 시스템 안정성을 보장한다.

**3) 스캔 비용의 현실적 분석**

1차 스캔(정규식 + 체크섬 검증)의 비용은 파일당 5-20ms 수준이다. 이는 Word 문서 저장에 소요되는 시간(50-200ms)보다 짧다. 1차 스캔에서 의심 텍스트가 발견된 경우에만 2차 NER 추론(100-500ms)이 수행되므로, 대부분의 파일 변경 이벤트에 대한 즉시 스캔은 사용자가 체감하기 어려운 수준의 CPU 사용으로 완료된다.

**4) `ReadDirectoryChangesW` 버퍼 관리**

버퍼 크기를 64KB로 설정하면 약 2,000-4,000개의 이벤트를 수용할 수 있다. 디바운스 + 폭주 감지를 병행하면, 버퍼 오버플로우가 발생하기 전에 eventual mode로 전환되므로 이벤트 유실 위험이 최소화된다. 만일 버퍼 오버플로우가 발생하더라도, USN 저널이 백업 소스 역할을 하므로 이벤트가 영구적으로 유실되지 않는다.

### 2.3 예상 반론에 대한 재반박

**반론 1: "Codex PoC에서 즉시 스캔의 CPU 사용률이 72.6%에 달했다."**

> 재반박: Codex PoC는 480개 이벤트를 디바운스 없이 모두 개별 스캔하는 최악의 시나리오를 모델링했다. 실제로는 (1) 디바운스로 이벤트가 합산되고, (2) 핵심 3폴더에서 초당 200개 이벤트가 2초간 지속되는 상황은 일반 사무 환경에서 극히 드물며, (3) 폭주 감지 임계치에 도달하면 즉시 eventual mode로 전환된다. PoC의 `immediate` 정책은 안전장치가 없는 순수 즉시 스캔이며, 우리가 제안하는 "디바운스 + 폭주 시 모드 전환" 구조와는 다르다.

**반론 2: "기존 솔루션 15년 운영에서 업무 방해가 핵심 불만이었다. 즉시 스캔은 같은 실수를 반복한다."**

> 재반박: 기존 솔루션의 업무 방해는 "풀스캔 시 CPU 급증"과 "반복 팝업"이 원인이었지, "실시간 감지" 자체가 문제는 아니었다. 기존 솔루션은 애초에 실시간 감지 기능이 없었다. 우리의 즉시 스캔은 1차 경량 스캔(정규식+체크섬, 5-20ms)만 즉시 수행하고, 중량 스캔(NER)은 의심 파일에 한해 별도 처리한다. 이는 기존 풀스캔과 근본적으로 다른 접근이다.

**반론 3: "사용자가 파일을 저장하는 바로 그 순간에 CPU를 사용하면, 아무리 적은 양이라도 체감된다."**

> 재반박: 파일 저장 순간에 5-20ms의 CPU 사용은 Windows Defender의 실시간 보호가 사용하는 리소스보다 적다. 모든 Windows PC에서 이미 AV의 실시간 스캔이 파일 쓰기 시점에 동작하고 있으며, 사용자는 이를 인지하지 못한다. 우리의 1차 스캔이 AV 실시간 보호보다 가벼운 수준이라면, 사용자 체감에 미치는 영향은 무시할 수 있다. 또한 디바운스(500ms 대기)를 적용하면 저장 행위가 완료된 후에 스캔이 시작되므로, 저장 작업과 직접 경쟁하지 않는다.

## 3. 입장 B 옹호 (큐잉 + 리소스 거버너)

### 3.1 핵심 논거

1. **"Quiet Agent"가 이 제품의 존재 이유이다.** 기존 솔루션 15년 운영에서 가장 큰 불만은 "업무 방해"였다. 50,000대 PC의 사용자들이 반복적으로 제기한 핵심 페인 포인트를 해소하는 것이 이 제품의 존재 이유이며, 아키텍처의 모든 결정은 이 원칙에 종속되어야 한다. "5-20ms라서 괜찮다"는 엔지니어의 판단이 아니라, "사용자가 절대 인지하지 못한다"는 보장이 필요하다.

2. **스캔 타이밍의 결정권은 워처가 아니라 시스템 전체 상태에 있어야 한다.** 워처는 "무엇이 변경되었는가"만 알 뿐, "지금 스캔해도 되는가"를 판단할 위치에 있지 않다. CPU 사용률, 사용자 활동 상태, 현재 포그라운드 앱, 큐 백로그 크기, 배터리 상태(노트북) 등 다중 신호를 종합하여 스캔 타이밍을 결정하는 것은 별도의 리소스 거버너의 책임이다.

3. **이벤트 폭주는 "예외"가 아니라 "일상"이다.** Word 자동 저장(10분 간격), Chrome 다운로드, 압축 풀기, 메신저 파일 수신 등 일반 사무 환경에서도 버스트성 이벤트는 하루에도 수십 회 발생한다. "폭주 시 eventual mode 전환"이라는 안전장치가 빈번하게 작동한다면, 이는 사실상 기본 동작 모드가 eventual이 되어야 한다는 의미이다.

4. **Codex PoC가 수치로 증명했다.** 즉시 스캔은 CPU 점유율 72.6%, 큐잉+고정 예산(10%)은 P95 지연 60.9초, 동적 거버닝(active 10%/idle 50%)은 CPU 32.3%에 P95 지연 17.5초이다. 동적 거버닝이 CPU 사용과 지연 사이의 최적 균형점을 제공하며, 이 균형점은 "즉시 스캔"의 어떤 변형으로도 도달할 수 없다.

### 3.2 기술적 근거

**1) 리소스 거버너 아키텍처**

```
[워처 스레드]
  ReadDirectoryChangesW → 이벤트 수신
    → 파일 경로 + 타임스탬프를 lock-free 큐에 push
    → 즉시 반환 (CPU 비용 < 0.01ms)

[거버너 스레드] (100ms 주기)
  입력 신호:
    ├── GetLastInputInfo() → 마지막 사용자 입력 시각
    ├── GetSystemTimes() → 시스템 CPU idle 비율
    ├── 큐 크기 → 백로그 수준
    └── 포그라운드 윈도우 클래스 → 사용자 활동 유형

  결정 로직:
    IF 사용자 idle >= 5초 AND CPU idle > 70%:
      스캔 예산 = 높음 (CPU 50%, 배치 크기 10)
    ELIF 사용자 idle >= 2초 AND CPU idle > 50%:
      스캔 예산 = 보통 (CPU 20%, 배치 크기 3)
    ELIF CPU idle > 80%:
      스캔 예산 = 최소 (CPU 10%, 배치 크기 1)
    ELSE:
      스캔 예산 = 0 (대기)

[스캔 워커 스레드]
  거버너의 예산에 따라 큐에서 pop → 스캔 수행
  스캔 완료 후 소요 시간 보고 → 거버너가 다음 예산 조정에 반영
```

**2) 우선순위 큐 설계**

모든 이벤트가 동일한 우선순위를 갖는 것은 아니다. 다음 기준으로 우선순위를 차등 부여한다:

| 우선순위 | 조건 | 근거 |
|:---------|:-----|:-----|
| Critical | USB 드라이브에서의 파일 이동 | 외부 반출 위험 |
| High | 새 파일 생성 (이전에 스캔 안 됨) | PII 노출 창 시작 |
| Normal | 기존 파일 수정 | 내용 변경 가능성 |
| Low | 파일 이름 변경만 | 내용 미변경 |
| Deferred | 임시 파일 패턴 (`~$*`, `*.tmp`) | 최종 파일만 스캔 필요 |

Critical 우선순위 이벤트는 리소스 거버너의 예산과 무관하게 즉시 처리하여, 보안적으로 중요한 시나리오에서의 지연을 방지한다.

**3) 디바운스 + 큐잉 통합 전략**

```
이벤트 수신 → 경로별 디바운스 맵 확인
  → 동일 경로의 이벤트가 300ms 내에 존재?
    → Yes: 타이머 리셋, 우선순위 = max(기존, 신규)
    → No: 새 디바운스 타이머 등록
  → 타이머 만료 시: 우선순위 큐에 push
```

디바운스는 워처 스레드에서 수행하여 큐의 불필요한 팽창을 방지한다. Word 자동 저장 시 발생하는 10개 이상의 이벤트가 1개의 큐 항목으로 합산된다.

**4) 배치 처리 효율성**

큐에서 이벤트를 하나씩 처리하는 것보다 배치로 처리하는 것이 효율적이다:

- **파일 시스템 캐시 활용:** 같은 디렉토리의 파일들을 연속으로 스캔하면 디렉토리 메타데이터가 캐시에 남아 I/O가 감소한다.
- **정규식 엔진 웜업:** 정규식 패턴 컴파일은 1회만 필요하고, 배치 내에서 재사용된다.
- **NER 추론 배칭:** 의심 텍스트가 여러 파일에서 추출되면, ONNX Runtime에 배치로 전달하여 추론 효율을 높인다.
- **결과 보고 배칭:** 서버로의 결과 전송도 배치로 수행하여 네트워크 오버헤드를 줄인다.

**5) Windows 유휴 감지의 정교함**

`GetLastInputInfo`만으로는 "사용자가 쉬는 중"을 판단하기 부족하다. 다음 신호들을 조합한다:

- `GetLastInputInfo`: 키보드/마우스 입력이 없는 시간
- `GetSystemTimes`: 시스템 전체 CPU idle 비율
- 포그라운드 윈도우: 화면 보호기, 잠금 화면 여부
- `PowerReadACValueIndex`: 전원 정책(절전 모드 임박 여부)
- 프로세스 목록: CPU 집약적 앱(영상 편집, 컴파일 등) 실행 중 여부

이 다중 신호 조합으로 "사용자가 잠시 손을 뗀 순간"(2-5초)에도 경량 스캔을 수행할 수 있으며, "사용자가 회의/퇴근으로 자리를 비운 상황"(5분 이상)에서는 공격적으로 백로그를 처리할 수 있다.

### 3.3 예상 반론에 대한 재반박

**반론 1: "PII 노출 창이 길어지면 컴플라이언스 위반이다."**

> 재반박: 한국 개인정보보호법(PIPA)은 "개인정보를 즉시 탐지해야 한다"는 시간 요건을 명시하지 않는다. 요구사항은 "기술적/관리적 보호 조치"의 존재이며, "파일 변경 후 수분 내 탐지"는 충분한 보호 조치로 인정된다. 실제로 대부분의 기업이 현재 사용하는 솔루션은 일일 1회 또는 주간 풀스캔 방식이며, 이조차 컴플라이언스를 충족한다. 큐잉 방식의 P95 지연 17.5초(동적 거버닝)는 기존 풀스캔 대비 수천 배 빠른 것이다. 또한 Critical 우선순위(USB 반출 등)는 즉시 처리하므로, 실질적 보안 위험은 최소화된다.

**반론 2: "디바운스 + 폭주 전환이면 충분하다. 별도의 리소스 거버너는 과도한 복잡도이다."**

> 재반박: 디바운스와 폭주 전환은 "이벤트 수"만 고려하고 "시스템 상태"는 고려하지 않는다. 이벤트가 초당 5개(디바운스 후 2개)뿐이더라도, 사용자가 바로 그 순간 Excel에서 복잡한 계산을 실행 중이라면 추가 CPU 사용은 체감될 수 있다. 리소스 거버너는 이벤트율뿐 아니라 CPU 상태, 사용자 활동, 시스템 전체 부하를 고려하므로 "사용자가 절대 인지하지 못한다"는 보장에 가장 가까운 메커니즘이다. 구현 복잡도 측면에서, 거버너의 핵심은 100ms 주기의 상태 폴링 + 조건 분기이며, 이는 수십 줄의 코드로 구현 가능한 단순한 상태 머신이다.

**반론 3: "사용자가 PII 파일을 만들고 즉시 외부로 전송하면 감지하지 못한다."**

> 재반박: 이 시나리오는 큐잉 방식의 실질적 약점이다. 이에 대한 대응 전략은 두 가지이다. 첫째, 외부 전송 경로(USB, 클라우드 동기화 폴더, 메일 클라이언트 첨부 등)에 대한 이벤트는 Critical 우선순위로 분류하여 즉시 처리한다. 둘째, 이메일이나 메신저를 통한 전송은 네트워크 지연이 있으므로 수초 내 스캔 완료로도 "전송 완료 전 감지"가 가능하다. USB 복사는 Critical 즉시 처리로 대응한다. 즉, "모든 이벤트를 즉시 스캔"할 필요 없이, "보안적으로 중요한 이벤트만 즉시 처리"하는 선별적 접근이 효율적이다.

**반론 4: "Gemini의 Minifilter 분석은 MVP 범위를 벗어나는 것 아닌가?"**

> 재반박: Gemini가 Minifilter(`IRP_MJ_WRITE` 인터셉트)를 언급한 것은 즉시 스캔의 극단적 형태가 갖는 위험을 설명하기 위한 것이다. MVP에서 제안하는 것은 Minifilter가 아니라 사용자 모드의 `ReadDirectoryChangesW` 워처이므로, 파일 쓰기를 차단하지는 않는다. 그러나 핵심 메시지는 동일하다: 사용자의 I/O 활동과 동시에 CPU 집약적 작업을 수행하면 리소스 경합이 발생한다. 이는 Minifilter든 워처 + 즉시 스캔이든 동일한 문제이다.

## 4. 중립 심판 평가

### 4.1 입장 A의 강점과 약점

**강점:**

- **보안적 동기가 명확하다.** PII 노출 창 최소화는 보안 제품으로서의 정당한 목표이며, 고객(특히 CISO/DPO)에게 "실시간 감지"를 어필할 때 즉시 스캔은 직관적으로 이해하기 쉬운 가치이다.
- **디바운스 + 폭주 전환 설계가 실용적이다.** 무조건적인 즉시 스캔이 아니라 안전장치를 갖춘 단계적 접근이므로, 순수 즉시 스캔의 위험을 상당 부분 완화한다.
- **구현 복잡도가 상대적으로 낮다.** 디바운스 타이머 + 이벤트율 임계치 + 모드 전환은 리소스 거버너의 다중 신호 조합보다 구현과 디버깅이 단순하다.
- **USN 저널 폴백이 이벤트 유실을 방지한다.** `ReadDirectoryChangesW` 버퍼 오버플로우 시에도 USN 저널이 안전망 역할을 하므로, 데이터 무결성이 보장된다.

**약점:**

- **"5-20ms는 체감하기 어렵다"는 주장이 검증되지 않았다.** 단일 파일 기준으로는 맞지만, 사용자가 10개 파일을 연속 저장하거나 압축을 풀면 누적 부하가 발생한다. 디바운스가 이를 완화하지만, "사용자가 절대 인지하지 못한다"는 수준의 보장은 아니다.
- **폭주 전환 임계치 설정이 어렵다.** "초당 50개"라는 임계치는 어떤 근거인가? 너무 높으면 CPU 폭주를 막지 못하고, 너무 낮으면 거의 항상 eventual mode에서 동작하게 된다. 이 임계치는 PC 성능에 따라 달라져야 하며, 결국 시스템 상태를 고려해야 한다는 입장 B의 논거로 수렴한다.
- **기존 솔루션의 교훈을 충분히 반영하지 못한다.** 15년간 50,000대 PC에서 "업무 방해"가 핵심 불만이었다는 것은 단순한 고객 피드백이 아니라, 이 제품이 존재하는 시장적 이유이다. "기존과 다르다"는 기술적 설명은 타당하지만, 사용자의 심리적 신뢰("이 에이전트는 내 PC를 느리게 만들지 않는다")를 구축하려면 "즉시 스캔하되 가볍다"보다 "시스템이 한가할 때만 스캔한다"가 더 강력한 메시지이다.
- **"시스템 상태 무관하게 스캔"이라는 점이 구조적 약점이다.** 디바운스와 폭주 전환은 이벤트율만 고려하고, CPU 부하나 사용자 활동 상태를 고려하지 않는다. 사용자가 화상회의 중이라 CPU가 70%를 차지하고 있을 때 파일 하나가 변경되면, 이벤트율은 낮지만 추가 CPU 사용은 화상회의 품질에 영향을 줄 수 있다.

### 4.2 입장 B의 강점과 약점

**강점:**

- **제품 차별화 전략과 완벽히 정합한다.** "Quiet Agent"가 제품의 핵심 차별화 축이라면, 아키텍처 전체가 이 목표에 종속되어야 한다. 리소스 거버너는 "시스템 상태를 고려한 스캔 타이밍 결정"이라는 원칙을 구조적으로 보장하며, 이는 단순한 디바운스로는 달성할 수 없는 수준이다.
- **PoC 수치가 주장을 뒷받침한다.** Codex의 PoC가 즉시 스캔(CPU 72.6%)과 동적 거버닝(CPU 32.3%, P95 지연 17.5초)의 트레이드오프를 정량적으로 제시했다. 입장 A는 "디바운스를 적용하면 달라진다"고 반박하지만, 디바운스 적용 후의 수치는 PoC로 검증되지 않았다.
- **우선순위 큐가 보안과 UX를 동시에 해결한다.** Critical 우선순위(USB 반출 등)를 즉시 처리하고 나머지는 거버너에 위임하는 설계는, 즉시 스캔의 보안 이점을 선별적으로 수용하면서도 전체적인 "Quiet Agent" 원칙을 유지한다.
- **확장성이 뛰어나다.** 리소스 거버너는 향후 배터리 상태, 네트워크 상태, 관리 정책(업무 시간대 설정) 등의 추가 신호를 쉽게 수용할 수 있는 확장 가능한 구조이다.

**약점:**

- **구현 복잡도가 높다.** 리소스 거버너의 다중 신호 수집, 상태 머신, 동적 예산 조정은 디바운스 + 임계치 전환보다 구현과 테스트가 복잡하다. MVP에서 거버너의 정교한 신호 조합까지 구현하면 개발 일정이 늘어날 수 있다.
- **P95 지연 17.5초는 "실시간"이라 부르기 어렵다.** 고객이 "실시간 감지"를 기대하고 구매했을 때, "보통은 17초 후에 감지됩니다"는 설명이 필요할 수 있다. 마케팅 메시지와 기술적 현실 사이의 괴리가 발생할 수 있다.
- **거버너의 매개변수 튜닝이 필요하다.** idle 임계치(5초), CPU idle 임계치(70%), 예산 레벨 등은 환경에 따라 다르게 설정해야 한다. 기본값이 모든 고객사에 적합하지 않을 수 있으며, 이는 추가적인 설정/관리 복잡도를 발생시킨다.
- **큐 크기 관리가 필요하다.** 사용자가 장시간 집중 작업 중이면(예: 연속 3시간 문서 작업, idle 없음) 큐가 계속 쌓인다. Gemini가 제안한 "큐 > 1,000건 시 배치 모드 전환"은 합리적이나, 이 역시 추가 로직이다.

### 4.4 조건부 결론

이 논쟁의 핵심은 **"보안 속도"와 "사용자 무방해" 중 어느 쪽에 가중치를 더 두는가**이다.

**결정적 맥락: 기존 솔루션 15년 운영의 교훈**

이 제품이 탄생하게 된 직접적 원인이 "기존 솔루션의 업무 방해"라는 점을 무시할 수 없다. 기존 솔루션 사용자 50,000명이 15년간 반복적으로 제기한 1위 불만이 "CPU 급증/업무 방해"였다면, 신규 제품의 아키텍처는 이 문제를 **구조적으로 불가능하게 만드는** 방향으로 설계되어야 한다. "이번에는 가볍게 만들었으니 괜찮습니다"보다 "시스템이 한가할 때만 동작하도록 설계했으므로 원천적으로 업무 방해가 불가능합니다"가 고객에게 더 강력한 메시지이다.

**수치 기반 판단:**

- Codex PoC 기준, 즉시 스캔의 CPU 점유율(72.6%)은 디바운스를 적용하더라도 버스트 상황에서 상당한 수준일 것이다. 입장 A는 "디바운스 후에는 다르다"고 주장하지만, 이를 PoC로 검증하지 않았다.
- 동적 거버닝의 P95 지연 17.5초는 기존 일일/주간 풀스캔 대비 수천 배 빠르며, 대부분의 컴플라이언스 요건을 충족한다.
- Critical 우선순위 즉시 처리를 도입하면, 보안적으로 중요한 시나리오(USB 반출 등)에서의 PII 노출 창은 입장 A와 동일하게 최소화된다.

**구현 현실성:**

- 리소스 거버너의 MVP 버전은 복잡할 필요가 없다. `GetLastInputInfo`(idle 시간) + `GetSystemTimes`(CPU idle 비율) 두 신호만으로도 핵심 기능은 구현 가능하다. 포그라운드 앱 분석, 배터리 상태 등은 후속 버전에서 추가할 수 있다.
- 우선순위 큐는 Rust의 `BinaryHeap` 또는 `crossbeam-channel`로 구현 가능하며, MVP에서는 2단계(Critical/Normal)로 단순화해도 충분하다.

## 5. 최종 추천안

**추천: 입장 B (큐잉 + 리소스 거버너) 채택, 입장 A의 디바운스 설계와 폭주 안전장치를 통합**

### 핵심 설계 원칙

1. **워처는 "감지"만, 거버너가 "실행"을 결정한다.** `ReadDirectoryChangesW` 워처는 이벤트를 디바운스 처리하여 큐에 적재하는 역할만 수행한다. 스캔 실행 여부와 타이밍은 리소스 거버너가 결정한다.

2. **Critical 경로는 즉시 처리한다.** USB 드라이브 파일 이동 등 보안적으로 중요한 이벤트는 거버너를 우회하여 즉시 스캔한다. 이는 입장 A의 보안적 동기를 선별적으로 수용한 것이다.

3. **거버너의 MVP 버전은 단순하게 시작한다.** Phase 1에서는 `GetLastInputInfo` + `GetSystemTimes` 두 신호만 사용하고, Phase 2에서 포그라운드 앱 분석, 배터리 상태 등을 추가한다.

### MVP 구현 사양

```
[워처 레이어]
├── ReadDirectoryChangesW (3폴더: 바탕화면/문서/다운로드)
├── 디바운스: 경로별 500ms 타이머 (입장 A의 디바운스 설계 수용)
├── 폭주 안전장치: 큐 크기 > 1,000 → 워처 일시 중단, USN 증분으로 전환
└── 출력: 디바운스된 이벤트를 우선순위 큐에 push

[우선순위 큐]
├── Critical: 즉시 처리 (거버너 우회)
│   ├── USB 드라이브 관련 파일 이동
│   └── 관리 정책에서 "즉시 감지"로 지정한 경로
└── Normal: 거버너 제어하에 처리
    └── 그 외 모든 파일 변경 이벤트

[리소스 거버너 (MVP)]
├── 폴링 주기: 200ms
├── 입력 신호:
│   ├── GetLastInputInfo() → idle_seconds
│   └── GetSystemTimes() → cpu_idle_pct
├── 예산 결정:
│   ├── idle_seconds >= 5 AND cpu_idle_pct > 60% → 배치 10, CPU 한도 50%
│   ├── idle_seconds >= 2 AND cpu_idle_pct > 40% → 배치 3, CPU 한도 20%
│   ├── cpu_idle_pct > 70% → 배치 1, CPU 한도 10%
│   └── 그 외 → 대기
└── 스캔 워커에 예산 전달

[스캔 워커]
├── 거버너 예산에 따라 큐에서 pop
├── 1차 스캔: 정규식 + 체크섬 (5-20ms/파일)
├── 의심 파일 → 2차 큐 (NER 추론, idle 시 처리)
└── 각 스캔 소요 시간을 거버너에 피드백

[나머지 폴더]
└── USN 저널 기반 증분 스캔 (기존 합의대로)
```

### Phase 0 검증 항목

Phase 0(2-4주)에서 반드시 실측해야 하는 항목:

| 측정 항목 | 목적 | 방법 |
|:----------|:-----|:-----|

| 디바운스 후 실제 스캔 대상 수 | 큐 크기 예측 | 위 데이터에서 디바운스 시뮬레이션 |

### 이 추천이 양측 입장을 어떻게 수용하는가

- **입장 A에서 수용한 것:**
  - 디바운스 전략 (경로별 500ms 타이머)
  - `ReadDirectoryChangesW` 버퍼 오버플로우 대비 USN 폴백
  - PII 노출 창 최소화 동기 (Critical 우선순위 즉시 처리)
  - 폭주 시 모드 전환 개념 (큐 크기 기반 USN 전환)

- **입장 B에서 수용한 것:**
  - 워처의 역할을 큐잉으로 제한
  - 리소스 거버너에 의한 스캔 타이밍 결정
  - 시스템 상태(CPU, 사용자 활동) 기반 동적 예산
  - 배치 처리 효율 극대화
  - "Quiet Agent" 원칙의 구조적 보장

- **절충 지점:**
  - 거버너의 MVP 버전을 2개 신호(`GetLastInputInfo`, `GetSystemTimes`)로 단순화하여 구현 복잡도를 입장 A 수준에 근접시킴
  - Critical 우선순위 도입으로 "보안적으로 중요한 이벤트는 즉시 처리"라는 입장 A의 핵심 가치를 선별적으로 보존
  - Phase 0에서 "디바운스 적용 후 즉시 스캔의 실제 CPU 부하"를 측정하여, 입장 A의 미검증 주장에 데이터 기반 판단 근거를 마련
