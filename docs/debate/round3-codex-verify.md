# Round 3 Verification — Codex

**Date:** 2026-02-10  
**Reviewer:** Codex  
**Target:** `docs/debate/consensus.md`, `docs/debate/unresolved.md`

Claude가 정리한 합의/미합의 문서를 Codex의 Round 1-2 원문(`docs/debate/round1-codex.md`, `docs/debate/round2-codex.md`)과 대조해, **Codex 의견 반영의 정확성**과 **구현 가능성** 관점에서 검증했다.

---

## 1. 합의 결과 중 Codex의 의견이 잘못 반영된 것은?

1. **논점 3(기술 스택)에서 “클라이언트 = Rust 단일 언어 확정”은 Codex 입장을 과도하게 단정함.**  
   Codex의 실제 입장은 “기본안은 전체 Rust”에 동의하되, 팀 역량이 극단적으로 .NET 편향인 경우에 한해 **FFI가 아닌 프로세스 경계(IPC)로만** “C# 서비스 + Rust 워커” 예외안을 열어두는 것이었다. (`docs/debate/round2-codex.md`)  
   따라서 합의문 표현은 “기본안 = Rust 단일”로는 가능하지만, “단일 언어 확정(예외 없음)”으로 읽히는 문구는 Codex 의견을 축소/왜곡한다.

2. **논점 5(보안)에서 “클라이언트 mTLS는 Phase 2”를 전원 동의처럼 고정한 것은 Codex 의견과 미세하게 어긋남.**  
   Codex는 MVP에서 mTLS를 “강제”하지 말자는 점에는 강하게 동의했지만, 설계는 최소한 **Token-only 모드와 mTLS+Token 모드를 모두 수용**하도록 잡아 “고객 환경에 따라” 단계적으로 올릴 수 있게 하자는 입장이었다. (`docs/debate/round2-codex.md`)  
   합의문대로 “MVP에서는 클라이언트 mTLS를 아예 안 한다”로 범위를 고정하려면, 그 결정을 “합의”가 아니라 “스코프 선택”으로 명시하는 편이 공정하다.

3. **“전원 동의/확정”으로 표기된 일부 수치(예: Panic Mode 트리거 CPU 15%, I/O 50MB/s, 워커 메모리 200MB, crate 10-12 등)는 Codex가 합의한 ‘의견’이라기보다 Claude가 제시한 초기 값에 가깝다.**  
   Codex는 “Panic Mode/Quarantine/가드레일이 필수”에는 동의했으나, 트리거 수치 자체를 확정한 적은 없다. (`docs/debate/round1-codex.md`, `docs/debate/round2-codex.md`)  
   이 수치들은 Phase 0 텔레메트리로 튜닝되어야 하므로, “확정”이 아니라 “초기값(조정 전제)”로 표기하는 것이 Codex 관점에서 더 정확하다.

4. **논점 4(MVP 범위)에서 “실시간 워처” 표현은 Codex의 ‘큐잉까지만’ 원칙을 충분히 담지 못해 오해 소지가 있음.**  
   Codex는 핵심 폴더 워처 자체엔 동의하지만, 워처 이벤트 수신 후 즉시 스캔이 아니라 **큐 적재 후 리소스 거버너가 실행 시점을 결정**해야 “Quiet Agent” 목표와 충돌하지 않는다고 명시했다. (`docs/debate/round2-codex.md`)  
   합의문에서 “실시간 워처”가 “즉시 스캔”으로 해석될 수 있어, Codex 의도가 약화되어 있다.

## 2. 구현 관점에서 합의된 결정 중 실행 불가능한 것은?

“물리적으로 불가능”한 합의는 거의 없지만, **현재 합의문 표현 그대로는 구현 스펙으로 고정하기 어려운 항목**이 있다.

1. **DRM “감지(헤더/매직넘버)”를 일반해처럼 전제하는 부분은 과도함.** (`docs/debate/consensus.md`)  
   벤더 SDK/필터 드라이버 없이도 “일부 DRM 포맷”은 시그니처로 식별 가능하지만, **모든 DRM 파일을 확실히 판별**하는 것은 불가능하다.  
   구현 스펙은 “알려진 벤더/시그니처에 한해 `SUSPECTED_DRM` 분류”처럼 범위를 제한하거나, “탐지 실패도 가능”을 계약/지원 매트릭스에 명시해야 한다.

2. **USN Journal을 증분 스캔의 백본으로 삼는 결정은 타당하지만, ‘USN 불가/비정상’ 시나리오가 스펙에 없으면 운영상 실행 불가에 가까워진다.** (`docs/debate/consensus.md`)  
   구현에는 최소한 “USN 비활성/권한 부족/저널 롤오버/볼륨 비-NTFS”에서의 폴백(예: 주기적 느린 스캔, 또는 해당 볼륨 Limited/Unsupported 처리) 규칙이 필요하다.

3. **Panic Mode/가드레일의 구체 수치(CPU%, I/O MB/s, 타임아웃, 메모리 상한 등)를 ‘확정값’으로 쓰는 것은 실행 불가(또는 즉시 오동작) 가능성이 높다.** (`docs/debate/consensus.md`)  
   엔드포인트 다양성(코어 수, 디스크 종류, EDR 훅, VDI 등) 때문에 “고정 수치”는 환경에 따라 너무 민감하거나 너무 둔해진다. 합의문 수치는 “초기값”으로 두고, Phase 0에서 per-device 기준(코어 수 정규화 등)으로 보정하는 전제가 필요하다.

## 3. 미합의 항목 중 프로토타입으로 빨리 검증할 수 있는 것은?

1. **미합의 1(Aho-Corasick 스킵 범위):** 키워드 프리필터가 Regex 단계까지 스킵해도 되는지  
   프로토타입: 실제 코퍼스(또는 익명화 샘플)에서 “키워드 0개 파일”을 분리하고, 그 집합에서 구조화 PII(주민/카드/이메일/전화) Regex 매치 비율을 측정한다.  
   결론 기준: 키워드 0개 집합에서 구조화 PII 매치가 의미 있게 나오면 “Regex는 항상 실행”이 안전측이다.

2. **미합의 2(NER 로드/언로드):** 세션 단위 로드/언로드가 thrashing 없이 가능한지  
   프로토타입: i3/4GB급에서 ONNX Runtime 초기 로드 시간(콜드/웜), RSS 증가량, 프로세스 스폰 비용을 측정한다. 이벤트가 잦은 시나리오(분당 N회)에서 로드/언로드 반복 시 지연 스파이크 여부를 본다.

3. **미합의 8(실시간 스캔 타이밍):** 즉시 스캔 vs 큐잉+리소스 거버너  
   프로토타입: ReadDirectoryChangesW 이벤트 폭주(빌드 출력, unzip, git checkout 등)를 재현하고, “즉시 스캔”과 “큐잉 후 idle 실행” 두 모드의 CPU/IO/지연(P95)을 비교한다.  
   성공 기준: 사용자 입력/포그라운드 앱 활동 중 CPU 스파이크가 보이면 즉시 스캔은 MVP 정책으로 부적합.

4. **미합의 7(규칙 형식):** MVP부터 열거형 기반 JSON/YAML 스키마로 갈지, 코드 정의로 갈지  
   프로토타입: “표현식 금지, 열거형만” 스키마를 최소 5개 규칙(주민/카드/전화/이메일/사업자)로 작성하고, 서명된 번들 업데이트(검증/적용/롤백)까지 end-to-end로 붙여본다.  
   판단 포인트: 운영(핫픽스) 속도와 안전성(검증/롤백)이 실제로 개선되는지.

5. **미합의 3(TPM 우선 바인딩 vs 퍼지 핑거프린팅):** TPM 2.0 가용성  
   프로토타입: 파일럿 100-500대에서 TPM 존재/활성화/소유권 상태를 수집하는 진단 커맨드(에이전트/스크립트)를 배포해 분포를 확인한다.  
   결론 기준: TPM 활성화 비율이 높으면 TPM 키 기반 바인딩이 1차가 되고, 낮으면 퍼지 + 재바인딩 UX가 사실상 필수다.
