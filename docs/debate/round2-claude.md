## 논점 1: NER 모델 선택

### 받은 비평 요약
- **Gemini:** 대체로 동의. 모델 파일 암호화(디스크 저장 시 암호화, 메모리 로드 시 복호화)를 필수로 추가할 것을 권고.
- **Codex:**
  - Claude가 제시한 F1(Naver NER/KLUE-NER)은 **일반 NER 벤치마크**이며, PII 특화 성능이 아니다. Hard negative(이상/박수/김밥 등 사람 이름으로 오인 가능한 일반 단어)에 대한 FP 비용 모델이 다르다.
  - 토크나이저가 병목이다. 모델 추론 10ms라 해도 end-to-end는 2-5배 튄다.
  - INT8 양자화는 엔티티 타입별(PER, ADDR)로 민감하게 흔들리며, threshold 재캘리브레이션 없이는 FP가 늘어난다.
  - CPU ISA(AVX2/VNNI) 차이로 기기별 지연 편차가 크다. "AI 예산(분당 최대 N회)"이 필요하다.
  - Lazy load/unload(5분 미사용 시 언로드)는 실시간 이벤트가 잦은 환경에서 thrashing을 유발한다.

### Claude의 응답

**[부분 수용]**

1. **벤치마크 F1 한계 — 수용.** Codex의 지적이 정확하다. Round 1에서 내가 "F1 80-85% 하락 가능"이라고 경고한 것 자체가 이 한계를 인식한 것이었으나, 보다 명확히 해야 한다. **수정:** PII 도메인 전용 평가셋(합성 급여명세서, 계약서, 엑셀 고객 리스트)을 Phase 0에서 구축하고, 공개 벤치마크 F1은 참고 지표로만 사용한다. "F1 87%"를 설계 근거로 인용하지 않는다.

2. **토크나이저 병목 — 수용.** 이 부분은 내 설계에서 누락된 현실적 제약이다. HuggingFace Tokenizers의 Rust 바인딩은 빠르지만, 문자열 정규화(Unicode NFKC) → WordPiece 분절 → ID 변환까지 포함하면 모델 추론과 비슷하거나 더 걸릴 수 있다. **수정:** Phase 0 벤치마크에 "토크나이저 포함 end-to-end 지연"을 측정 항목으로 추가한다. 토크나이저만 별도 프로파일링하여 병목 시 사전 토크나이징(배치 처리)을 검토한다.

3. **INT8 엔티티별 민감도 — 수용.** INT8 양자화 후 PER/ADDR 타입의 precision/recall이 FP32 대비 얼마나 변하는지를 타입별로 측정해야 한다. **수정:** 양자화 후 엔티티 타입별 threshold를 재캘리브레이션하는 과정을 Phase 0에 포함한다.

4. **AI 예산 — 수용.** Codex/Gemini Review 모두 "분당 최대 N회" 하드캡을 제안했고, 이는 저사양 PC에서 NER이 CPU를 독점하지 않도록 하는 현실적 안전장치다. **수정:** NER 호출을 "분당 AI 예산" + "파일당 최대 K개 구간"으로 이중 제한한다. 예산 소진 시 해당 기간의 나머지 후보는 Regex 결과만으로 판정하고, 다음 주기에 NER 재시도 큐에 넣는다.

5. **Lazy load/unload thrashing — 부분 수용.** Codex의 우려는 타당하나, 해결책이 "항상 상주"는 아니다. 14MB INT8 모델을 항상 메모리에 올리면 4GB PC에서 상시 3.5%를 점유한다. **수정안:** "5분 미사용 시 언로드" → **"스캔 세션 단위 로드/언로드"**로 변경. 스케줄 스캔 시작 시 로드, 스캔 완료 후 언로드. 실시간 모니터링(Phase 2)에서는 debounce window(30초) 내 이벤트 누적 후 한 번 로드하여 배치 처리하고, idle 5분 후 언로드. 이렇게 하면 잦은 로드/언로드 반복을 피하면서도 비스캔 시간에 메모리를 회수한다.

6. **모델 파일 암호화(Gemini) — 부분 수용.** 디스크 암호화는 합리적이나, 암호화/복호화 자체가 장애 원인이 되지 않도록 해야 한다. **수정:** AES-256-GCM으로 모델 파일 암호화, 키는 DPAPI(Windows Data Protection API)로 보호. 단, Codex가 경고한 것처럼 "보안 기능이 장애 원인"이 되는 시나리오에 대비하여, 복호화 실패 시 NER을 비활성화하고 Regex-only 모드로 폴백하는 안전장치를 둔다. mlock/zeroize/honey-pot 같은 메모리 보호는 MVP에서 제외한다.

## 논점 2: 탐지 파이프라인 구조

### 받은 비평 요약
- **Gemini:** 파이프라인 구조에 동의. Stage 2에 Aho-Corasick 키워드 프리필터를 추가하여, 키워드가 없는 파일은 Regex도 건너뛰는 초고속 스킵 제안.
- **Codex:**
  - Claude의 포맷 커버리지(EML/MSG/HWP/PDF 등)가 넓어 **스코프 폭발** 위험이 크다.
  - 스코어링 가중치(x1.2, x0.5 등)는 고정값으로는 빠르게 무너진다. 평가셋 + 회귀 테스트 + 카나리/롤백이 필요하다.
  - 화이트리스트/블랙리스트는 "기능"이 아니라 **정책 배포 시스템**(서명 + 승인 + 롤백)이다. Hot reload만으로는 사고가 난다.
  - ZIP 지원은 zip bomb 대응(재귀 깊이, 해제 용량, 파일 수, 타임아웃, 경로 트래버설) 없이는 치명적이다.
  - "실시간"에서 USN catch-up이 보완이 아니라 **본체**다.

### Claude의 응답

**[부분 수용]**

1. **Aho-Corasick 프리필터(Gemini) — 수용.** 좋은 최적화 제안이다. 키워드("주민", "전화", "카드", 숫자 패턴 등)가 전혀 없는 파일은 Regex 단계조차 불필요하다. **수정:** 파이프라인에 Stage 1.5로 Aho-Corasick 키워드 존재 여부 체크를 추가한다. 이는 `aho-corasick` 크레이트로 수 마이크로초 내 판별 가능하며, 코드/로그 파일 등 PII가 없는 파일을 대량으로 걸러낸다.

2. **포맷 커버리지 스코프 폭발 — 수용.** Round 1에서 이미 "Phase 0에서 포맷별 추출 성공률 정량 측정"을 제안했으나, 설계 문서 자체에는 EML/MSG/HWP/PDF를 모두 나열해 놓아 혼란을 준 것은 사실이다. **수정:** 설계 문서의 포맷 목록을 3단계로 재분류한다.
   - **MVP (Tier 1):** txt, csv, log, docx, xlsx, pptx (OOXML)
   - **Phase 2 (Tier 2):** PDF(텍스트 레이어만), HWP(실험적)
   - **Phase 3 (Tier 3):** EML, MSG, 스캔 PDF(OCR), DRM 내부 스캔
   - Codex의 "txt/csv/log + OOXML만" MVP 범위에 동의한다.

3. **스코어링 가중치 — 수용.** "x1.2", "x0.5" 같은 고정 매직넘버가 실운영에서 무너진다는 지적은 완전히 맞다. **수정:** MVP에서는 가중치 기반 연속 스코어를 제거하고, **이산 판정(CONFIRMED / SUSPECT / DISMISSED)** 3단계로 단순화한다. CONFIRMED = Checksum 검증 통과, SUSPECT = Regex 매칭 + 검증 불가(이름/주소 등), DISMISSED = 컨텍스트 룰로 제외. 세밀한 스코어링은 평가셋이 축적된 Phase 2에서 도입한다.

4. **정책 배포 시스템 — 수용.** Codex의 비판이 정확하다. 화이트리스트/블랙리스트의 hot reload는 편의 기능이지, 안전장치가 아니다. **수정:** 정책/룰 업데이트 흐름을 다음과 같이 재설계한다.
   - 서버에서 서명된 룰 번들(JSON + Ed25519 서명) 발행
   - 에이전트에서 서명 검증 후 적용
   - 적용 후 이상 탐지(탐지 수 급증/급감) 시 자동 롤백
   - MVP에서는 카나리(일부 에이전트 먼저 적용) + 수동 롤백. 자동 롤백은 Phase 2.

5. **ZIP bomb — 수용.** 이 부분은 내 설계에서 완전히 누락되었다. **수정:** 아카이브 처리 정책을 명시한다.
   - 재귀 깊이 최대 3단계
   - 총 해제 크기 최대 500MB
   - 파일 수 최대 1,000개
   - 단일 파일 추출 타임아웃 10초
   - 경로 트래버설(../../) 검출 시 즉시 중단
   - 이 제한을 초과하면 "ARCHIVE_LIMIT_EXCEEDED"로 분류하고 스캔 중단.

6. **USN이 본체 — 수용.** Round 1에서 "MVP는 스케줄 기반 증분 스캔"을 제안한 것과 일맥상통하지만, Codex의 표현이 더 정확하다. **수정:** USN Journal 기반 변경 파일 목록 추출을 증분 스캔의 **기본 메커니즘**으로 명시한다. ReadDirectoryChangesW는 "핵심 폴더 한정의 보조 실시간 알림"으로 격하하고, Phase 2에서만 도입한다. 용어를 "USN 기반 증분 + 선택적 실시간 워처"로 정리한다.

## 논점 3: 기술 스택 (Rust vs 대안)

### 받은 비평 요약
- **Gemini:** Client Rust, Server Go에 전원 동의. 이견 없음.
- **Codex:**
  - 서버 아키텍처에서 Claude가 제안한 Kafka/Elasticsearch는 MVP에 과하다.
  - Rust에서 Windows API/COM/FFI(DRM SDK, IFilter 등)는 난이도가 급상승하며, 작은 팀에서 병목이 된다.

### Claude의 응답

**[수용]**

1. **Kafka/Elasticsearch MVP 제외 — 수용.** 이 비판은 완전히 맞다. 내 설계 문서(`system-architecture.md`)에서 NATS JetStream, TimescaleDB, Redis 등을 열거한 것은 50,000대 최종 형태를 설계한 것이었으나, MVP에서 이 스택을 올리면 운영 복잡도가 개발을 죽인다. **수정:** MVP 서버 스택을 다음으로 단순화한다.
   - **Go + Fiber(HTTP/gRPC)**
   - **PostgreSQL 단일 인스턴스** (결과 저장, 정책, 디바이스 등록 모두)
   - **Redis** (세션, 캐시 — 선택적, 없어도 동작)
   - 메시지 큐 없음. 에이전트→서버는 배치 HTTP 업로드.
   - Phase 2에서 NATS 도입 (에이전트 수 1,000대 초과 시).
   - Phase 3에서 Elasticsearch 도입 (로그 검색/분석 필요 시).

2. **Windows COM/FFI 난이도 — 수용.** DRM SDK 연동, IFilter 호출, COM 기반 문서 처리는 Rust에서 unsafe 코드와 ABI 관리가 불가피하다. 이것이 내가 Round 1에서 "DRM은 Phase 2"라고 제안한 핵심 이유이기도 하다. **수정 사항 없음** — 기존 입장(DRM/IFilter는 Phase 2 이후, 별도 워커 프로세스에서 FFI 격리)을 유지한다. 단, Codex가 지적한 대로 이 영역은 팀에 Windows 시스템 프로그래밍 경험자가 반드시 필요하며, Rust 경험만으로는 부족하다는 점을 리스크로 명시한다.

## 논점 4: MVP 범위

### 받은 비평 요약
- **Gemini:**
  - NER은 MVP에 포함하되 "Labs(베타)" 기본 비활성화(Opt-in).
  - 실시간은 핵심 경로(바탕화면, 문서)만. 나머지는 스케줄 증분.
  - HWP는 "실험적 기능"으로 격하. `hwp-rs` v0.2.0은 프로덕션 부적합.
  - 에이전트 자동 업데이트 제외. 기존 PMS/AD 활용.
- **Codex:**
  - MVP In: 정형 PII + checksum + 컨텍스트 룰 + USN 증분 + 핵심 폴더 워처 + 워커 격리 + Panic mode + 최소 서버 콘솔.
  - MVP Out(표시만): DRM "UNSCANNABLE" 분류, NER 베타(하드캡).
  - MVP Out: HWP/PDF/MSG, 자동 업데이트, Kafka/ES, 멀티테넌시.

### Claude의 응답

**[부분 수용]**

1. **NER의 MVP 포함 여부 — Codex에 동의, Gemini에 부분 반대.** Gemini는 "Labs 비활성화로 포함"을 제안했으나, "포함"이라는 사실 자체가 QA 범위를 늘린다. 비활성화 상태여도 설치/업데이트/설정 UI/에러 핸들링이 필요하다. **유지:** NER은 MVP 바이너리에 포함하되, 서버 정책으로 활성화/비활성화를 제어한다. MVP 기본값은 비활성화. 파일럿 고객 중 옵트인한 그룹에서만 베타 테스트. 이는 Gemini의 "Labs" 컨셉과 Codex의 "베타 하드캡"을 절충한 것이다.

2. **실시간 범위 — Gemini에 부분 동의로 수정.** Round 1에서 "MVP에서 실시간 완전 제외"라고 했으나, Gemini의 "핵심 경로만 실시간"은 합리적 절충이다. **수정:** MVP에서 `바탕화면`, `문서(Documents)`, `다운로드(Downloads)` 3개 폴더만 ReadDirectoryChangesW로 모니터링한다. 이벤트 폭주 시 자동으로 증분 모드로 전환(Codex의 "eventual mode 자동 전환"과 동일). 나머지 볼륨은 USN 기반 증분 스캔만.

3. **HWP 실험적 격하 — 완전 동의.** Gemini/Codex 모두 `hwp-rs` 미성숙을 지적했고, 내 Round 1에서도 동일한 우려를 표명했다. HWP를 "지원"이라고 하면 고객은 100% 동작을 기대한다. **유지:** HWP는 MVP Tier 2(Phase 2)로 분류. Phase 0에서 파싱 성공률 측정 후 판단.

4. **자동 업데이트 제외 — 수용.** Gemini의 제안에 동의한다. 자동 업데이트는 서명 키 관리, 롤아웃/롤백, 실패 복구 런북까지 포함하면 하나의 서브시스템이다. **수정:** MVP는 MSI 패키지 + 기존 배포 시스템(AD/SCCM/PMS)으로 배포. 에이전트에 버전 체크 API만 구현하여 "업데이트 가능" 알림을 서버 콘솔에 표시.

## 논점 5: 보안/라이선스 설계

### 받은 비평 요약
- **Gemini:** mTLS 유지하되 Join Token + 인증서 발급 하이브리드. 인증서 갱신 실패 시 Token Fallback 예비. Claude의 "원문 미전송" 원칙 철저히 동의.
- **Codex:**
  - **Certificate Pinning은 엔터프라이즈에서 연결 불가를 만든다.** TLS inspection(프록시) 환경에서 pinning은 정상 트래픽도 차단. 기본값은 pinning OFF가 현실적.
  - **mTLS 50k 운영은 PKI "프로젝트"다.** 발급/갱신/회수 자동화, 오프라인 노트북 복구, 만료 브릭 대응 플로우가 없으면 운영에서 터진다.
  - **타임스탬프 anti-replay(±30초)는 NTP 미정렬 PC에서 실패한다.** request_id/idempotency + device_seq가 더 안전.
  - **Fuzzy HW Fingerprinting은 WMI 현실 때문에 오탐이 많다.** 메인보드/디스크 시리얼이 비거나 중복되는 PC가 많다(가상화, 벤더 구현 차이, 권한 부족). 퍼지 매칭은 "정당한 교체 차단"과 "복제 허용" 모두 만든다. 콘솔 재바인딩 UX 필수.
  - **모델 암호화/anti-debug는 지원 비용을 올린다.** "보안 기능이 장애 원인"이 되는 것이 가장 위험. 옵션화 필요.
  - **"원문 미전송"이라도 파일명 자체가 민감할 수 있다.** Path privacy mode 또는 로컬 증거 뷰어 UX 필요.

### Claude의 응답

**[부분 수용]**

1. **Certificate Pinning 기본 OFF — 수용.** Codex의 지적이 결정적이다. 한국 대기업의 **80% 이상이 SSL/TLS inspection 프록시**(Palo Alto, Fortigate 등)를 사용한다. 여기서 pinning을 기본 활성화하면 "제품이 회사 네트워크에서 안 됨"이라는 도입 차단 사유가 된다. 내 설계에서 Certificate Pinning을 "기본 보안"으로 제안한 것은 엔터프라이즈 네트워크 현실을 과소평가한 것이다. **수정:** Pinning은 기본 OFF, 고객 옵션(폐쇄망/직접 연결 환경)으로 전환. 대신 TLS 1.3 + 서버 인증서 검증(시스템 CA 스토어 기반)을 기본으로 한다.

2. **mTLS 단계적 도입 — 기존 입장 유지.** Round 1에서 이미 "MVP: Join Token + JWT, mTLS는 서버 간만, 클라이언트 mTLS는 Phase 2"라고 제안했고, 이것은 Gemini/Codex 모두의 우려와 부합한다. Codex의 "Bootstrap Token → CSR → 짧은 수명 cert" 프로세스는 Phase 2 도입 시 참조하겠지만, MVP에서는 과잉이다. **수정 사항 없음.**

3. **Anti-replay 메커니즘 — 수용.** 타임스탬프 ±30초 윈도우는 NTP가 정확한 환경을 전제한 것이었고, 실제 기업 PC(특히 노트북의 절전/하이버네이트 후 시계 드리프트)에서는 불안정하다. **수정:** Anti-replay를 `request_id(UUID v7) + device_seq(단조 증가 시퀀스)` 기반으로 변경한다. 서버는 device별 마지막 seq를 추적하고, seq가 역행하면 거부. 타임스탬프는 로깅/감사 목적으로만 사용하고 검증에는 쓰지 않는다.

4. **Fuzzy HW Fingerprinting — 부분 수용.** Codex의 WMI 현실 지적은 유효하다. 실제로:
   - 가상 머신(VDI)에서 CPU/BIOS/디스크 시리얼이 동일한 복제본이 생긴다.
   - 일부 벤더(특히 소규모 OEM)는 시리얼을 "To Be Filled"로 출하한다.
   - 권한 부족(비관리자 계정)으로 WMI 쿼리가 실패할 수 있다.

   그러나 **퍼지 핑거프린팅 자체를 폐기하는 것은 반대한다.** 15년 운영에서 Machine ID 단일 바인딩의 고객 불만이 너무 빈번했기 때문이다. **수정:**
   - 핑거프린트는 "라이선스 바인딩"이 아니라 **"이상 탐지 신호"**로 격하한다.
   - 핑거프린트 유사도가 임계치 이하로 떨어지면 즉시 차단이 아니라 **서버에 "재검증 필요" 플래그**를 올리고, 관리자가 콘솔에서 승인/거부한다.
   - WMI 쿼리 실패 항목은 가중치 0으로 처리(비교에서 제외).
   - Codex가 제안한 **콘솔 재바인딩 UX**는 필수 — 이것은 MVP에 포함한다.

5. **모델 암호화 부작용 — 부분 수용.** 논점 1에서 이미 응답했듯이, 복호화 실패 시 Regex-only 폴백을 두어 "보안 기능이 장애 원인"이 되지 않도록 한다. mlock/zeroize/honey-pot은 MVP에서 제외하며, 이는 Codex의 "난독화 절제" 노선에 동의한다.

6. **파일명 민감성 — 수용.** 이 부분은 내 설계에서 간과한 현실적 시나리오다. 파일명에 "홍길동_급여명세서_2026.xlsx" 같은 PII가 포함되는 것은 한국 기업에서 흔하다. **수정:**
   - 데이터 전송 레벨에 **Path Privacy Mode**를 추가한다.
   - Level 1 (기본): 파일명 전송, 경로는 해시.
   - Level 2: 파일명도 해시. 서버에는 확장자 + 크기 + 탐지 결과만 전송.
   - Level 3: 모든 메타데이터 해시. 관리자는 에이전트 로컬에서만 원본 확인 가능.
   - 고객사 보안 정책에 따라 관리자가 레벨을 선택.

## 논점 7: 경쟁 차별화 전략

### 받은 비평 요약
- **Gemini:** Claude의 "Privacy by Design" + "경량 에이전트"에 동의. 여기에 **"Explainable Detection(탐지 근거 명시)"**을 추가. 메시지: "오탐인지 아닌지 근거를 보여줍니다."
- **Codex:**
  - 원문을 안 보내면 관리자 조치가 어려워 **"쓸모 없는 정확도"**가 된다. 파일 단위 우선순위, 조치 상태 추적(open/ack/resolved), 로컬 증거 확인 같은 **workflow가 MVP부터** 필요하다.
  - 룰/모델 업데이트 안전성이 차별화의 핵심이다. "업데이트 후 장애가 없냐"를 고객은 본다.

### Claude의 응답

**[수용]**

1. **Explainable Detection(Gemini) — 수용.** 좋은 제안이다. 경쟁 솔루션 대부분은 "주민등록번호 발견"만 알려주고, **왜 그것이 주민등록번호인지**(어떤 패턴 매칭, 어떤 Checksum 통과, 주변에 어떤 키워드가 있었는지)를 보여주지 않는다. 이것은 관리자의 오탐 판단을 돕고, 신뢰도를 높인다. **수정:** 탐지 결과에 `detection_reason` 필드를 추가한다.
   - 예: `"reason": "regex:rrn_pattern matched, checksum:luhn_mod11 passed, context:keyword '주민' found within 50 chars"`
   - 이 필드는 서버 전송 대상이며, 원문 PII 없이도 판단 근거를 전달할 수 있어 Privacy by Design과 양립 가능하다.

2. **관리자 조치 워크플로우(Codex) — 수용.** "원문 미전송 + 조치 불가"는 Codex가 정확히 짚은 모순이다. 관리자가 탐지 결과만 보고 아무것도 할 수 없으면 솔루션의 가치가 없다. **수정:** MVP 서버 콘솔에 최소 조치 워크플로우를 포함한다.
   - **조치 상태:** OPEN → ACKNOWLEDGED → RESOLVED (3단계)
   - **로컬 증거 확인:** 관리자가 특정 탐지 건을 선택하면, 에이전트에 "증거 스냅샷 요청" 명령을 보낸다. 에이전트는 해당 파일의 탐지 주변 텍스트(앞뒤 20자)를 **마스킹 처리**하여 서버에 전송. 관리자는 마스킹된 컨텍스트로 판단.
   - **일괄 조치:** "이 파일을 암호화/삭제/격리" 같은 원격 명령은 Phase 2. MVP에서는 "사용자에게 알림 전송" + "조치 상태 변경"만.

3. **업데이트 안전성 — 수용.** 논점 2에서 이미 응답한 "서명 + 카나리 + 롤백"과 동일한 맥락이다. 이를 차별화 메시지에 반영한다. **수정:** 차별화 축을 4개로 확정한다.
   - **(1) Quiet Agent:** CPU/IO 가드레일 + Panic Mode + 워커 격리.
   - **(2) Privacy by Default:** 원문 미전송 기본 + Path Privacy Mode + 마스킹 증거.
   - **(3) Explainable Detection:** 탐지 근거 명시 + 관리자 조치 워크플로우.
   - **(4) Safe Updates:** 서명된 룰/모델 번들 + 카나리 배포 + 자동 롤백.

### Claude의 응답

**[수용]**

이 비판은 이번 라운드에서 가장 중요한 관점 전환이다. 내 Round 1 설계는 "무엇을 탐지하는가"에 집중했지, "무엇이 잘못될 때 어떻게 살아남는가"를 체계적으로 다루지 않았다. Codex의 지적대로, 50,000대 엔드포인트 제품에서는 **기능 장애보다 에이전트 장애가 더 치명적**이다. 에이전트가 PC를 느리게 하거나, 크래시하거나, 네트워크를 차단하면 고객은 즉시 제거한다.

**수정: 운영 안전장치를 아키텍처 문서의 독립 섹션으로 격상.**

1. **Panic Mode (안전 모드):**
   - 트리거: 최근 1시간 내 워커 크래시 3회 이상 / CPU 15% 초과 5분 연속 / 디스크 I/O 50MB/s 초과 2분 연속.
   - 동작: 실시간 워처 중지, NER 비활성화, 스케줄 스캔 간격 2배 연장. 서버에 "PANIC_MODE_ENTERED" 이벤트 보고.
   - 복구: 다음 정책 동기화 시 서버가 정상 모드 복귀 명령 전송. 또는 24시간 후 자동 복귀 시도.

2. **Quarantine (격리 큐):**
   - 동일 파일 추출/스캔 실패 2회 → 24시간 스킵 목록 등록.
   - 실패 사유(타임아웃, 크래시, 권한 오류, 파일 잠김) 분류 저장.
   - 서버에 Quarantine 목록 배치 보고. 관리자가 "재시도" 또는 "영구 제외" 선택 가능.

3. **Support Matrix (지원 매트릭스):**
   - **Full Support:** 로컬 NTFS 볼륨.
   - **Limited:** 네트워크 드라이브(SMB) — 스케줄 스캔만, 실시간 미지원. 성능 보장 없음.
   - **Unsupported:** FAT32, OneDrive(클라우드 전용 파일), USB 이동식 미디어 — "연결 시 단발 스캔" 옵션(Phase 2).
   - 이 매트릭스를 **영업 자료와 계약서에도 반영**하여, 고객 기대와 제품 능력을 일치시킨다.

4. **Rollout Discipline (배포 규율):**
   - 룰/모델 업데이트: 서명 필수 + 카나리 그룹(전체의 1-5%)에 먼저 적용 + 24시간 모니터링 + 이상 없으면 전체 롤아웃.
   - 에이전트 바이너리 업데이트: MVP에서는 수동(PMS/AD). Phase 2에서 자동 업데이트 도입 시 동일한 카나리/롤백 원칙 적용.
   - 롤백: 서버에서 "이전 버전으로 복귀" 명령 전송 가능. 에이전트는 이전 룰 번들을 로컬에 1세대 보관.

### Round 2 핵심 교훈

1. **Codex의 가장 큰 기여:** "이론 vs 코드" 관점에서 설계의 빈틈을 지적한 것. 특히 Certificate Pinning의 엔터프라이즈 현실, 타임스탬프 anti-replay의 NTP 문제, HW Fingerprinting의 WMI 현실은 실무 경험에서만 나오는 통찰이다.

2. **Gemini의 가장 큰 기여:** Aho-Corasick 프리필터와 Explainable Detection 제안. 전자는 성능 최적화, 후자는 제품 차별화에 직접 기여한다.

3. **Claude의 인정:** Round 1 설계는 "무엇을 만들 것인가"에 편중되어 있었고, "무엇이 깨질 것인가"와 "어떻게 살아남을 것인가"에 대한 체계적 대비가 부족했다. Round 2에서 운영 안전장치와 제품화 현실을 반영하여 설계를 대폭 보강한다.
