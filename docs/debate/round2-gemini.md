## 논점 1: NER 모델 선택

### 받은 비평 요약
- **Claude:** KoELECTRA-Small(INT8) 선택에 동의하나, **실제 기업 문서에서의 F1 하락** 우려. MVP에서 NER을 **Beta**로 제한하고 A/B 테스트 제안.
- **Codex:** "AI on Endpoint" 자체가 위험. 라벨 설계(Product Code vs Phone) 문제 지적. **AI 예산(분당 N회)** 하드캡 필요.

### Gemini의 응답
- **[수용]**

**상세 내용:**
1.  **Beta 전환:** MVP 단계에서 NER을 **기본 비활성화(Opt-in) 또는 "실험적 기능"**으로 격하하는 것에 전적으로 동의합니다.
2.  **역할 재정의:** NER의 역할을 "탐지(Detection)"에서 **"검증/승격(Verification/Boosting)"**으로 축소합니다. 정규식으로 찾은 후보 중 모호한 것(라벨 없는 전화번호 등)의 신뢰도를 높이는 보조 수단으로만 사용합니다.
3.  **AI 예산 도입:** Codex가 제안한 **Token Bucket 알고리즘 기반의 "AI Budget (예: 분당 30회)"**을 도입하여, 어떤 경우에도 CPU 점유율이 임계치를 넘지 않도록 강제합니다.
4.  **토크나이저 성능:** Claude의 우려와 달리, Rust 기반 `tokenizers` 크레이트는 Python 대비 압도적인 성능(GB당 20초 미만)을 보여주므로, Rust 클라이언트에서의 병목은 상대적으로 덜할 것입니다. 단, 이를 감안해도 하드캡은 필수입니다.

## 논점 2: 탐지 파이프라인 구조

### 받은 비평 요약
- **Claude:** "실시간" 정의의 모호함 지적. NER 격리(Isolation) 필수.
- **Codex:** `ReadDirectoryChangesW`는 이벤트 유실 발생. **USN Journal이 백본(Backbone)**이어야 함. 텍스트 추출기 격리 및 타임아웃 필수.

### Gemini의 응답
- **[수용]**

**상세 내용:**
1.  **실시간성 재정의:** "즉시 탐지" 마케팅 용어를 폐기하고, **"Eventual Consistency (최종 일관성)"**를 기술적 목표로 삼습니다. `ReadDirectoryChangesW`는 "핵심 경로(바탕화면, 문서)"에만 적용하고, 전체 볼륨은 USN Journal 기반의 증분 스캔이 담당합니다.
2.  **Worker 격리:** 텍스트 추출(Parsing)과 NER 추론은 반드시 **별도 프로세스(Worker)**에서 실행하며, 메인 서비스는 이를 감시(Watchdog)하는 구조를 확정합니다.
3.  **타임아웃/Quarantine:** 추출기 프로세스에 **Hard Timeout(예: 30초)**을 걸고, 3회 이상 크래시를 유발한 파일은 `Quarantine DB`에 등록하여 재스캔을 영구히(또는 다음 업데이트까지) 방지합니다.

## 논점 3: 기술 스택 (Rust vs 대안)

### 받은 비평 요약
- **Claude:** C#/.NET 하이브리드 반대. Rust 클라이언트 + Go 서버 확정.
- **Codex:** 팀 역량 부족 시 C# 고려 가능. Rust 크레이트 생태계의 미성숙함(Windows API, HWP 등) 우려.

### Gemini의 응답
- **[부분 수용]** (C# 하이브리드는 반대, HWP 리스크는 인정)

**상세 내용:**
1.  **Rust 고수:** 엔드포인트 에이전트의 성능과 안정성을 위해 **Rust 단일 언어 원칙**을 고수합니다. C# 혼용은 배포 사이즈 증가, 런타임 의존성, FFI 복잡도 등 득보다 실이 큽니다.
2.  **Go 서버:** 서버 사이드는 생산성이 높은 Go로 확정합니다.
3.  **HWP 리스크 대응:** Codex가 지적한 `hwp-rs`의 미성숙함은 심각한 리스크입니다. 이에 대한 대안으로 **HWP 지원을 MVP에서 공식 제외**하거나, **Python/C++로 작성된 안정적인 파서를 바이너리 형태로 래핑하여 호출**하는 방식을 "Phase 0"에서 검증 후 결정하겠습니다. (논점 8에서 상술)

## 논점 5: 보안/라이선스 설계

### 받은 비평 요약
- **Claude:** 5만 대 인증서 관리(PKI)는 운영 지옥. Join Token + JWT 제안.
- **Codex:** Pinning은 기업 프록시 환경에서 장애 유발. Pinning 기본 Off 제안.

### Gemini의 응답
- **[수용]**

**상세 내용:**
1.  **인증 체계 변경:** mTLS를 "Phase 2 목표"로 미루고, MVP는 **"Enrollment Token(1회용) -> Long-lived JWT(기기 종속)"** 방식을 채택합니다. 이는 운영 복잡도를 획기적으로 낮춥니다.
2.  **Pinning 제거:** Certificate Pinning은 기본 **Disabled**로 설정하고, 보안이 극도로 중요한 고객사만 옵션으로 켜도록 변경합니다.
3.  **라이선스:** Codex의 "Signed Lease + Server Enforcement" 모델을 채택하되, 오프라인 Grace Period를 넉넉히(예: 30일) 두어 오탐으로 인한 차단을 방지합니다.
