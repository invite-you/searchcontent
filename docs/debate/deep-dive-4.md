# Deep Dive 4: 모델 파일 보호 수준 (암호화 vs 서명)

**토론 일자:** 2026-02-11
**참여자:** 입장A 옹호자, 입장B 옹호자, 중립 심판

---

## 1. 배경 및 쟁점 요약

PII Scanner 클라이언트는 KoELECTRA 기반 NER 모델을 ONNX 형식으로 탑재하여 비정형 개인정보를 탐지한다. 이 모델 파일(약 14MB)은 클라이언트 PC의 디스크에 저장되며, 스캔 시 메모리에 로드된다. 쟁점은 이 모델 파일의 **보호 수준**을 어디까지 설정할 것인가이다.

**핵심 쟁점:**
- **입장 A (Gemini+Claude 절충):** MVP부터 모델 파일 디스크 암호화 필수(at-rest encryption). AES-256-GCM으로 암호화하고, Windows DPAPI로 키를 보호한다. 메모리 로드 시에만 복호화하며, 복호화 실패 시 NER을 비활성화하고 Regex-only 모드로 폴백한다. IP 보호가 주 목적이다.
- **입장 B (Codex):** 암호화보다 서명 검증(무결성)이 우선이다. at-rest 암호화는 IP 보호 효과가 제한적이므로(메모리 덤프로 우회 가능), MVP에서는 서명 검증을 필수로 하고 at-rest 암호화는 옵션으로 둔다. 자체 파인튜닝 모델 배포 시점(Phase 2)에서 암호화를 기본으로 격상한다.

**합의 사항:** 세 팀 모두 "서명 검증(Ed25519)은 MVP 필수"에 동의한다.

**합의 실패 이유:** "at-rest 암호화가 MVP에 꼭 필요한가"에서 갈린다. 입장 A는 보안 기본값으로 필수를 주장하고, 입장 B는 비용 대비 효과가 제한적이므로 옵션으로 충분하다고 주장한다.

**고려해야 할 핵심 요소:**
- MVP 모델은 공개 KoELECTRA 기반이므로 독자적 IP 가치가 제한적
- ONNX 모델의 역공학 난이도
- Windows DPAPI 키 관리의 운영 복잡도
- AES-256-GCM 복호화의 모델 로드 시 성능 영향
- 기업 환경에서 메모리 덤프 공격의 현실적 위협 수준
- 공급망(supply chain) 공격 벡터
- 경쟁사의 모델 보호 관행

## 2. 입장 A 옹호 (MVP부터 암호화 필수)

### 2.1 핵심 논거

1. **심층 방어(Defense in Depth) 원칙:** 서명 검증은 무결성만 보장하고, 기밀성은 보장하지 않는다. 모델 파일이 평문으로 디스크에 존재하면, 파일 탐색기로 바로 복사할 수 있다. 암호화는 이 "우발적 유출"을 차단하는 추가 방어 계층이다. 보안은 단일 방어선이 아닌 다층 방어가 원칙이며, MVP라 해도 예외가 아니다.

2. **기업 고객의 보안 기대치:** 수만 대 PC를 운영하는 기업은 보안 솔루션 도입 시 "모델 파일이 평문으로 노출되어 있다"는 사실을 감사(audit)에서 문제 삼을 수 있다. "서명만 했고 암호화는 하지 않았습니다"라는 답변은 기업 보안팀의 체크리스트를 통과하기 어렵다. 특히 한국 기업의 ISMS-P 인증 심사에서 "민감 데이터 처리 컴포넌트의 at-rest 보호"는 흔한 점검 항목이다.

3. **MVP에서 나중에 추가하기 어렵다:** 암호화를 Phase 2로 미루면, 이미 배포된 수천 대의 클라이언트에서 평문 모델 파일을 암호화된 파일로 마이그레이션해야 한다. 이 마이그레이션 자체가 복잡하고 장애 가능성이 있다. 처음부터 암호화 파이프라인을 구축하면 이 비용을 회피할 수 있다.

4. **성능 오버헤드가 미미하다:** PoC 결과에 따르면 14MB 모델의 AES-256-GCM 복호화는 약 34ms이다. 모델 로드 자체가 수백 ms~수 초 소요되므로, 34ms의 추가 오버헤드는 사용자 체감에 전혀 영향이 없다.

### 2.2 기술적 근거

**암호화 파이프라인 설계:**

```
[빌드 서버 (배포 시)]
1. 모델 파일을 Ed25519로 서명 (무결성 보장)
2. AES-256-GCM으로 암호화 (기밀성 보장)
3. 암호화된 모델 + 서명을 번들로 패키징

[클라이언트 PC (설치 시)]
1. DPAPI를 통해 AES-256 키를 현재 PC/사용자에 바인딩하여 저장
2. 키는 Windows 자격 증명 저장소에 보관

[클라이언트 PC (스캔 시)]
1. DPAPI로 AES-256 키 복원
2. 모델 파일 복호화 (AES-256-GCM, ~34ms)
3. 복호화된 데이터의 서명 검증 (Ed25519, ~38ms)
4. ONNX Runtime에 모델 로드
5. 스캔 완료 후 메모리 해제
```

**DPAPI 키 보호의 장점:**
- Windows 운영체제에 내장된 키 관리 인프라를 활용하므로 별도 키 서버가 불필요하다.
- DPAPI는 사용자의 로그인 자격 증명(또는 머신 자격 증명)으로 키를 보호한다. 디스크에서 키 파일을 복사해도 다른 PC/사용자에서는 복호화할 수 없다.
- Active Directory 환경에서 DPAPI 마스터 키 백업이 자동으로 이루어지므로, 사용자 프로필 손상 시에도 도메인 컨트롤러를 통한 복구가 가능하다.

**복호화 실패 시 폴백 전략:**
- DPAPI 키 복원 실패, AES-GCM 복호화 실패, 서명 검증 실패 등 모든 보호 계층의 실패에 대해 **NER 비활성화 + Regex-only 모드**로 전환한다.
- 실패 이벤트를 서버에 보고하여 관리자가 원격 조치할 수 있게 한다.
- Regex-only 모드에서도 구조화 PII(주민번호, 카드번호, 전화번호 등)는 정상 탐지되므로, 핵심 기능은 유지된다.

**성능 측정 (PoC 기준):**

| 연산 | 14MB 모델 기준 소요 시간 |
|------|--------------------------|
| SHA-256 해시 | 37.91ms |
| Ed25519 서명 검증 | 0.16ms |
| AES-256-GCM 복호화 | 33.89ms |
| **합계 (서명+암호화)** | **~72ms** |
| **서명만** | **~38ms** |
| **차이** | **~34ms** |

34ms의 차이는 ONNX Runtime 모델 로드(500ms~2초) 대비 무시할 수 있는 수준이다.

### 2.3 예상 반론에 대한 재반박

**반론 1: "메모리 덤프로 복호화된 모델을 추출할 수 있으므로 at-rest 암호화는 무의미하다."**

> 재반박: 메모리 덤프 공격은 **관리자 권한 + 디버깅 도구 + 프로세스 식별 능력**을 요구한다. 기업 환경에서는 엔드포인트 보안 솔루션(EDR)이 이러한 행위를 탐지하고 차단한다. 반면 디스크의 평문 파일 복사는 일반 사용자 권한으로도 가능하다. 공격의 난이도를 한 단계 올리는 것 자체가 보안의 가치이다. "완벽하지 않으므로 하지 않는다"는 보안의 논리가 아니다.

**반론 2: "MVP 모델은 공개 KoELECTRA 기반이므로 IP 가치가 없다."**

> 재반박: 두 가지 관점에서 반론한다. 첫째, MVP 모델이라 해도 PII 탐지에 특화된 **파인튜닝 데이터와 레이블링 전략**이 반영되어 있다. 모델 가중치에서 학습 데이터의 특성을 역추론하는 것은 충분히 가능하다. 둘째, 암호화 파이프라인을 Phase 2에서 추가하면 마이그레이션 비용이 발생한다. 지금 34ms의 오버헤드로 파이프라인을 구축해두면, Phase 2에서 자체 모델로 전환할 때 추가 작업 없이 바로 보호된다.

**반론 3: "DPAPI 키 관리가 운영 복잡도를 높인다."**

> 재반박: DPAPI는 Windows가 자동으로 관리하는 인프라이다. 개발자가 별도로 키 저장소를 구현하거나, 키 로테이션 로직을 작성할 필요가 없다. `CryptProtectData`/`CryptUnprotectData` API 호출 두 개로 구현이 완료된다. Active Directory 환경에서는 마스터 키 백업도 자동이다. 운영 복잡도는 "DPAPI 실패 시 폴백"만 구현하면 되며, 이는 입장 B에서도 동의한 Regex-only 폴백과 동일하다.

## 3. 입장 B 옹호 (서명만 필수, 암호화는 Phase 2)

### 3.1 핵심 논거

1. **위협 모델의 정확한 분석:** 모델 파일 보호에서 가장 심각한 위협은 **변조(tamper)**이지 **유출(theft)**이 아니다. 공격자가 모델 파일을 변경하여 특정 PII 패턴을 무시하게 만들면, 탐지 체계 전체가 무력화된다. 서명 검증은 이 위협을 직접 차단한다. 반면 at-rest 암호화가 방어하는 "유출"은, 클라이언트 PC에 키도 함께 존재하는 이상 결정적 방어가 될 수 없다.

2. **MVP 모델의 IP 가치 현실:** MVP에서 사용할 모델은 공개된 KoELECTRA 사전학습 가중치에 NER 태스크를 파인튜닝한 것이다. 기반 아키텍처와 사전학습 가중치는 누구나 접근 가능하다. 파인튜닝 레이어의 가치는 있으나, 이것이 "AES-256-GCM + DPAPI 키 관리"라는 운영 복잡도를 정당화할 만큼의 IP인지는 의문이다. 진정한 IP 가치는 자체 대규모 데이터로 학습한 Phase 2 모델에서 발생한다.

3. **"키가 클라이언트에 존재하는 암호화"의 본질적 한계:** at-rest 암호화에서 복호화 키는 반드시 클라이언트 PC에 존재해야 한다. DPAPI로 보호하더라도, 같은 사용자 컨텍스트(또는 로컬 관리자)가 DPAPI를 호출하면 키가 복원된다. 이는 "문을 잠그고 열쇠를 문 앞에 놓아둔 것"에 비유할 수 있다. 캐주얼한 복사는 막지만, 의도적 공격에는 무력하다. 이 제한적 효과를 위해 MVP의 개발 범위를 늘리는 것은 비효율적이다.

4. **공급망 공격이 더 현실적 위협이다:** 실제 기업 환경에서 모델 파일에 대한 위협은 "누군가 디스크에서 복사해 간다"보다 "업데이트 과정에서 변조된 모델이 배포된다"가 더 현실적이다. 서명 검증은 이 공급망 공격을 직접 차단한다. 암호화는 공급망 공격에 대해 아무런 방어력이 없다.

### 3.2 기술적 근거

**서명 기반 보호 설계:**

```
[빌드 서버 (배포 시)]
1. 모델 + 토크나이저 + config를 단일 .pii 번들(ZIP/Tar)로 패키징
2. 번들 전체를 Ed25519 개인키로 서명
3. 서명을 번들 메타데이터에 포함

[클라이언트 PC (로드 시)]
1. 번들의 Ed25519 서명 검증 (공개키는 바이너리에 내장)
2. 서명 유효 → 모델 로드
3. 서명 무효 → 로드 거부, NER 비활성화, 서버에 알림

[업데이트 체인]
1. 서버에서 새 모델 번들 배포 시에도 동일 서명 검증
2. 번들 내에 버전 번호 포함 → 다운그레이드 공격 방지
3. 이전 버전으로 롤백 시에도 서명 유효성 확인
```

**서명 검증의 보안 강도:**
- Ed25519 서명은 256비트 보안을 제공하며, 위조가 계산적으로 불가능하다.
- 서명 검증 비용은 0.16ms로 사실상 무시할 수 있는 수준이다.
- 공개키만 클라이언트에 존재하므로, 클라이언트 PC가 탈취되어도 새로운 서명을 생성할 수 없다. 이는 암호화에서 "키와 데이터가 같은 곳에 존재하는" 문제와 근본적으로 다르다.

**암호화 대비 서명의 장애 모드 비교:**

| 장애 상황 | 서명만 | 서명 + 암호화 |
|-----------|--------|---------------|
| 모델 파일 손상 | 서명 검증 실패 → 명확한 오류 | 복호화 실패 → 원인 불명확 (키 문제? 파일 손상?) |
| 사용자 프로필 재생성 | 영향 없음 | DPAPI 키 손실 → 모델 사용 불가 |
| OS 재설치 (동일 PC) | 영향 없음 | DPAPI 키 손실 → 모델 재배포 필요 |
| Active Directory 이관 | 영향 없음 | DPAPI 마스터 키 복원 필요 |
| BitLocker 복구 모드 | 영향 없음 | DPAPI 접근에 영향 가능 |

**DPAPI 의존의 운영 리스크:**
- 사용자 암호 초기화 시 DPAPI 마스터 키가 무효화될 수 있다(도메인 환경에서는 백업 복구 가능, 로컬 계정에서는 불가).
- Windows 재설치, 프로필 삭제/재생성 시 DPAPI 키가 소실된다.
- 이러한 장애는 "NER이 갑자기 안 된다"로 나타나며, 원인 진단이 어렵다. 지원 티켓의 원인 중 상당수가 이런 "암호화 키 소실"이 될 수 있다.

**경쟁사 관행 분석:**
- 대부분의 엔드포인트 보안 솔루션은 탐지 엔진(시그니처, 모델)에 대해 **서명 검증**은 수행하지만, at-rest 암호화는 하지 않는다. 모델 가중치 자체의 기밀성보다 무결성을 우선시하는 것이 업계 관행이다.
- Windows Defender의 정의 파일, CrowdStrike의 센서 업데이트 등도 서명 기반 검증을 사용하며, 디스크 암호화는 적용하지 않는다.

### 3.3 예상 반론에 대한 재반박

**반론 1: "기업 보안팀이 at-rest 보호를 요구할 것이다."**

> 재반박: 기업 보안팀이 실제로 검증하는 것은 "민감 데이터의 at-rest 보호"이다. 모델 파일은 "민감 데이터"가 아니라 "탐지 도구"이다. ISMS-P나 ISO 27001에서 "AI 모델 파일의 디스크 암호화"를 명시적으로 요구하는 조항은 없다. 보안팀이 실제로 요구하는 것은 "탐지 엔진이 변조되지 않음을 보장"하는 것이며, 이는 서명 검증이 직접 해결한다. 만약 특정 고객이 at-rest 보호를 강하게 요구한다면, 그때 해당 고객 배포에 한해 옵션으로 활성화하면 된다.

**반론 2: "암호화 파이프라인을 나중에 추가하면 마이그레이션 비용이 크다."**

> 재반박: 마이그레이션 비용은 과장되었다. 모델 업데이트는 서버 주도로 이루어지며, 새 버전 배포 시 암호화된 번들로 교체하면 된다. 기존 평문 모델 파일은 새 번들 설치 시 덮어쓰기되므로 별도 마이그레이션 로직이 불필요하다. 클라이언트 업데이트 메커니즘이 이미 존재하므로, "모델 포맷이 바뀌었으니 새 버전을 받으세요"는 정상적인 업데이트 프로세스의 일부이다.

**반론 3: "34ms의 오버헤드는 미미하므로 하지 않을 이유가 없다."**

> 재반박: 문제는 34ms가 아니라, 34ms 뒤에 숨어 있는 **운영 복잡도**이다. DPAPI 키 생성, 키 백업/복구 전략, 프로필 손상 시 대응, AD 이관 시 키 이전, 비도메인 PC 지원, DPAPI 실패 진단 로직 등이 모두 추가된다. PoC에서 측정한 34ms는 "행복한 경로(happy path)"의 비용일 뿐이고, 실제 운영에서는 장애 경로(unhappy path)의 비용이 훨씬 크다. MVP에서는 행복한 경로와 장애 경로 모두를 최소화해야 한다.

## 4. 중립 심판 평가

### 4.1 입장 A의 강점과 약점

**강점:**
- **심층 방어 논리가 건전하다:** 서명(무결성) + 암호화(기밀성)의 조합은 보안 원칙에 부합한다. "하나가 뚫려도 다른 하나가 방어"하는 구조는 이론적으로 옳다.
- **성능 영향이 미미하다:** 34ms의 추가 비용은 사용자 체감에 전혀 영향이 없다. "성능 때문에 못 한다"는 반론은 성립하지 않는다.
- **기업 영업에서 어필한다:** "AES-256 암호화 + DPAPI 키 보호"는 보안 체크리스트에서 눈에 띄는 항목이다. 경쟁 PT에서 차별점으로 내세울 수 있다.
- **Regex-only 폴백이 안전망을 제공한다:** 복호화 실패가 전체 서비스 중단이 아닌 기능 축소(NER 비활성화)로 이어지므로, 장애 영향을 제한한다.

**약점:**
- **위협 모델이 모호하다:** "IP 보호"를 주장하지만, MVP 모델은 공개 KoELECTRA 기반이다. 보호할 IP의 실체가 불분명한 상태에서 암호화 인프라를 구축하는 것은 시기상조일 수 있다.
- **DPAPI 의존이 장애 표면을 넓힌다:** DPAPI는 Windows가 관리하지만, 그 관리가 투명하지 않다. 사용자 암호 변경, 프로필 손상, OS 재설치 등에서 키가 소실되는 시나리오가 다양하며, 각각에 대한 복구 절차가 필요하다. 이는 MVP의 지원 부담을 늘린다.
- **"문을 잠그고 열쇠를 문 앞에 놓아두는" 구조의 본질적 한계를 해소하지 못한다:** DPAPI 키는 결국 클라이언트 PC에 존재하며, 로컬 관리자 권한으로 복원 가능하다. 결의가 있는 공격자에 대한 방어력은 제한적이다.

### 4.2 입장 B의 강점과 약점

**강점:**
- **위협 모델 분석이 정확하다:** 모델 변조가 모델 유출보다 더 심각한 위협이라는 분석은 타당하다. 변조된 모델은 "PII가 있는데 없다고 보고"하는 결과를 초래하며, 이는 솔루션의 존재 이유를 부정한다.
- **장애 표면이 최소화된다:** 서명 검증만 사용하면 DPAPI 키 관리에 따른 모든 장애 시나리오가 사라진다. "서명 유효/무효"의 이진 판정은 장애 원인 진단이 명확하다.
- **업계 관행에 부합한다:** 엔드포인트 보안 제품들이 탐지 엔진에 서명 검증을 적용하고 at-rest 암호화는 하지 않는 것이 일반적인 관행이다.
- **공급망 공격 방어에 집중한다:** 실제로 발생 가능성이 높은 위협(업데이트 변조)에 자원을 집중하는 것이 합리적이다.
- **MVP 스코프를 관리한다:** 암호화 관련 개발/테스트/운영 비용을 절약하여 MVP의 핵심 기능(탐지 정확도, 안정성)에 집중할 수 있다.

**약점:**
- **"캐주얼 복사" 방어를 포기한다:** 디스크의 평문 모델 파일은 일반 사용자도 복사할 수 있다. 경쟁사가 고객사 PC에서 모델 파일을 획득하는 시나리오에서 방어 수단이 없다.
- **Phase 2 전환 시점의 판단이 어렵다:** "자체 모델의 IP 가치가 충분해지면 암호화를 도입한다"는 전략이지만, 그 "충분한 시점"을 어떻게 판단할지 모호하다.
- **기업 보안팀의 기대와 불일치할 가능성:** 일부 보안 의식이 높은 고객은 "왜 모델 파일이 평문인가"를 질문할 수 있다. "업계 관행"이라는 답변이 모든 고객을 설득하지는 못한다.

### 4.3 핵심 트레이드오프 분석

| 평가 항목 | 입장 A (암호화 필수) | 입장 B (서명만 필수) |
|-----------|----------------------|----------------------|
| **무결성 보장** | 상 (서명 + 암호화 이중 검증) | 상 (서명 단독으로 충분) |
| **기밀성 보장** | 중 (DPAPI 한계 내에서 유효) | 하 (평문 노출) |
| **운영 안정성** | 중 (DPAPI 장애 시나리오 존재) | 상 (장애 표면 최소) |
| **구현 복잡도** | 중 (DPAPI 통합 + 폴백 로직) | 하 (서명 검증만) |
| **MVP 개발 속도** | 중 (암호화 파이프라인 추가 공수) | 상 (최소 범위) |
| **지원 부담** | 중~상 (키 소실 관련 티켓) | 하 (단순한 장애 모드) |
| **영업 어필** | 상 ("AES-256 암호화" 키워드) | 중 ("서명 검증" 키워드) |
| **공급망 방어** | 상 (서명이 담당) | 상 (서명이 담당) |
| **미래 확장성** | 상 (파이프라인 이미 존재) | 중 (Phase 2에서 추가 필요) |

**결정적 변수:** MVP 모델의 실질적 IP 가치.

- **모델이 공개 KoELECTRA를 그대로 파인튜닝한 수준이면:** IP 가치가 제한적이므로 암호화의 비용 대비 효과가 낮다. 입장 B가 합리적이다.
- **모델에 독자적 학습 데이터, 커스텀 아키텍처, 도메인 특화 지식이 반영되면:** IP 가치가 높으므로 암호화가 정당화된다. 입장 A가 합리적이다.

현재 MVP 단계에서는 전자에 해당할 가능성이 높다. ONNX 형식의 역공학 난이도도 고려해야 하는데, ONNX는 공개 형식이므로 모델 구조와 가중치를 그대로 읽을 수 있다. 그러나 이것은 암호화 여부와 무관하게 메모리 로드 시점에서 동일하게 적용되는 문제이다.

**기업 환경에서 메모리 덤프 공격의 현실성:**
- 기업 PC에는 EDR(Endpoint Detection and Response)이 배포되어 있어 프로세스 메모리 덤프를 탐지/차단한다.
- 디버거 연결, `MiniDumpWriteDump` API 호출 등은 보안 정책으로 제한된다.
- 그러나 로컬 관리자 권한이 있으면 우회 가능하다. 다만, 관리자 권한 보유자가 모델을 탈취할 동기가 있는 시나리오는 내부 위협(insider threat)이며, 이는 암호화로 방어할 범위를 넘어선다.

### 4.4 조건부 결론

양측 논거를 종합하면, 핵심은 **"MVP에서 보호해야 할 IP가 실제로 존재하는가"**이다.

1. **MVP 모델이 공개 KoELECTRA 기반 파인튜닝인 경우 (현재 계획):** 모델 자체의 IP 가치가 제한적이다. at-rest 암호화로 보호하려는 대상의 실질적 가치가 낮은 반면, DPAPI 키 관리에 따른 운영 복잡도와 지원 부담은 확실하다. 이 경우 입장 B가 더 합리적이다.

2. **향후 자체 대규모 데이터로 학습한 모델을 배포하는 경우:** IP 가치가 본격적으로 발생하며, 이때 at-rest 암호화의 비용이 정당화된다. 단, 이 시점에서도 "키가 클라이언트에 존재하는 한계"는 동일하므로, 암호화는 "캐주얼 복사 방지" 수준의 효과임을 인식해야 한다.

3. **입장 A의 "나중에 추가하면 마이그레이션 비용이 크다"는 주장에 대해:** 입장 B의 반론이 유효하다. 모델 업데이트가 서버 주도이고 기존 파일을 덮어쓰는 구조이므로, 암호화된 포맷으로의 전환은 정상적인 업데이트 프로세스로 처리 가능하다. 다만, 클라이언트 코드에 복호화 로직을 추가하는 업데이트가 모델 포맷 변경보다 선행되어야 하므로, 2단계 롤아웃(코드 업데이트 → 암호화 모델 배포)이 필요하다. 이는 복잡하지만 불가능하지 않다.

## 5. 최종 추천안

**추천: 입장 B (서명 검증 필수 + at-rest 암호화는 옵션) 채택, 단 아래 조건 부여**

### 구현 전략

```
[Phase 1: MVP]
├── 필수: Ed25519 서명 검증
│   ├── 모델 + 토크나이저 + config를 단일 서명 번들로 패키징
│   ├── 서명 검증 실패 시 → NER 비활성화, Regex-only 폴백
│   ├── 번들 내 버전 번호 포함 (다운그레이드 방지)
│   └── 서명 공개키는 클라이언트 바이너리에 내장
├── 필수: 안전한 업데이트 체인
│   ├── 서버 → 클라이언트 모델 배포 시 TLS + 서명 이중 검증
│   └── 롤백 시에도 서명 유효성 확인
├── 옵션: at-rest 암호화 (기본 OFF)
│   ├── 서버 정책으로 특정 고객/그룹에 한해 활성화 가능
│   ├── 활성화 시: AES-256-GCM + DPAPI 키 보호
│   └── 복호화 실패 시: NER 비활성화 + Regex-only 폴백
└── 구현: 암호화 모듈을 인터페이스로 분리하여 Phase 2 전환에 대비

[Phase 2: 자체 모델 배포 시]
├── at-rest 암호화를 기본 ON으로 격상
├── DPAPI 외 TPM 기반 키 보호 옵션 추가 검토
├── ONNX Runtime 커스텀 로더를 통한 메모리 내 복호화 최적화
└── 모델 난독화(obfuscation) 추가 검토 (구조 역공학 방지)
```

### 핵심 설계 원칙

1. **무결성이 기밀성보다 우선한다.** 모델 변조는 탐지 체계를 무력화하는 치명적 위협이다. 서명 검증은 어떤 경우에도 생략할 수 없다. 암호화는 서명 검증 위에 얹는 선택적 계층이다.

2. **암호화 모듈은 인터페이스로 분리한다.** MVP에서 암호화가 옵션이더라도, 코드 구조상 `ModelLoader` 트레이트(또는 인터페이스)를 통해 `PlainLoader`와 `EncryptedLoader`를 교체 가능하게 설계한다. Phase 2 전환 시 코드 변경을 최소화한다.

3. **모든 보호 계층의 실패는 Regex-only 폴백으로 수렴한다.** 서명 검증 실패, 복호화 실패 등 어떤 보호 관련 오류도 서비스 중단이 아닌 기능 축소(NER 비활성화)로 처리한다. 이는 "사용자 업무를 방해하지 않는다"는 제품의 핵심 원칙과 일치한다.

4. **옵션 암호화 활성화는 서버 정책으로 제어한다.** 클라이언트가 스스로 암호화 여부를 결정하지 않는다. 서버에서 정책을 내려보내 특정 고객사/그룹에 한해 암호화를 활성화할 수 있게 한다. 이는 고객별 보안 요구사항에 유연하게 대응할 수 있게 한다.

5. **서명 번들 포맷은 확장 가능하게 설계한다.** 번들 헤더에 `encryption: none | aes-256-gcm` 필드를 포함하여, 클라이언트가 번들을 받았을 때 암호화 여부를 자동으로 판별하고 적절한 로더를 선택한다.

### 리스크 완화 방안

| 리스크 | 완화 방안 |
|--------|-----------|
| 고객사가 at-rest 보호를 필수로 요구 | 옵션 암호화를 해당 고객에 한해 활성화 (서버 정책) |
| 평문 모델 파일의 캐주얼 복사 | Phase 1에서 파일 ACL(접근 제어 목록)로 일반 사용자 읽기 차단, 서비스 계정만 접근 허용 |
| Phase 2 암호화 전환 시 마이그레이션 문제 | 2단계 롤아웃: (1) 복호화 가능 클라이언트 배포 → (2) 암호화 모델 배포 |
| 서명 공개키 교체 필요 | 번들 헤더에 키 ID 포함, 클라이언트에 복수 공개키 내장하여 키 로테이션 지원 |
| 공급망 공격으로 변조된 모델 배포 | 빌드 서버의 서명 개인키를 HSM 또는 격리된 환경에서 관리 |

### 이 추천이 양측 입장을 어떻게 수용하는가

- **입장 A에서 수용한 것:** AES-256-GCM + DPAPI 설계를 옵션 구현으로 포함. 복호화 실패 시 Regex-only 폴백 필수. 암호화 모듈을 인터페이스로 분리하여 Phase 2 전환 준비. 파일 ACL을 통한 기본적 접근 제한.
- **입장 B에서 수용한 것:** 서명 검증을 MVP 필수로 채택. at-rest 암호화를 MVP 기본값에서 제외. 안전한 업데이트 체인(서명 번들, 롤백 검증)을 핵심 요구사항으로 포함. 서명 번들 포맷 설계.
- **절충 지점:** 암호화를 "하지 않는다"가 아니라 "옵션으로 둔다"로 결정. 이는 보안 요구사항이 높은 고객에게는 활성화하고, 그렇지 않은 환경에서는 운영 복잡도를 최소화하는 실용적 선택이다. 또한 파일 ACL을 통해 암호화 없이도 "캐주얼 복사"를 어느 정도 방어하여 입장 A의 우려를 완화한다.
