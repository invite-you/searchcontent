# Feasibility Review (PII Scanner)

> 기준 문서: `AGENTS.md` (프로젝트 개요/목표/기술 방향)

## 1) 구성요소별 기술 난이도 평가

| 구성요소 | 핵심 난점 | 난이도(1~3인 팀) |
|---|---|---|
| Windows 클라이언트 에이전트(서비스/백그라운드) | 안정적 서비스 운영, 권한/정책, 업데이트/복구 | 중 |
| 실시간 파일 감지(생성/수정 이벤트 기반) | 이벤트 폭주, 중복/누락, 잠긴 파일/부분쓰기 처리, 백프레셔 | 상 |
| 증분 스캔(캐시/체크포인트) | 파일 해시/메타데이터 캐시 설계, 재부팅/이동/리네임 정합성 | 중 |
| 텍스트 추출(파일 포맷 처리) | 포맷 다양성, 인코딩, 큰 파일/바이너리, 추출 품질이 탐지 품질을 좌우 | 상 |
| 1차 탐지(정규식+체크섬) | 한국형 PII(주민번호/카드/계좌 등) 오탐 억제, 컨텍스트 룰 설계 | 중 |
| 2차 탐지(NER, KoELECTRA + ONNX 양자화/CPU) | 속도/메모리, 모델 배포/업데이트, 임계치/캘리브레이션, 도메인 시프트 | 상 |
| 하이브리드 오케스트레이션("의심 텍스트에만 NER") | "의심" 정의/튜닝, 누락 vs 비용 트레이드오프 | 상 |
| 서버(디바이스/정책/결과 수집) | 수만 대 배치/재시도/중복 방지, 장기 저장, 멀티테넌시 | 중~상 |
| 리포팅/관리 인사이트(대시보드) | 데이터 모델(파일/위치/유형/시간), 집계/추세, 권한 모델 | 중 |
| 상용 보안/라이선스 | 코드 서명/업데이트 신뢰, 장치 식별, 키 관리, 조작 방지 | 상 |
| DRM 파일 대응 | "스캔 불가" 영역(탐지 공백)과 정책/커뮤니케이션 | 상 |

## 2) 1~3인 팀 기준 MVP 범위 제안과 Phase별 로드맵

### MVP (목표: 현장 파일럿에서 가치 증명)

* 대상: 100~500대 파일럿(기업 1곳) 수준 운영 가능
* 클라이언트: 지정 폴더 "증분(스케줄)" 스캔 + 변경 이벤트 수집(즉시 스캔은 제한적으로)
* 클라이언트: 파일 타입은 텍스트 계열 우선(txt/csv/log/json/xml 등), 큰/바이너리 파일 제외
* 클라이언트: 1차 탐지(정규식 + 체크섬)로 구조화 PII 우선(주민번호/카드/이메일/전화 등)
* 클라이언트: 성능 가드레일(CPU/IO 제한, 파일 크기 상한, 백오프/큐)
* 클라이언트: 서버 전송은 원문이 아니라 최소 증거(요약/카운트/마스킹 스니펫/해시)
* 서버: 디바이스 등록, 정책 배포(스캔 폴더/룰셋/스케줄), 결과 수집 API
* 서버: 기본 대시보드(조직/PC/폴더별 발견량, 상위 PII 유형)
* MVP에서 제외(또는 "표시만"): DRM 판독/탐지(대신 "스캔 불가 분류 + 현황 리포트")
* MVP에서 제외(또는 "표시만"): 고난도 포맷의 대규모 지원(추출 난이도 높은 포맷은 후순위)
* MVP에서 제외(또는 "표시만"): "모든 변경에 대한" 완전한 실시간 즉시 검사

### Phase 로드맵(권장)

1. Phase 0: 설계/검증 기반(2~4주): PII 분류 체계/오탐-미탐 정의, 성능 SLO(예: CPU/지연)와 운영 시나리오 정의, 샘플 데이터셋/평가 파이프라인 준비
1. Phase 1: MVP(6~10주): 증분 스캔 + 정책/서버/리포트 최소 기능으로 "정확도 + 운영성" 증명
1. Phase 2: 베타(8~12주): 실시간 스캔(큐/백프레셔 포함) 본격화, "의심 구간"에만 NER 적용(이름/주소 등 비정형 PII 확대), 오탐 억제(예외 allow/deny, 관리자 피드백 루프)
1. Phase 3: 상용화/확장(지속): 보안/라이선스 하드닝(키관리, 서명, 업데이트, 멀티테넌시), 포맷 확장(우선순위 기반), 운영/모니터링, 대규모 롤아웃/업데이트 전략

## 3) 기술 스택에 대한 의견 (Rust가 정말 최선인가?)

* 클라이언트 에이전트 관점(장점): Rust는 성능/메모리 안전성/단일 바이너리로 "수만 대" 운영에 유리
* 클라이언트 에이전트 관점(리스크): Windows 서비스/업데이트/파일 감시/포맷 처리에서 소규모 팀 개발 속도 저하 가능
* 권장 결론(팀 역량 기준): 팀이 Rust 숙련이면 Rust로 가되 코어를 모듈화(텍스트 추출/모델 추론/룰 엔진 분리)해서 단계적 확장
* 권장 결론(팀 역량 기준): 팀이 Windows/.NET 강하면 C#(.NET)로 MVP를 빠르게 만든 뒤 병목 코어만 Rust 라이브러리로 분리하는 하이브리드도 현실적
* NER/ONNX 관점: 언어 선택보다 "ONNX Runtime 연동/모델 최적화/캘리브레이션/배포"가 더 큰 비용 요인
* 서버 관점: 서버는 수집/정책/리포팅 중심이면 Rust가 필수는 아니며, 1~3인 팀은 팀 주력 스택(Go/.NET 등)으로 속도를 내는 편이 합리적

## 4) 기술적 리스크 상위 5개와 완화 전략

1. 오탐/과탐(특히 한국어 비정형 PII): 완화 - 평가셋/지표(FP/FN 비용 모델) 정의, 룰 + 컨텍스트 결합, 임계치 캘리브레이션, 조직별 예외/튜닝(allow/deny), 피드백 기반 개선 루프
1. 엔드포인트 성능/업무 방해(실시간 포함): 완화 - 증분 스캔(해시/메타 캐시), 우선순위 큐/백프레셔/백오프, 파일 크기/타입 제한, 낮은 우선순위 실행, 부분쓰기 안정화(지연 재시도), CPU/IO 상한
1. 텍스트 추출 품질/포맷 커버리지: 완화 - "추출기 플러그인" 구조, MVP는 텍스트 계열부터, 포맷별 골든 샘플/회귀 테스트, 추출 실패는 "스캔 불가"로 표기해 운영 가시성 확보
1. NER CPU 추론(속도/메모리/배포/업데이트): 완화 - 의심 구간에만 NER(게이팅), 양자화/스레딩 튜닝, 모델 버전 관리/점진 배포, 결과 캐시/중복 제거, 오프라인 환경 고려한 로컬 모델 관리
1. 상용 보안/라이선스/조작 방지: 완화 - 코드 서명 및 업데이트 신뢰 체인, mTLS/장치 ID, 비밀정보 OS 보안 저장소 활용, 최소 권한/로그/감사, 서버 멀티테넌시 격리(테넌트 키/정책/데이터)
