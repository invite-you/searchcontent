## 1. 기존 문제 해결 검증

### 1.1 높은 오탐/과탐률이 정말 해결되는가?

**결론: 개선은 되지만, 설계서가 주장하는 만큼 "획기적"이지 않을 가능성이 높다.**

**정규식+체크섬의 실질적 기여:**
- 주민등록번호, 카드번호 등 체크섬이 있는 정형 PII는 이미 기존 솔루션도 높은 정확도로 탐지한다. 15년 운영 경험이 있다면 기존 솔루션도 체크섬 검증을 하고 있을 것이다.
- 진짜 문제는 전화번호(체크섬 없음, 신뢰도 0.70), 계좌번호(신뢰도 0.50), 이메일(신뢰도 0.85) 등 **체크섬이 없는 유형**인데, 여기에 대한 해결책이 "문맥 분석"뿐이다.
- 문맥 분석의 가중치(x1.2, x0.5 등)는 **실측 데이터 없이 설정한 추정치**이다. 실제 50,000대 환경에서 이 가중치가 효과적인지 검증할 방법이 설계에 포함되어 있지 않다.

**NER 기반 비정형 PII의 현실적 한계:**
- detection-engine.md는 KoELECTRA-Small의 F1 85.40을 근거로 제시하지만, 이는 **Naver NER 일반 벤치마크**에서의 수치이다.
- PII 도메인에 특화된 NER의 실제 성능은 벤치마크와 크게 다를 수 있다. 특히 한국인 이름은 2~3음절로 일반 명사와 구분이 매우 어렵다. "이상", "박수", "김밥" 같은 Hard Negative 사례에서 실제 F1은 70~80% 수준으로 하락할 가능성이 있다.
- Fine-tuning 데이터 50,000문장 확보 계획이 있으나, **합성 데이터의 한계**를 언급하지 않는다. 실제 기업 문서의 패턴은 합성 데이터와 상당히 다를 수 있다.

**핵심 지적: 오탐률의 정량적 목표가 없다.** "획기적으로 낮출 것"이라고만 되어 있고, 현재 오탐률 대비 목표 오탐률(예: 기존 15% → 목표 5%)이 명시되어 있지 않다. 측정할 수 없으면 개선할 수도 없다.

**대안:**
- MVP 단계에서 A/B 테스트 프레임워크를 포함하여 기존 솔루션과 병행 운영하면서 정량적 비교를 수행할 것
- 오탐률/미탐률 목표를 PII 유형별로 명시할 것 (예: 주민번호 오탐률 < 1%, 이름 오탐률 < 10%)
- 문맥 분석 가중치를 경험적으로 튜닝할 수 있는 자동화된 벤치마크 파이프라인을 설계에 포함할 것

### 1.2 사용자 업무 방해가 실제로 줄어드는가?

**결론: 리소스 관리 전략은 잘 설계되어 있으나, 몇 가지 비현실적 가정이 있다.**

**긍정적 부분:**
- Resource Governor의 사용자 활동 감지 기반 동적 조절은 좋은 접근이다.
- I/O 우선순위 설정, 프레젠테이션 모드 감지, 배터리 모드 대응 등 세심한 설계가 돋보인다.

**비현실적 가정들:**
- "상주 메모리 ~21MB" (detection-engine.md 3.5절): NER 모델 미로드 상태 기준이다. 실제 운영에서 NER 호출이 빈번하면 50MB가 거의 상시 사용될 것이고, 텍스트 추출 버퍼까지 포함하면 **실제 상주 메모리는 100MB 이상**이 될 가능성이 높다.
- system-architecture.md에서는 "전체 Agent 메모리 상한: 300MB"라고 하면서 detection-engine.md에서는 "피크 125MB"라고 한다. **문서 간 수치 불일치**가 있다.
- NER Lazy Loading 후 "5분 미사용 시 언로드"라고 하지만, 실시간 감시에서 의심 패턴이 연속 발생하면 모델 로드/언로드가 반복되어(thrashing) 오히려 성능이 나빠질 수 있다.

**대안:**
- NER 모델의 로드/언로드 전략을 시간 기반(5분)이 아닌, 사용 빈도와 시스템 메모리 상태 기반으로 변경할 것
- 실제 운영 시나리오(예: 사용자가 50개 파일을 동시에 다운로드)에서의 리소스 사용 시뮬레이션 결과를 포함할 것

### 1.3 실시간 감지가 정말 가능한가?

**결론: 부분적으로 가능하지만, ReadDirectoryChangesW의 알려진 한계를 과소평가하고 있다.**

**ReadDirectoryChangesW의 알려진 문제점:**
1. **버퍼 오버플로**: 대량 파일 변경(예: Git checkout, 대규모 복사/붙여넣기) 시 이벤트 유실이 발생한다. 버퍼 오버플로 시 `lpBytesReturned`가 0이 되며, 유실된 이벤트를 복구할 수 없다.
2. **네트워크 드라이브 제한**: 네트워크 경로 감시 시 버퍼 64KB 제한이 있으며, 이를 초과하면 `ERROR_INVALID_PARAMETER` 발생.
3. **감시 대상 상한**: `MAXIMUM_WAIT_OBJECTS` = 64개 제한. 수십 개 디렉토리를 감시하려면 별도 스레드가 필요하다.
4. **NTFS 이외 파일시스템**: FAT32에서는 일부 이벤트를 놓칠 수 있다.

설계서에서는 "USN Journal로 보완"이라고 하지만:
- USN Journal은 **관리자 권한이 필요**하다. Windows 서비스로 동작하면 가능하지만, 일부 기업 환경에서 보안 정책으로 제한될 수 있다.
- USN Journal은 **NTFS 전용**이다. FAT32, exFAT 외장 드라이브는 커버하지 못한다.
- USN Journal 읽기는 볼륨 단위이므로, 대상 디렉토리 필터링 로직이 필요한데 이 부분의 구체적 설계가 없다.

**대안:**
- ReadDirectoryChangesW 버퍼 오버플로 시 자동으로 해당 디렉토리의 증분 스캔을 트리거하는 폴백(fallback) 로직을 명시할 것
- 네트워크 드라이브, 외장 드라이브 시나리오에 대한 명확한 정책(지원/미지원)을 문서화할 것

### 1.4 비정형 PII 탐지 정확도가 상용 수준에 도달할 수 있나?

**결론: 불확실하다. 파수(Fasoo)가 이미 이 영역에 진출해 있으며, 따라잡기 쉽지 않다.**

파수는 2022년에 이미 AI-R Privacy(AI 레이더)를 출시하여 비정형 데이터 내 개인정보를 탐지하고 있으며, **F1 Score 93.1%**를 공개적으로 주장하고 있다. 파수는 자체 딥러닝 기술 + NLP + OCR을 결합했다.

본 설계에서 KoELECTRA-Small INT8로 달성할 수 있는 현실적 F1은:
- 일반 벤치마크: 85.4%
- PII 특화 Fine-tuning 후: 예상 87~90% (낙관적 추정)
- 실제 기업 문서: 예상 80~85% (도메인 갭 반영)

파수 대비 8~13%p 낮은 성능으로 시작하게 될 가능성이 높다. 특히 이름+주소 조합 탐지에서 파수는 이미 15가지 유형을 지원하는 반면, 이 설계는 PER, LOC, ORG 3개 유형에서 시작한다.

**핵심 지적: Fine-tuning 데이터 구축의 난이도를 과소평가하고 있다.** 50,000문장 합성 데이터를 생성하는 것과, 실제 기업 문서에서의 정확도는 별개의 문제이다. 합성 데이터는 실제 문서의 다양한 서식, 오타, 약어, 비표준 표현을 반영하지 못한다.

**대안:**
- 초기 버전에서는 비정형 PII를 "베타 기능"으로 명시하고, 고객 피드백을 통해 점진적으로 개선할 것
- 파일럿 고객사에서 실제 문서(비식별화 후)를 확보하여 Fine-tuning 데이터를 보강하는 프로세스를 설계에 포함할 것
- OCR 미지원(v1)은 비정형 PII 탐지의 큰 공백이다. 스캔본 PDF, 이미지 첨부 문서를 놓치게 된다.

## 2. 구현 현실성

### 2.1 소규모 팀(3~5명)이 구현할 수 있는가?

**결론: 현재 설계 전체를 구현하려면 최소 2년 이상이 필요하며, 3~5명으로는 매우 도전적이다.**

**필요 역량 분석:**

| 영역 | 필요 기술 | 난이도 | 예상 공수 |
|------|----------|--------|----------|
| Rust 클라이언트 코어 | Rust, Tokio, Windows API, FFI | 높음 | 6~9개월 |
| NER 파이프라인 | ML, ONNX, 데이터 구축, Fine-tuning | 높음 | 3~6개월 |
| 텍스트 추출 (8개+ 형식) | 파일 포맷 파싱, 인코딩 | 중간 | 3~4개월 |
| Go 서버 + API | Go, gRPC, PostgreSQL, NATS | 중간 | 4~6개월 |
| React 관리자 대시보드 | React, TypeScript, 차트 라이브러리 | 중간 | 3~4개월 |
| 보안 체계 (mTLS, 라이선스 등) | 암호학, PKI, JWT | 높음 | 3~4개월 |
| 설치/업데이트 | WiX/MSI, 자동 업데이트 | 중간 | 1~2개월 |
| 테스트/QA | 통합 테스트, 다양한 Windows 환경 | 중간 | 지속적 |

**총 예상 공수: 23~35인월 (단일 스레드 기준)**

5명 팀으로 병렬 진행해도 5~7개월이 걸리는데, 이는 각 멤버가 해당 영역의 전문가라는 가정하에서이다. 현실적으로는 Rust + ML + 보안 + Go + React 전 영역을 동시에 커버하는 5인 팀을 구성하기가 매우 어렵다.

### 2.2 기술 스택 복잡도

**핵심 우려: 기술 스택이 지나치게 넓다.**

이 프로젝트에서 사용하는 주요 기술/언어를 나열하면:
- **Rust** (클라이언트, 10개 크레이트)
- **Go** (서버 API)
- **TypeScript/React** (대시보드)
- **Python** (NER Fine-tuning, ONNX 변환 - 명시되지 않았으나 필수)
- **Protocol Buffers** (gRPC 스키마)
- **SQL** (PostgreSQL)
- **YAML/JSON** (패턴 정의, 설정)

3~5명 팀에서 이 모든 스택을 유지보수하는 것은 비현실적이다.

**숨겨진 복잡도(Hidden Complexity):**

1. **HWP 파싱**: `hwp-rs` 크레이트는 성숙도가 낮다. crates.io에서 `hwp` 크레이트의 최근 업데이트 상태를 보면, 프로덕션 수준의 HWP 파싱을 위해 상당한 자체 개발이 필요할 수 있다. 더 성숙한 `hwpers` 크레이트가 있지만 이것도 전체 HWP 5.0 지원에 대한 검증이 필요하다.
2. **DRM 연동**: Fasoo, SoftCamp, MarkAny 각 벤더의 SDK를 Rust FFI로 연동하는 것은 문서화되지 않은 SDK 동작, C/C++ 메모리 관리, 크래시 격리 등 상당한 난이도가 있다. 각 DRM 벤더와의 기술 협력 체결 자체도 수개월이 걸린다.
3. **Windows 서비스 안정성**: Windows 서비스로서 무인 환경에서 수개월간 안정적으로 동작해야 한다. 메모리 누수, 핸들 누수, 서비스 복구 정책 등 안정성 검증에 상당한 시간이 필요하다.
4. **ONNX Runtime 배포**: ONNX Runtime의 Rust 바인딩(`ort` 크레이트)은 네이티브 라이브러리(`onnxruntime.dll`)를 함께 배포해야 한다. 이 DLL의 크기는 ~30~60MB이며, 다양한 Windows 환경(AVX2/AVX-512 유무)에 대한 호환성 테스트가 필요하다.
5. **Protobuf 스키마 관리**: Rust(`prost`) + Go(`google.golang.org/grpc`) 간 `.proto` 스키마 동기화 및 하위 호환성 관리는 별도의 CI 파이프라인이 필요하다.

**대안:**
- 서버를 Go 대신 Rust로 통일하여 기술 스택을 줄이거나, 반대로 클라이언트도 Go/C++로 개발하는 것을 고려할 것 (Rust + Go 이중 유지보수 부담 제거)
- 대시보드는 MVP에서 최소 기능만 구현하고, 기존 오픈소스 관리 도구(Grafana 등)를 활용하는 것을 검토할 것
- DRM 연동은 v2 이후로 완전히 미루고, v1에서는 "DRM 파일 감지 → 스캔 불가 보고"만 구현할 것

## 3. 상용 솔루션 대비 차별점

### 3.1 기존 경쟁 솔루션 분석

| 솔루션 | 개발사 | 주요 특징 |
|--------|--------|----------|
| **Privacy-i** | 소만사 | 시장 1위 엔드포인트 DLP, 13개+ PII 패턴, EDR 통합, 원에이전트 |
| **FDR + AI-R Privacy** | 파수 | AI 기반 비정형 PII 탐지(F1 93.1%), OCR 연동, 15가지 유형, DRM 자체 연동 |
| **Server-i** | 소만사 | 서버 내 개인정보 탐지, SKT/KB국민은행 등 대형 고객사 보유 |

### 3.2 경쟁 우위 진단

**주장하는 차별점 vs 현실:**

| 주장 | 현실적 평가 |
|------|-----------|
| NER 기반 비정형 PII 탐지 | 파수가 이미 AI-R Privacy로 F1 93.1% 달성. 후발주자로서 성능 격차 극복 필요 |
| 실시간 파일 감지 | 소만사 Privacy-i도 실시간 감시 기능 보유. 차별점이 아님 |
| 낮은 오탐률 | 정량적 비교 데이터 없이 "낮다"고 주장할 수 없음 |
| CPU 효율성 | 기존 솔루션도 유휴 시간 활용, 리소스 제한 기능 보유. 정량적 비교 필요 |
| 증분 스캔 | 이것은 기본 기능이지 차별점이 아님 |

**실질적 차별화 가능 영역:**
1. **Rust 기반 경량 에이전트**: 기존 솔루션 대부분 C++/C# 기반. Rust의 메모리 안전성과 단일 바이너리 배포는 진짜 장점
2. **현대적 아키텍처**: 기존 솔루션은 10~15년 된 레거시. gRPC, 컨테이너 기반 서버는 운영 효율성에서 우위
3. **오픈된 패턴 확장 체계**: JSON 스키마 기반 커스텀 패턴 정의는 기업 맞춤화에 유리
4. **프라이버시 by Design**: Level 1(메타데이터만) 기본 전송은 개인정보보호법 강화 추세에 부합

**핵심 지적: "NER 기반 비정형 PII 탐지"를 핵심 차별점으로 내세우기에는 파수가 이미 앞서 있다.** 차별점을 재정의해야 한다. "경량 + 안전한 에이전트"와 "프라이버시 보호 설계"가 더 현실적인 차별화 포인트이다.

### 3.3 "NER 기반 비정형 PII 탐지"가 실제로 경쟁 우위가 되는가?

**결론: 단독으로는 아니다.**

- 파수 AI-R Privacy가 이미 Transformer 기반 모델로 F1 93.1%를 달성했다.
- 이 설계에서 KoELECTRA-Small INT8으로 시작하면, 파수 대비 성능 열위에서 출발한다.
- NER 자체보다 **"정규식+NER 하이브리드에 의한 비용 효율성"** (GPU 불필요, 경량 에이전트)이 차별점이 될 수 있다.
- 파수의 AI-R Privacy가 서버 측 추론인지 클라이언트 측 추론인지 확인이 필요하다. 만약 서버 측이라면, 본 솔루션의 "클라이언트 로컬 NER"은 차별점이 될 수 있다.

## 4. MVP vs 후순위 경계

### 4.1 현실적인 MVP 범위 제안

설계서 3개의 Phase 구분을 종합하면, Phase 1에 포함된 항목이 이미 너무 많다. 현실적인 MVP를 재정의한다:

**진짜 MVP (3~4개월, 5명 기준):**

| 포함 | 제외 (v2로 연기) |
|------|-----------------|
| 정규식+체크섬 정형 PII 탐지 | NER 비정형 PII 탐지 |
| 주요 5개 파일 형식 (TXT, DOCX, XLSX, PDF, CSV) | HWP, EML, MSG, PPT, RTF |
| 예약 스캔 (증분 스캔 포함) | 실시간 파일 감지 (FSW) |
| TLS 1.3 + API Key 인증 | mTLS, Certificate Pinning |
| 기본 온라인 라이선스 | 오프라인 라이선스, 퍼지 핑거프린팅 |
| 최소 관리 대시보드 (결과 조회, 정책 설정) | 통계/보고서, 컴플라이언스 대시보드 |
| Windows 10/11 지원 | DRM 파일, macOS/Linux |
| 단일 서버 배포 | Kubernetes, 수평 확장 |
| 기본 감사 로그 | 해시 체인, SIEM 연동 |

**근거:**
- NER을 v1에서 제외하는 이유: NER 모델 Fine-tuning + ONNX 변환 + Rust 연동 + 성능 최적화만으로 3~4개월이 소요될 수 있다. 정규식+체크섬만으로도 정형 PII는 충분히 탐지 가능하며, 이것만으로도 기존 솔루션의 핵심 기능을 커버한다.
- 실시간 감지를 v1에서 제외하는 이유: ReadDirectoryChangesW + USN Journal 하이브리드 구현과 디바운싱/큐잉 안정화에 상당한 시간이 필요하다. 예약 스캔으로도 일일 1회 증분 스캔이 가능하다.
- HWP를 v1에서 제외하는 이유: HWP 파싱의 Rust 생태계 성숙도가 낮다. Python 기반 라이브러리는 성숙하므로, v2에서 별도 프로세스로 Python 파서를 호출하는 전략도 가능하다.

### 4.2 Phase 구분 비판

**detection-engine.md**: Phase 구분이 명시적이지 않다. Stage 1/2/3 파이프라인이 있지만 이것은 실행 순서이지 개발 순서가 아니다.

**security.md**: Phase 1(MVP)/2/3으로 나뉘어 있으나, Phase 1에 "기본 감사 로그"와 "로컬 DPAPI 암호화"가 포함되어 있다. DPAPI 연동은 Windows 전문 지식이 필요한 중간 난이도 작업이다.

**system-architecture.md**: Phase 구분이 서버 확장 전략(단일 서버 → 확장)에만 있고, 클라이언트 기능의 Phase 구분이 없다.

**핵심 지적: 3개 문서의 Phase가 서로 정렬되어 있지 않다.** 통합 로드맵이 필요하다.

## 5. 설계 간 불일치 및 갭 분석

### 5.1 기술 선택 충돌

| 항목 | detection-engine.md | system-architecture.md | security.md | 문제점 |
|------|-------------------|----------------------|-------------|--------|
| **메모리 사용량** | 상주 21MB, 피크 125MB | 상주 ~200MB(최소), ~300MB(권장) | 언급 없음 | 2~10배 차이. 어느 수치가 맞는가? |
| **NER 모델 크기** | INT8 ~14MB(Small), ~50MB RAM | ~80MB(양자화 모델 기준) | 모델 암호화 시 복호화 오버헤드 미반영 | 50MB vs 80MB. 암호화 해제 시 추가 메모리 필요 |
| **디바운스 시간** | 500ms | 300ms | 언급 없음 | 500ms인가 300ms인가? |
| **gRPC** | 언급 없음 | Go + tonic(Rust) | gRPC + TLS 1.3 | 서버가 Go인데 gRPC는 Go 표준, 클라이언트는 Rust tonic. 일관성 있으나 명시적 합의 필요 |
| **증분 스캔 해시** | 파일 해시(SHA-256 또는 미지정) | xxHash(xxh3) | SHA-256(무결성 검증) | 어떤 해시를 증분 스캔에 쓰는가? |

### 5.2 빠진 영역

**어느 문서에도 다루지 않은 항목들:**

1. **설치/배포 전략의 구체적 설계**: WiX 설치 패키지 언급만 있고, 대규모 배포(SCCM, GPO, Intune 등 MDM 연동)에 대한 설계가 없다. 50,000대 PC에 어떻게 배포할 것인가?
2. **에러 처리 및 복구 전략**: 각 모듈의 에러 처리 정책이 없다. NER 추론 실패 시, 텍스트 추출 실패 시, 서버 연결 실패 시 각각 어떻게 처리하는가?
3. **로깅 및 진단**: `tracing` 크레이트 언급만 있고, 로그 레벨 정책, 로그 로테이션, 로그 크기 제한, 원격 진단 방법이 없다.
4. **사용자 인터페이스(UI)**: 클라이언트 트레이 아이콘, 알림 팝업, 설정 화면 등 사용자 대면 UI 설계가 완전히 빠져 있다. "팝업 최소화"라고 했는데, 최소화된 팝업은 어떤 형태인가?
5. **다국어/현지화**: 관리자 대시보드와 클라이언트 UI의 언어 지원 범위가 명시되지 않았다.
6. **백업 및 재해 복구(DR)**: 서버 데이터 백업 전략, DB 복구 시간 목표(RTO/RPO)가 없다.
7. **테스트 전략**: 단위 테스트, 통합 테스트, E2E 테스트, 성능 테스트에 대한 전략이 없다. 특히 50,000대 규모의 부하 테스트 방법론이 필요하다.
8. **마이그레이션 전략**: 기존 솔루션(15년 운영)에서의 전환 계획이 없다. 기존 솔루션과 병행 운영 기간, 데이터 마이그레이션 등.

### 5.3 통합 시 발생할 수 있는 문제

1. **gRPC 스키마 진화**: 서버(Go)와 클라이언트(Rust)가 동시에 업데이트되지 않는 환경에서, Protobuf 스키마의 하위 호환성이 깨지면 대규모 장애가 발생한다.
2. **정책 전파 지연**: 서버에서 정책 변경 → 클라이언트 반영까지의 지연 시간에 대한 SLA가 없다. 50,000대에 정책을 동시에 푸시하면 서버에 순간 부하가 집중된다.
3. **시간 동기화**: security.md의 리플레이 방어에서 ±30초 허용이지만, 기업 환경에서 NTP 동기화가 안 되는 PC도 있다. 시간이 크게 어긋난 PC에서 인증 실패가 반복될 수 있다.

## 7. 종합 결론

### 7.1 설계의 강점

- **하이브리드 탐지 전략**(정규식 → NER)의 기본 아이디어는 합리적이다.
- **리소스 관리 설계**(Resource Governor)는 기존 솔루션의 핵심 페인포인트를 정확히 겨냥하고 있다.
- **보안 설계**는 매우 철저하며, 특히 데이터 최소화 원칙과 3단계 전송 모델은 높이 평가할 만하다.
- **법적/컴플라이언스 고려**가 설계 초기부터 반영된 것은 상용 제품으로서 중요한 장점이다.

### 7.2 핵심 개선 필요 사항

1. **정량적 목표 설정**: 오탐률/미탐률의 PII 유형별 정량 목표를 설정하고 측정 방법을 정의할 것
2. **MVP 범위 대폭 축소**: NER과 실시간 감지를 v2로 미루고, 정규식+체크섬 기반 예약 스캔으로 빠르게 출시할 것
3. **경쟁 차별점 재정의**: "NER 기반 비정형 탐지"가 아닌, "경량 에이전트 + 프라이버시 보호 + 현대적 관리 체계"를 1차 차별점으로 재설정할 것
4. **기술 스택 단순화**: Rust(클라이언트) + Go(서버) + React(대시보드)는 유지하되, 필요시 Python 보조(NER Fine-tuning, HWP 파싱)를 공식적으로 인정할 것
5. **문서 간 수치 정렬**: 메모리 사용량, 디바운스 시간 등 3개 문서 간 불일치를 해소할 것
6. **빠진 영역 보완**: 대규모 배포 전략, 사용자 UI, 테스트 전략, 마이그레이션 계획을 추가할 것
7. **통합 로드맵 작성**: 3개 문서의 Phase를 하나의 통합 타임라인으로 정렬할 것

### 7.3 최종 평가

이 설계는 기존 솔루션의 문제점을 정확히 진단하고 있으며, 기술적으로 올바른 방향을 제시하고 있다. 그러나 **scope가 과대하고, 일부 핵심 가정(NER 성능, 메모리 사용량, 개발 공수)이 낙관적이다.** MVP 범위를 대폭 줄이고, 파일럿 고객의 실 데이터를 기반으로 점진적으로 확장하는 전략이 성공 확률을 높일 것이다.

특히 **파수가 이미 AI 기반 비정형 PII 탐지 시장에 진입한 상황**에서, NER을 핵심 차별점으로 내세우기보다는 "Rust 기반 경량 에이전트의 비침습적 탐지"와 "Privacy by Design"을 중심으로 포지셔닝하는 것이 현실적이다.
