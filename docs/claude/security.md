## 1. 클라이언트-서버 통신 보안

### 1.1 프로토콜 선택: gRPC + TLS 1.3

**선택: gRPC (Protocol Buffers over HTTP/2)**

| 기준 | gRPC | REST |
|------|------|------|
| 직렬화 | Protocol Buffers (바이너리) | JSON (텍스트) |
| 전송 효율 | 페이로드 5~7배 작음 | 상대적으로 큼 |
| 양방향 스트리밍 | 네이티브 지원 | WebSocket 별도 필요 |
| 스키마 강제 | .proto 파일로 계약 강제 | OpenAPI 선택적 |
| HTTP/2 멀티플렉싱 | 기본 지원 | HTTP/1.1이 일반적 |
| 보안 인터셉터 | 미들웨어 체인 기본 지원 | 프레임워크 의존 |

**선택 근거:**
- 수만 대 클라이언트가 주기적으로 탐지 결과를 전송하므로, 바이너리 직렬화와 HTTP/2 멀티플렉싱에 의한 대역폭 절감이 결정적
- 양방향 스트리밍으로 서버→클라이언트 정책 푸시(정책 변경, 긴급 스캔 명령)가 별도 채널 없이 가능
- `.proto` 파일로 API 계약이 강제되어 클라이언트-서버 간 프로토콜 불일치 방지
- Rust 생태계에서 `tonic` 크레이트가 성숙하여 gRPC 구현에 문제 없음

### 1.2 전송 계층 보안: mTLS (Mutual TLS)

#### 인증서 아키텍처

```
[사내 루트 CA]
    |
    +-- [서버 중간 CA]
    |       +-- 서버 인증서 (*.pii-scanner.example.com)
    |
    +-- [클라이언트 중간 CA]
            +-- 클라이언트 인증서 (기기별 발급)
```

**mTLS 동작 흐름:**
1. 클라이언트가 서버에 접속하면 서버가 서버 인증서 제시
2. 클라이언트가 서버 인증서를 검증 (Certificate Pinning 적용)
3. 서버가 클라이언트 인증서를 요구
4. 클라이언트가 기기별 클라이언트 인증서 제시
5. 서버가 클라이언트 인증서를 클라이언트 중간 CA로 검증
6. 양방향 인증 완료 후 TLS 1.3 세션 수립

#### Certificate Pinning 전략

- **핀닝 대상:** 서버 중간 CA의 공개키(SPKI, Subject Public Key Info)를 핀닝
  - 리프 인증서 핀닝은 인증서 갱신 시 클라이언트 전체 업데이트 필요하므로 부적합
  - 중간 CA 핀닝은 리프 인증서를 자유롭게 교체 가능하면서도 MITM 방어 가능
- **백업 핀:** 현재 중간 CA + 예비 중간 CA 2개의 SPKI 해시를 내장
- **핀 업데이트:** 정책 푸시 채널을 통해 신규 핀 해시 배포 가능 (기존 핀으로 인증된 채널에서만)

#### 인증서 관리

| 항목 | 정책 |
|------|------|
| 서버 인증서 유효기간 | 90일, 자동 갱신 (ACME 또는 내부 CA 자동화) |
| 클라이언트 인증서 유효기간 | 1년, 만료 30일 전 자동 갱신 요청 |
| 인증서 폐기(Revocation) | OCSP Stapling + 단기 CRL (24시간 주기) |
| 키 저장 | Windows: TPM 2.0 또는 DPAPI로 개인키 보호 |
| TLS 버전 | TLS 1.3만 허용 (1.2 이하 거부) |
| 암호 스위트 | TLS_AES_256_GCM_SHA384, TLS_CHACHA20_POLY1305_SHA256 |

### 1.3 클라이언트 인증 체계: 인증서 + API Token 이중 인증

mTLS로 전송 계층에서 기기를 인증하되, 애플리케이션 계층에서 추가 인증을 수행한다.

```
[인증 흐름]

1. 초기 등록 (Enrollment):
   관리자가 관리 콘솔에서 등록 토큰(OTP) 발급
   → 클라이언트 설치 시 등록 토큰 입력
   → 서버가 토큰 검증 후 클라이언트 인증서 + API Token 발급
   → 클라이언트가 인증서를 DPAPI/TPM에 저장

2. 일상 통신:
   mTLS 연결 수립 (기기 인증서로 전송 계층 인증)
   → gRPC metadata에 API Token 포함 (애플리케이션 계층 인증)
   → 서버가 인증서 CN과 API Token의 기기 ID 일치 여부 검증

3. 토큰 갱신:
   API Token 유효기간: 24시간
   → 만료 1시간 전 자동 갱신 요청 (mTLS 채널 통해)
   → 서버가 새 토큰 발급, 이전 토큰은 5분 유예 후 폐기
```

**이중 인증의 필요성:**
- mTLS만으로는 인증서 탈취 시 대응이 느림 (CRL 전파 지연)
- 단기 수명 API Token으로 인증서 폐기 전에도 즉시 접근 차단 가능
- 토큰 블랙리스트 서버 측 관리로 실시간 차단

### 1.4 메시지(Payload) 레벨 암호화

**결론: 선택적 적용 (민감 필드만)**

TLS 1.3이 전송 구간을 완전히 암호화하므로 전체 페이로드 이중 암호화는 불필요하다. 다만, 다음의 경우 payload-level encryption을 적용한다:

| 대상 | 암호화 방식 | 근거 |
|------|-------------|------|
| 탐지된 PII 원문 스니펫 | AES-256-GCM (서버 공개키로 봉인) | TLS 종단 후 서버 메모리에 평문 노출 최소화 |
| 라이선스 토큰 | JWE (JSON Web Encryption) | 로그에 노출되어도 복호화 불가 |
| 일반 메타데이터 (파일명, 해시 등) | TLS만으로 충분 | 이중 암호화 오버헤드 불필요 |

**PII 스니펫 봉인(Envelope Encryption) 흐름:**
1. 클라이언트가 랜덤 DEK(Data Encryption Key) 생성
2. DEK로 PII 스니펫을 AES-256-GCM 암호화
3. 서버의 RSA-OAEP 공개키로 DEK를 암호화
4. 암호화된 스니펫 + 암호화된 DEK를 함께 전송
5. 서버는 개인키로 DEK 복호화 → DEK로 스니펫 복호화
6. 복호화된 PII는 서버의 보안 메모리(Secure Enclave 또는 HSM)에서만 처리

### 1.5 리플레이 공격(Replay Attack) 방어

```
[모든 gRPC 요청에 포함되는 보안 헤더]

message SecurityHeader {
  string  request_id  = 1;  // UUID v7 (시간 순서 보장)
  int64   timestamp   = 2;  // Unix timestamp (밀리초)
  string  nonce       = 3;  // 32바이트 랜덤 (crypto-secure)
  bytes   hmac        = 4;  // HMAC-SHA256(API_Token, request_id + timestamp + nonce + body_hash)
}
```

**방어 메커니즘:**

1. **Timestamp 검증:** 서버가 요청의 timestamp와 현재 시각의 차이를 확인. ±30초 초과 시 거부
2. **Nonce 중복 검사:** 서버가 30초 윈도우 내 nonce를 Redis에 캐시. 중복 nonce 즉시 거부
3. **HMAC 검증:** API Token을 키로 사용한 HMAC-SHA256으로 요청 무결성 검증. 변조된 요청 탐지
4. **Sequence Number:** 클라이언트별 단조 증가 시퀀스 번호로 순서 역전(out-of-order) 탐지

**MITM 방어 요약:**
- mTLS + Certificate Pinning → 연결 자체의 가로채기 방지
- Payload Encryption → TLS 종단 후에도 PII 보호
- HMAC + Nonce + Timestamp → 리플레이 및 변조 방지

## 2. 라이선스 시스템 설계

### 2.1 라이선스 모델 개요

```
[구독형 라이선스 계층 구조]

조직 (Organization)
 └── 라이선스 계약 (License Agreement)
      ├── 라이선스 풀 (License Pool): 최대 N대 기기
      │    ├── 기기 1 (활성)
      │    ├── 기기 2 (활성)
      │    └── ...
      └── 기능 티어 (Feature Tier)
           ├── Standard: 정규식+체크섬 탐지, 예약 스캔
           ├── Professional: + NER 탐지, 실시간 감시
           └── Enterprise: + DRM 파일 탐지, 커스텀 정책, API 연동
```

### 2.2 라이선스 토큰 구조

서버 발급 라이선스 토큰은 서명된 JWT(JWS) 형식으로 설계한다.

```json
{
  "header": {
    "alg": "ES256",
    "typ": "JWT",
    "kid": "license-signing-key-2026-01"
  },
  "payload": {
    "iss": "pii-scanner-license-server",
    "sub": "device:a1b2c3d4-hardware-fingerprint",
    "aud": "pii-scanner-client",
    "iat": 1738800000,
    "exp": 1741478400,
    "nbf": 1738800000,
    "jti": "lic-uuid-unique-id",
    "org_id": "org-samsung-12345",
    "tier": "professional",
    "features": ["regex_scan", "checksum_scan", "ner_scan", "realtime_watch"],
    "max_devices": 5000,
    "hw_fingerprint": "sha256:abcdef1234567890...",
    "grace_period_hours": 720,
    "offline_max_days": 30
  },
  "signature": "ES256 서명"
}
```

**서명 알고리즘:** ES256 (ECDSA P-256)
- RSA 대비 키/서명 크기가 작아 토큰 경량화
- 서버의 개인키로 서명, 클라이언트에 내장된 공개키로 오프라인 검증 가능

### 2.3 하드웨어 핑거프린팅 (Hardware Fingerprinting)

단일 하드웨어 값에 의존하면 해당 값 변경 시 라이선스가 무효화되므로, 여러 요소를 조합한 퍼지 핑거프린트(Fuzzy Fingerprint)를 사용한다.

```
[핑거프린트 요소 및 가중치]

요소                     가중치    수집 방법
────────────────────────────────────────────────
CPU ID (CPUID)           25%      CPUID 명령어
메인보드 시리얼           25%      WMI Win32_BaseBoard
디스크 시리얼 (부팅 디스크) 20%    WMI Win32_DiskDrive
MAC 주소 (내장 NIC)       15%      GetAdaptersInfo (가상 NIC 제외)
Windows SID               10%      LookupAccountName
BIOS 시리얼               5%      WMI Win32_BIOS
```

**퍼지 매칭 알고리즘:**
1. 각 요소를 개별 해시: `SHA-256(element_value + salt)`
2. 가중 유사도 계산: 변경된 요소의 가중치 합산
3. 변경 허용 임계값: 30% 미만이면 동일 기기로 판정
4. 30~50% 변경: 서버에 재인증 요청 (관리자 승인 또는 자동)
5. 50% 초과: 새 기기로 판정, 기존 라이선스 슬롯 해제 필요

**변경 시나리오 대응:**
- 디스크 교체 (20%): 자동 허용 → 핑거프린트 업데이트
- 디스크 + NIC 교체 (35%): 서버에 자동 재인증 요청
- 메인보드 교체 (25% + 연관 변경): 관리자 승인 필요

### 2.4 오프라인 환경 대응

폐쇄망(Air-gapped) 환경의 기업도 지원해야 하므로 오프라인 라이선스 체계를 설계한다.

```
[온라인 라이선스 검증 흐름]

클라이언트 ──(mTLS)──> 서버
  1. 매 기동 시 라이선스 검증 요청
  2. 서버가 현재 토큰 유효성 + 기기 핑거프린트 확인
  3. 유효하면 갱신된 토큰 발급 (24시간 유효)
  4. 클라이언트가 토큰을 로컬 캐시에 암호화 저장

[오프라인 모드 전환]

서버 접속 실패 시:
  → 캐시된 토큰의 exp 확인
  → exp 미도과: 정상 동작 (최대 30일)
  → exp 도과 + grace_period 내: 제한 모드 (스캔 가능, 결과 로컬 저장, 서버 전송 보류)
  → grace_period 초과: 기능 비활성화, 관리자 알림 표시

[오프라인 전용 라이선스 (폐쇄망)]

  1. 관리자가 관리 콘솔에서 오프라인 라이선스 파일(.lic) 생성
  2. .lic 파일: 서명된 JWT + 유효기간(최대 1년) + 기기 핑거프린트 목록
  3. USB 또는 내부망으로 클라이언트에 배포
  4. 클라이언트가 내장 공개키로 서명 검증
  5. 핑거프린트 목록과 자신의 핑거프린트 매칭
  6. 갱신 시 관리자가 새 .lic 파일 생성하여 교체
```

**그레이스 기간 (Grace Period) 정책:**

| 상황 | 그레이스 기간 | 동작 |
|------|-------------|------|
| 서버 일시 장애 | 24시간 (토큰 유효기간) | 완전 정상 동작 |
| 서버 장기 장애 | 7일 | 정상 동작 + 로컬 큐잉 |
| 네트워크 단절 (오프라인 모드) | 30일 | 정상 동작 + 로컬 큐잉 |
| 라이선스 만료 | 7일 | 제한 모드 (신규 스캔 중단, 기존 결과 열람만) |
| 라이선스 만료 + 그레이스 초과 | - | 기능 비활성화, 언인스톨 유도 안내 |

### 2.5 라이선스 위반 탐지 및 대응

```
[서버 측 위반 탐지 로직]

1. 동시 접속 초과 (Over-provisioning)
   → 라이선스 풀의 max_devices 대비 활성 기기 수 모니터링
   → 초과 시 관리자에게 알림 + 신규 기기 등록 차단

2. 핑거프린트 클로닝 감지
   → 동일 핑거프린트에서 서로 다른 IP/네트워크에서 동시 접속
   → 이상 탐지 시 해당 기기의 토큰 즉시 폐기 + 관리자 알림

3. 토큰 변조 감지
   → ES256 서명 검증 실패
   → 즉시 차단 + 보안 이벤트 로깅

4. 시간 조작 감지 (시계 롤백)
   → 클라이언트 시각과 서버 시각의 편차 모니터링
   → 클라이언트 로컬에 단조 증가 카운터 유지
   → 시계가 과거로 돌아가면 카운터 불일치로 탐지

5. 대응 단계
   Level 1: 관리자 알림 (대시보드 + 이메일)
   Level 2: 해당 기기 기능 제한
   Level 3: 해당 기기 라이선스 폐기
   Level 4: 조직 전체 라이선스 감사 요청
```

### 2.6 엔터프라이즈 볼륨 라이선스 관리

```
[볼륨 라이선스 관리 체계]

조직 관리자 포털
 ├── 라이선스 현황 대시보드
 │    ├── 전체 보유 라이선스 수
 │    ├── 활성 기기 수 / 비활성 기기 수
 │    ├── 기능 티어별 배분 현황
 │    └── 만료 예정 알림
 ├── 기기 관리
 │    ├── 기기 등록/해제
 │    ├── 기기 그룹 (부서별, 위치별)
 │    └── 기기 상태 모니터링
 ├── 라이선스 풀 관리
 │    ├── 풀 생성/분할 (부서별 할당)
 │    ├── 유동적 재배분 (Floating License)
 │    └── 자동 회수: 90일 미접속 기기는 슬롯 반환
 └── 감사 및 보고
      ├── 라이선스 사용률 보고서
      ├── 위반 이력 보고서
      └── 컴플라이언스 보고서
```

**Floating License 모델:**
- 전체 풀에서 동시 접속 수 기준으로 라이선스 소비
- 예: 5,000대 기기 중 동시 가동 비율이 70%이면, 3,500개 라이선스로 운영 가능
- 피크 시간대 초과 시 큐잉(대기열) 처리하여 즉시 차단 방지

## 3. 바이너리 보호 및 리버스 엔지니어링 대응

### 3.1 현실적 보호 수준 평가

> **핵심 원칙:** 완벽한 리버스 엔지니어링 방지는 불가능하다. 목표는 "크랙 비용 > 정품 구매 비용"을 달성하는 것이다.

2025년 이후 AI 코딩 어시스턴트의 발전으로 디컴파일 난이도가 크게 낮아졌다. 따라서 바이너리 보호에 과도한 비용을 투자하기보다, **서버 검증 의존도를 높이는** 전략이 더 효과적이다.

### 3.2 Rust 바이너리 특성 및 보호 전략

**Rust의 기본적 보호 특성:**
- 정적 링킹(static linking)으로 외부 의존성 내재화 → 심볼 참조 분석 어렵게 함
- 제네릭 단형화(monomorphization)로 코드 구조 난독화 효과
- 패닉 핸들링 코드가 산재하여 제어 흐름 분석 복잡화

**컴파일 단계 보호:**

```toml
# Cargo.toml - Release 프로필
[profile.release]
opt-level = 3           # 최대 최적화 (인라이닝으로 함수 경계 흐림)
lto = "fat"             # 전체 LTO (크로스 모듈 최적화로 분석 어렵게)
codegen-units = 1       # 단일 코드젠 유닛 (최적화 극대화)
strip = "symbols"       # 심볼 테이블 제거
panic = "abort"         # 언와인드 정보 제거
debug = false           # 디버그 정보 제거
```

**추가 보호 계층:**

| 계층 | 기법 | 구현 방법 | 비용/복잡도 |
|------|------|-----------|-------------|
| 1 | 문자열 암호화 | `obfstr` 또는 `litcrypt` 크레이트로 컴파일 시 문자열 암호화, 런타임 복호화 | 낮음 |
| 2 | 제어 흐름 평탄화 | OLLVM (LLVM Obfuscator) 기반 Rust 빌드 파이프라인 | 중간 |
| 3 | 무결성 자가 검증 | 바이너리의 SHA-256 해시를 서버에서 주기적으로 확인 | 낮음 |
| 4 | 안티 디버깅 | `IsDebuggerPresent`, `NtQueryInformationProcess` 등 Windows API 호출 | 낮음 |
| 5 | 안티 탬퍼링 | 코드 섹션 CRC 주기적 검증 | 중간 |

### 3.3 안티 디버깅 전략

```rust
// 의사 코드 - 안티 디버깅 체크 (다중 기법 조합)

fn anti_debug_check() -> bool {
    // 1. Windows API 기반 디버거 감지
    let api_check = IsDebuggerPresent() || CheckRemoteDebuggerPresent();

    // 2. 타이밍 체크 (디버거 실행 시 지연 발생)
    let start = rdtsc();
    // ... 의미 없는 연산 ...
    let elapsed = rdtsc() - start;
    let timing_check = elapsed > THRESHOLD;

    // 3. 브레이크포인트 탐지 (INT3 = 0xCC)
    let bp_check = scan_code_section_for_breakpoints();

    // 4. 디버깅 도구 프로세스 감지
    let tool_check = check_debug_tool_processes(); // x64dbg, IDA, Ghidra 등

    api_check || timing_check || bp_check || tool_check
}

// 탐지 시 대응: 크래시가 아닌 잘못된 동작을 유발
// (크래시는 디버거에게 정확한 체크 위치를 알려주므로)
fn on_debug_detected() {
    // 점진적으로 잘못된 탐지 결과 생성
    // 라이선스 토큰을 조용히 무효화
    // 서버에 탬퍼링 이벤트 보고 (가능 시)
}
```

**안티 디버깅 정책:**
- 디버거 탐지 시 즉시 종료 대신 **조용히 오동작**하게 설계 (honey-pot 접근)
- 체크 로직을 코드 전반에 분산 배치 (단일 패치로 우회 불가)
- 릴리스마다 체크 위치와 방식을 랜덤하게 변경

### 3.4 NER 모델 파일 보호

NER 모델(ONNX 포맷)은 핵심 지적 재산이므로 별도 보호한다.

```
[모델 보호 전략]

1. 모델 암호화 저장
   - AES-256-GCM으로 모델 파일 암호화
   - 복호화 키는 바이너리에 난독화하여 내장 + 서버에서 키 파편(key shard) 보완
   - 키 = local_shard XOR server_shard (양쪽 모두 있어야 복호화 가능)

2. 메모리 내 보호
   - 모델 로딩 시 복호화된 데이터는 mlock()으로 페이징 방지
   - 추론 완료 후 모델 메모리 영역 즉시 zeroize

3. 모델 무결성 검증
   - 모델 파일의 SHA-256 해시를 서버에서 주기적으로 확인
   - 해시 불일치 시 모델 사용 중단 + 재다운로드

4. 서버 의존적 모델 로딩 (Enterprise 티어)
   - 모델의 핵심 레이어 가중치를 서버에서 실시간 스트리밍
   - 로컬에는 불완전한 모델만 저장
   - 오프라인 환경: 암호화된 전체 모델 제공 (오프라인 라이선스 연동)
```

### 3.5 비용-효과 분석 및 권장 수준

| 보호 수준 | 포함 기법 | 예상 비용 | 크랙 소요 시간(추정) | 권장 |
|-----------|---------|----------|-------------------|------|
| Level 1 (기본) | 심볼 제거 + LTO + 문자열 암호화 | 낮음 | 수일 | 최소 적용 |
| Level 2 (표준) | L1 + 안티 디버깅 + 무결성 검증 + 모델 암호화 | 중간 | 수주 | **권장** |
| Level 3 (강화) | L2 + OLLVM 제어 흐름 평탄화 + 서버 의존적 모델 로딩 | 높음 | 수개월 | Enterprise 티어 |
| Level 4 (극대) | L3 + 상용 보호 솔루션 (VMProtect 등) | 매우 높음 | 수개월~1년 | 불필요 (비용 대비 효과 낮음) |

**권장: Level 2 (표준)을 기본으로 적용, Enterprise 고객에게 Level 3 옵션 제공**

근거:
- 서버 검증에 의존하므로 바이너리 크랙만으로는 무제한 사용 불가
- Level 2 수준이면 캐주얼 크래킹 시도를 충분히 저지
- AI 지원 리버스 엔지니어링 시대에 과도한 바이너리 보호는 비용 낭비

## 4. 탐지 결과(개인정보) 보호

### 4.1 설계 원칙: 데이터 최소화 (Data Minimization)

> **핵심 원칙:** 탐지 솔루션 자체가 개인정보 침해의 수단이 되어서는 안 된다.

- 원본 PII를 가능한 한 저장하지 않는다
- 불가피하게 저장/전송 시 최소 범위로 마스킹 또는 토큰화
- 보존 기간을 최소화하고 자동 삭제 적용

### 4.2 로컬 저장 시 암호화

```
[로컬 저장 데이터 구조]

scan_results/
 ├── index.db          ← SQLite (암호화된 컬럼)
 │    ├── file_path: 평문 (파일 경로는 PII 아님)
 │    ├── file_hash: SHA-256 (변경 감지용)
 │    ├── scan_time: 평문
 │    ├── pii_type: 평문 ("주민등록번호", "이메일" 등)
 │    ├── pii_count: 평문
 │    ├── pii_snippet: AES-256-GCM 암호화 (마스킹된 스니펫)
 │    ├── pii_location: 암호화 (행/열 위치)
 │    └── confidence: 평문 (탐지 신뢰도)
 ├── temp/             ← 스캔 중 임시 파일 (완료 후 즉시 삭제 + zeroize)
 └── queue/            ← 서버 전송 대기 큐 (암호화)
```

**암호화 키 관리:**

```
[로컬 암호화 키 계층]

DPAPI Master Key (Windows 사용자 자격증명에 바인딩)
 └── Local KEK (Key Encryption Key): DPAPI로 보호
      └── DEK (Data Encryption Key): KEK로 암호화
           └── 개별 레코드 암호화에 사용
```

- **DPAPI 사용 근거:**
  - Windows 내장 API로 별도 키 관리 인프라 불필요
  - 사용자 자격증명에 바인딩되어 다른 사용자/기기에서 복호화 불가
  - TPM 2.0이 있는 환경에서는 TPM에 키 보호 위임 가능

- **키 로테이션:** DEK를 30일마다 자동 교체, 이전 DEK는 기존 데이터 복호화용으로 보존

### 4.3 서버 전송 전략: 메타데이터 우선, 원문 최소화

```
[서버 전송 데이터 3단계 모델]

Level 1: 메타데이터만 전송 (기본, 권장)
─────────────────────────────────────
{
  "device_id": "a1b2c3d4",
  "file_path_hash": "sha256:...",         // 파일 경로 해시 (원본 경로 비전송)
  "file_name": "고객명단.xlsx",            // 파일명 (관리자 식별용)
  "file_hash": "sha256:...",
  "scan_time": "2026-02-09T10:30:00Z",
  "pii_findings": [
    {
      "pii_type": "resident_registration_number",
      "count": 15,
      "confidence": 0.97,
      "location_summary": "Sheet1, rows 2-16, column C"
    }
  ]
}

Level 2: 마스킹된 스니펫 포함 (관리자 설정 시)
─────────────────────────────────────────────
위 Level 1 + {
  "masked_snippets": [
    "홍**  /  9501**-1******  /  서울시 강남구 **동"
  ]
}

Level 3: 원문 스니펫 포함 (감사/포렌식 목적, 별도 승인 필요)
──────────────────────────────────────────────────────────
위 Level 2 + {
  "raw_snippets": [
    // Envelope Encryption 적용 (1.4절 참조)
    { "encrypted_data": "base64...", "encrypted_dek": "base64..." }
  ]
}
```

**전송 레벨 정책:**
- Level 1이 기본값 (개인정보보호법 최소 수집 원칙 준수)
- Level 2/3은 조직 관리자가 명시적으로 설정해야 활성화
- Level 3 활성화 시 개인정보 수집 동의 고지 의무 안내

### 4.4 서버 측 데이터 보호

```
[서버 데이터 보호 아키텍처]

                    ┌─────────────────────────┐
                    │     Application Layer    │
                    │  (탐지 결과 처리 로직)    │
                    └────────────┬────────────┘
                                 │
                    ┌────────────▼────────────┐
                    │    Encryption Layer      │
                    │  AES-256-GCM (at rest)   │
                    │  Column-level encryption │
                    └────────────┬────────────┘
                                 │
                    ┌────────────▼────────────┐
                    │      Database Layer      │
                    │   PostgreSQL + pgcrypto  │
                    │   또는 TDE 지원 DB       │
                    └────────────┬────────────┘
                                 │
                    ┌────────────▼────────────┐
                    │    Storage Encryption    │
                    │  Volume-level (LUKS/     │
                    │  BitLocker/Cloud KMS)    │
                    └─────────────────────────┘

[접근 제어]
- 관리자 역할: 메타데이터 열람, 마스킹 스니펫 열람
- 보안 관리자 역할: 원문 스니펫 복호화 (MFA + 사유 입력 필수)
- 감사자 역할: 감사 로그만 열람
- 시스템 서비스 계정: 데이터 처리만, 직접 조회 불가
```

### 4.5 감사 로그 (Audit Log) 설계

```
[감사 로그 스키마]

{
  "log_id": "uuid-v7",
  "timestamp": "2026-02-09T10:30:00.123Z",
  "event_type": "PII_DATA_ACCESS | PII_DATA_DECRYPT | POLICY_CHANGE | LICENSE_EVENT | ...",
  "actor": {
    "type": "admin | system | auditor",
    "id": "user-id-or-service-account",
    "ip": "192.168.1.100",
    "session_id": "session-uuid"
  },
  "target": {
    "type": "scan_result | device | policy | license",
    "id": "target-resource-id"
  },
  "action": "VIEW_MASKED | VIEW_RAW | EXPORT | DELETE | MODIFY",
  "details": {
    "reason": "사유 (원문 접근 시 필수)",
    "data_scope": "영향받는 레코드 범위",
    "result": "SUCCESS | DENIED | ERROR"
  },
  "integrity": {
    "prev_log_hash": "sha256:이전 로그 해시 (체인)",
    "log_hash": "sha256:현재 로그 해시"
  }
}
```

**감사 로그 정책:**
- **무결성:** 해시 체인(hash chain)으로 로그 변조 탐지. 각 로그 항목이 이전 항목의 해시를 포함
- **불변성:** Append-only 저장소 (Write-Once-Read-Many). 삭제/수정 불가
- **보존 기간:** 최소 5년 (개인정보보호법 기록 보존 요건)
- **외부 전송:** SIEM 연동으로 중앙 집중 모니터링 가능
- **실시간 알림:** 원문 PII 접근, 대량 데이터 조회, 비정상 접근 패턴 시 관리자 알림

### 4.6 데이터 보존 및 삭제 정책

| 데이터 유형 | 보존 기간 | 삭제 방법 |
|------------|----------|----------|
| 로컬 스캔 결과 | 서버 전송 확인 후 7일 | Secure erase (zeroize) |
| 서버 메타데이터 | 정책 설정에 따름 (기본 1년) | 논리 삭제 → 30일 후 물리 삭제 |
| 마스킹 스니펫 | 정책 설정에 따름 (기본 90일) | 즉시 물리 삭제 |
| 원문 스니펫 | 최대 30일 (강제) | 즉시 물리 삭제 + 저장소 zeroize |
| 감사 로그 | 5년 (법적 요건) | 보존 기간 후 아카이브 → 10년 후 삭제 |
| 라이선스 이력 | 계약 종료 후 3년 | 논리 삭제 |

## 5. 법적/컴플라이언스 요건

### 5.1 개인정보보호법 준수 사항

#### 적용 법령
- **개인정보 보호법** (법률 제19234호, 2025.3.15 시행)
- **개인정보 보호법 시행령**
- **개인정보의 안전성 확보조치 기준** (고시)
- **개인정보의 기술적·관리적 보호조치 기준**

#### 핵심 준수 요건

**1) 수집·이용 최소화 원칙 (법 제3조, 제15조)**
```
[PII Scanner에의 적용]

문제: PII 탐지 솔루션이 탐지 과정에서 개인정보를 수집하게 됨
해법:
- 탐지 결과에 원본 PII 대신 메타데이터(유형, 건수, 위치)만 기본 수집
- 원문 수집 시 정보주체 동의 또는 법적 근거 필요
- 기업의 "내부 관리 목적의 개인정보 처리"로 정당한 이익 근거 주장 가능
  (단, 비례성 원칙 준수: 탐지 목적에 필요한 최소한만 수집)
```

**2) 안전성 확보조치 (법 제29조)**
- 접근 권한 관리: 역할 기반 접근 제어(RBAC)
- 접근 통제: 네트워크 단위 접근 통제 + 암호화 통신
- 개인정보 암호화: 전송 시 TLS 1.3, 저장 시 AES-256
- 접속 기록 관리: 감사 로그 최소 1년 보관 (5만 명 이상 시 2년)
- 악성프로그램 방지: 바이너리 무결성 검증

**3) 개인정보 처리방침 고지 의무 (법 제30조)**
```
[고객사에 제공해야 할 고지 템플릿]

PII Scanner 운영 시 조직이 고지해야 할 사항:
1. 개인정보 탐지 솔루션 운영 사실
2. 탐지되는 개인정보의 유형
3. 탐지 결과의 보존 기간 및 파기 방법
4. 탐지 결과에 대한 접근 권한자
5. 정보주체의 열람·정정·삭제 요구권
→ 제품에 고지 템플릿을 기본 제공하여 고객사의 법적 부담 경감
```

**4) 개인정보 전송요구권 (2025.3.15 시행)**
- 정보주체가 자신의 개인정보 전송을 요구할 수 있음
- PII Scanner 서버에 특정 정보주체의 탐지 기록이 있다면 전송 의무 발생 가능
- **대응:** 정보주체별 탐지 기록 검색 및 추출 API 제공

**5) 자동화된 의사결정에 대한 권리**
- NER 기반 자동 PII 탐지가 자동화된 의사결정에 해당할 수 있음
- **대응:** 자동 탐지 결과에 대한 이의 제기 프로세스를 관리 콘솔에 구현

### 5.3 개인정보 영향평가 (PIA) 대응

```
[PIA 지원 체계]

1. 사전 영향평가 도구 제공
   - PII Scanner 도입 시 고객사가 수행할 PIA 체크리스트 제공
   - 법적 근거, 처리 목적, 안전조치 등 항목별 가이드

2. 영향평가 지원 데이터
   - 처리하는 개인정보 유형 목록
   - 데이터 흐름도 (클라이언트 → 서버 → 저장소)
   - 적용된 보호조치 상세 목록
   - 제3자 제공 현황 (없음 또는 서버 호스팅 사업자)

3. PIA 의무 해당 여부 판단
   - 공공기관: 법적 의무 대상 (5만 명 이상 민감정보 처리 시)
   - 민간기업: 현행법상 의무는 아니나, 향후 확대 예정
   - 권장: 모든 고객사에 자발적 PIA 수행 권고

4. 정기 재평가
   - 주요 기능 업데이트 시 PIA 재수행 안내
   - 연 1회 이상 정기 재평가 권고
```

### 5.4 개인정보 처리 동의 및 고지 체계

```
[동의 체계 설계]

PII Scanner의 특수성: 솔루션 운영 주체는 기업(고용주), 정보주체는 직원 및 고객

시나리오 1: 기업 내부 PC 스캔 (직원 PC)

- 고지 의무: 직원에게 PII 탐지 솔루션 운영 사실 고지

- 탐지 결과에 고객 PII 원문을 수집하려면 별도 법적 근거 필요

- EU 직원의 PC에서 스캔 시 GDPR 적용
```
